## 第一层：内功 — 门禁系统的自我进化能力

### 1.1 系统定义与边界

**这个系统负责什么**：覆盖门禁是芽芽数据质量的最后防线。任何 knowledge.json / levels.json / knowledge_graph.json 的变更必须通过门禁才能提交。当前 10 条铁规（R1-R10）由 data_quality_gate.py 脚本自动化执行。

**这个系统不负责什么**：不负责修复数据（只报错）、不负责生成新 KP（S01负责）、不负责判断数据是否「好」（只判断是否「合规」）。

### 1.2 10 条铁规（R1-R10）

| 铁规 | 名称 | 检查逻辑 | 级别 |
|------|------|---------|------|
| R1 | 算分程序计算 | levelScore = mapped_value(curriculumLevel)，A=3/B=6/C=9/D=12 | BLOCK |
| R2 | 三文件交叉对账 | knowledge.json ↔ levels.json ↔ graph.json 的 code 集完全一致 | BLOCK |
| R3 | 字段区分度 | 新增字段 >80% 相同值 → WARN | WARN |
| R4 | 错因必须引用 | error_taxonomy.json 每个 code 在 bindings 中 ≥1 次 | BLOCK |
| R5 | 必填字段完整 | 每个 KP 必须包含 17 个字段 | BLOCK |
| R6 | 级别标签一致 | knowledge.json 的 curriculumLevel 与 levels.json 一致 | BLOCK |
| R7 | A级绑定≥3 | 每个 A 级 KP 在 bindings 中 ≥3 个错因 | BLOCK |
| R8 | 双向边一致 | 若 A.prerequisites 含 B 则 B.next 必含 A | BLOCK |
| R9 | 题库矩阵覆盖 | 每个 KP 的 questionTypes 非空 | BLOCK |
| R10 | scope 一致性 | 每个 KP 的 subjectCode/stage/contentType 等与文件 scope 一致 | BLOCK |

### 1.3 门禁执行流程

```bash
cd acceptance-artifacts && python3 data_quality_gate.py
```

任何 BLOCK → 禁止提交。9/10 PASS 不算过——必须是 10/10 PASS。

**AI-MUTABLE 阈值**：
- d5_r3_distribution_threshold = 0.8  <!-- AI-MUTABLE: d5_r3_distribution_threshold, type=float, range=[0.5, 0.95] -->
- d5_auto_upgrade_warn_count = 3  <!-- AI-MUTABLE: d5_auto_upgrade_warn_count, type=int, range=[2, 10] -->

### 1.4 当前能力缺口

1. 门禁是静态规则——不会根据数据规模自动调整阈值
2. 没有历史趋势检测——连续 3 次门禁通过的代码可能在积累隐患
3. WARN 仅打印日志——无升级机制（连续3次WARN应升级为BLOCK）
4. 新学科门禁未建立——R5 的 17 字段是数学专属的，语文/英语需要各自的必填字段集

### 1.5 禁用清单

- 禁止手动修改 data_quality_gate.py 的铁规逻辑而不走门禁进化流程
- 禁止提交被 BLOCK 的代码（CI 硬拦截）
- 禁止删除铁规（只能新增 R11+ 或标记退役）

## 第二层：元能力

### 2.1 进化路线图

单维度：从「静态门禁」到「自适应门禁+趋势预警」。

| 阶段 | 触发条件 | 系统能力 |
|------|---------|---------|
| 1（当前） | — | 10 条静态铁规，BLOCK/WARN 二元判定 |
| 2 | 门禁连续运行≥6个月，≥100次执行 | 新增趋势检测——R11 趋势门禁（连续3次 WARN→升级BLOCK）。新增 R12 回归检测（本次门禁通过但上次 BLOCK 的项是否退步） |
| 3 | 新学科接入（语文/英语） | 门禁自动适配——R5 必填字段集按 contentType/levelingFramework 动态切换 |
| 4 | 阶段3稳定≥6个月 | 门禁阈值自动调节——R3 区分度阈值按数据分布动态调整 |

**退化降级**：R11/R12 连续误报（人工撤销≥3次）→回退到阶段1。

### 2.2 不可修改边界

| 类型 | 项目 | 原因 |
|------|------|------|
| 防线本身 | R1/R2/R5/R8/R10 | 数据质量的最后防线，任何修改可能导致假数据通过 |
| 执行规则 | BLOCK 必须拦截提交 | 不可降级为 WARN |
| 执行规则 | 门禁不可被绕行 | CI 管道无豁免通道 |

### 2.3 间接修改边界（量化标准）

| 差异 | 处置 |
|------|------|
| <5% | 正常推进 |
| 5-15% | 审查员审核 |
| 15-30% | 产品经理签批 |
| >30% | 视为越界直接拒绝 |

### 2.4 应急熔断：轻量（暂停新增铁规）/ 子系统（全部门禁暂停进化，铁规照常执行）/ 全系统

### 2.5 异常进化处置：连续失败→60天冻结该铁规修改 / 长期停滞→信用分-5/月 / 门禁脚本崩溃→回退到上一可用版本

### 2.6 安全执行管道

影子模式（历史数据回放）→ A/B 测试 → 全量上线。每层有北极星前置检查。

### 2.7 环境差异化

| 环境 | 进化权限 | 北极星校验 |
|------|---------|-----------|
| 开发 | 全级别允许 | 仅底线信号 |
| 测试 | 低/中自动，高风险签批 | 全三层但负信号仅WARN |
| 影子 | 仅低风险 | 全三层但仅可验证信号 |
| 正式 | 低风险事后报告 | 全三层，任何恶化直接拦停 |

### 2.8 G1-G7卡点预埋

| 卡点 | S05对应位置 | 失败行为 |
|------|-----------|---------|
| G1 | 铁规修改提交前 | 拒绝提交 |
| G4 | commit前门禁通过 | CI阻断 |
| G6 | 新铁规影子模式前 | 拒绝进入管道 |

第四层：北极星

### 4.1 三层信号

**正信号**：门禁拦截到真实数据错误（不是误报）、新铁规有效防止了某类历史高发问题

**负信号**：误报率上升（BLOCK 了很多但其实没问题的提交，导致开发效率下降）、门禁执行耗时超过 30 秒

**底线信号**：门禁被绕过、铁规被私下修改、门禁脚本被替换

### 4.2 对端指标

| 优化指标 | 对端指标 |
|---------|---------|
| 门禁严格度（BLOCK率） | 开发提交效率不得下降 >50% |
| 新铁规数量 | 误报率不得超过 5% |

**语义一致性声明**：已通过。

### 大白话

覆盖门禁就是数据质量的「出厂质检」。10 条铁规是十道关卡——任何 KP 数据要上线，先过完这十关。R1 查分数是不是程序算的（不能手填），R2 查三个文件的编号能不能对上，R4 查每个错因有没有实际绑到知识点上，R8 查知识图谱的边是不是双向的——你连了 A→B 就必须有 B←A。

门禁不过就拦截提交，没商量。CI 管道自动跑，10/10 全绿才放行。门禁本身也要进化——从静态规则到自适应规则。比如连续 3 次 WARN 应该自动升级成 BLOCK（说明这个问题在积累），这是 R11 要做的事。

不可修改的底线：门禁规则不能被绕过。你不能说「这次 BLOCK 我先绕过去下次再修」。没有豁免通道。门禁是假数据的最后防线，以前旧版 6 维体系 73.8% 数据是假的——就是因为没有程序化门禁。

门禁执行脚本 data_quality_gate.py 本身也受保护——改它等于改质检标准，必须走完整进化流程+人工签批。
