## 第一层：内功 — 门禁系统的自我进化能力

### 1.1 系统定义与边界

**这个系统负责什么**：覆盖门禁是芽芽数据质量的最后防线。任何 knowledge.json / levels.json / knowledge_graph.json 的变更必须通过门禁才能提交。当前 10 条铁规（R1-R10）由 data_quality_gate.py 脚本自动化执行。

**这个系统不负责什么**：不负责修复数据（只报错）、不负责生成新 KP（S01负责）、不负责判断数据是否「好」（只判断是否「合规」）。

### 1.2 10 条铁规（R1-R10）

| 铁规 | 名称 | 检查逻辑 | 级别 |
|------|------|---------|------|
| R1 | 算分程序计算 | levelScore = mapped_value(curriculumLevel)，A=3/B=6/C=9/D=12 | BLOCK |
| R2 | 三文件交叉对账 | knowledge.json ↔ levels.json ↔ graph.json 的 code 集完全一致 | BLOCK |
| R3 | 字段区分度 | 新增字段 >80% 相同值 → WARN | WARN |
| R4 | 错因必须引用 | error_taxonomy.json 每个 code 在 bindings 中 ≥1 次 | BLOCK |
| R5 | 必填字段完整 | 每个 KP 必须包含 15 个必填字段 | BLOCK |
| R6 | 级别标签一致 | knowledge.json 的 curriculumLevel 与 levels.json 一致 | BLOCK |
| R7 | A级绑定≥3 | 每个 A 级 KP 在 bindings 中 ≥3 个错因 | BLOCK |
| R8 | 双向边一致 | 若 A.prerequisites 含 B 则 B.next 必含 A | BLOCK |
| R9 | 题库矩阵覆盖 | 每个 KP 的 questionTypes 非空 | BLOCK |
| R10 | scope 一致性 | 每个 KP 的 subjectCode/stage/contentType 等与文件 scope 一致 | BLOCK |

### 1.3 门禁执行流程

```bash
cd acceptance-artifacts && python3 data_quality_gate.py
```

任何 BLOCK → 禁止提交。10/10 PASS 才算过——9/10 PASS 不算过。

### 1.4 当前能力缺口

1. 门禁是静态规则——不会根据数据规模自动调整阈值
2. 没有历史趋势检测——连续 3 次门禁通过的代码可能在积累隐患
3. WARN 仅打印日志——无升级机制（连续3次WARN应升级为BLOCK）
4. 新学科门禁未建立——R5 的 15 字段是数学专属的，语文/英语需要各自的必填字段集

### 1.5 禁用清单

- 禁止手动修改 data_quality_gate.py 的铁规逻辑而不走门禁进化流程
- 禁止提交被 BLOCK 的代码（CI 硬拦截）
- 禁止删除铁规（只能新增 R11+ 或标记退役）

## 第二层：元能力 — 安全进化机制

### 2.1 自我进化路线图

单维度：从「静态门禁」到「自适应门禁+趋势预警」。

**数据闭环声明**：门禁系统直接读取 data_quality_gate.py 的 JSON 输出（BLOCK/WARN/PASS 判定列表）。此输出格式已在 10 号数据模型的 GateResult 表中定义。门禁历史数据（每次 commit 的 gate 通过率）由 09 号事件协议中的 `gate_executed` 事件记录。阶段2 所需的「≥100次执行」数据由 CI 管道自动积累，不需要额外定义数据采集中间阶段。

| 阶段 | 触发条件 | 系统能力 |
|------|---------|---------|
| 1（当前） | — | 10 条静态铁规，BLOCK/WARN 二元判定 |
| 2 | 门禁连续运行≥6个月，≥100次执行（由09号 gate_executed 事件计数） | 新增趋势检测——R11 趋势门禁（连续3次 WARN→升级BLOCK）。新增 R12 回归检测（本次门禁通过但上次 BLOCK 的项是否退步） |
| 3 | 新学科接入（语文/英语） | 门禁自动适配——R5 必填字段集按 contentType/levelingFramework 动态切换 |
| 4 | 阶段3稳定≥6个月 | 门禁阈值自动调节——R3 区分度阈值按数据分布动态调整 |

**退化降级规则**：R11/R12 连续误报（人工撤销≥3次）→回退到阶段1。

**AI-MUTABLE 阈值**：
- d5_r3_distribution_threshold = 0.8  <!-- AI-MUTABLE: d5_r3_distribution_threshold, type=float, range=[0.5, 0.95] -->
- d5_auto_upgrade_warn_count = 3  <!-- AI-MUTABLE: d5_auto_upgrade_warn_count, type=int, range=[2, 10] -->

### 2.2 不可修改边界

| 类型 | 项目 | 不可修改的原因 | AI的直接后果 | AI的正确行为 |
|------|------|--------------|------------|-------------|
| 质量门禁核心防线 | R1/R2/R4/R5/R8/R10（共6条铁规不可禁用） | 数据质量的最后防线——任何修改可导致假数据通过。参考旧版6维体系73.8%教训 | 假数据污染全系统→诊断全错→策略全错→学生被误导 | 可新增辅助铁规（R11+），绝不禁用或放宽核心铁规 |
| 全局外键标识符 | 门禁脚本路径和函数签名 | CI/CD管道依赖 data_quality_gate.py 的固定接口 | 管道静默失败→假数据提交成功 | 脚本重构需保持函数签名不变+通过新流水线验证 |
| 外部法规定义 | R5 必填字段集（15字段中有6个来自课标/教材强制要求） | 字段缺失→数据在教育局验收时不合格 | 合规风险→产品被教育局驳回 | 新增字段可以，绝不删除课标强制字段 |
| 通信协议基础字段 | 09号事件 gate_executed 的事件类型名和字段结构 | CI/CD管道监听此事件决定是否放行代码 | 管道静默失败→假数据通过 | 事件结构调整需所有消费方同步更新 |
| 学生原始行为数据 | 门禁不得访问学生个体作答数据（仅检查数据文件结构，不检查数据值） | 数据隐私——门禁是 CI 脚本，不应有学生数据访问权限 | 隐私泄漏→合规风险 | 门禁仅读取数据文件的结构和元数据 |
| 北极星底线信号覆盖项 | 门禁不可被绕过——CI管道无豁免通道 | CI绕过→假数据提交→全系统数据腐败 | 一次绕过=门禁系统名存实亡 | 紧急情况下产品经理手工签批可临时跳过，但72小时内必须补齐门禁 |
| Shareable验证标准阈值 | 铁规 BLOCK 判定不可降级为 WARN | 假数据通过门禁的唯一入口 | BLOCK变WARN=门禁变告警箱→无人理会 | 阈值调整须经北极星检查+教研审核+产品签批 |

### 2.3 间接修改边界（量化标准）

| 修改类型 | 间接影响的不可修改项 | 机制 | 差异阈值 | 处置 |
|---------|--------------------|------|---------|------|
| 修改 R3 区分度阈值 | 门禁误报率/漏报率 | 阈值过宽松→异常数据逃逸。阈值过严→正常数据被误杀 | <5% | 正常推进 |
| 修改 R5 必填字段集 | 数据合规性 | 移除课标强制字段→教育局验收失败 | 5-15% | 审查员审核 |
| 新增铁规改变 BLOCK 数 | CI/CD管道效率 | 误报过多→开发效率崩溃 | 15-30% | 产品经理签批 |
| 删除/禁用核心铁规 | 数据腐败防线 | 防线缺口→假数据入口 | >30% | 视为越界直接拒绝 |

### 2.4 越界告警分级处置

| 级别 | 触发条件 | 处置 |
|------|---------|------|
| 轻 | AI读取不可修改项意图 | 退回+记录 |
| 中 | AI生成方案含修改不可修改项 | 冻结该方向进化7天+人工审核 |
| 重 | AI已执行修改不可修改项 | 永久封禁该方向+回滚 |

### 2.5 异常进化处置

| 异常 | 检测 | 处置 |
|------|------|------|
| 连续失败 | 同铁规连续3次被人工撤销 | 60天冻结该铁规修改 |
| 长期停滞 | 同阶段≥18教学月 | 信用分-5/月（最低0分） |
| 门禁脚本崩溃 | 门禁执行失败 | 回退到上一可用版本+告警 |

### 2.6 应急熔断（三级）

- 轻量级：暂停新增铁规，已有铁规照常执行
- 子系统：全部门禁暂停进化，铁规照常执行
- 全系统：全停+回滚未完成的

### 2.7 环境差异化

| 环境 | 进化权限 | 北极星校验 |
|------|---------|-----------|
| 开发 | 全级别允许 | 仅底线信号 |
| 测试 | 低/中自动，高风险签批 | 全三层但负信号仅WARN |
| 影子 | 仅低风险 | 全三层但仅可验证信号 |
| 正式 | 低风险事后报告 | 全三层，任何恶化直接拦停 |

### 2.8 内容与代码进化分离

- 内容进化：铁规判定标准/阈值调整→人工确认→灰度验证→全量更新
- 代码进化：门禁执行逻辑/新铁规算法→北极星管道
- 混合进化：同步改标准+算法→两套流程都过

### 2.9 安全执行管道

新铁规→影子模式（历史数据回放）→A/B测试→全量上线。每层有北极星前置检查节点。

### 3.2 自我进化执行方法

五类角色权限台账（含权限回收规则）：

| 角色 | 权限 | 回收规则 |
|------|------|---------|
| 芽芽自己 | 低风险自主（趋势分析、建议新铁规），无检查模块修改权 | 连续3次建议铁规被驳回→冻结该方向30天 |
| AI编程工具 | 仅读取+方案生成，不得直接修改 data_quality_gate.py | — |
| 产品经理 | 最高权限+签批+熔断+紧急豁免 | 豁免后72小时未补门禁→取消豁免权30天 |
| 教研人员 | 中风险审核——新铁规的教育合理性确认 | 连续3次确认的铁规被撤销→暂停审核权 |
| 审查员 | 标记通过/疑似/不通过 | 连续3次审查结论被推翻→重新培训 |

五步流程：信号采集（≥2周门禁运行数据）→方案生成→审核（三级权限）→安全执行（影子→A/B→全量）→记录验证（30天稳定期）。

**信用分三子分**：low_risk_score / mid_risk_score / high_risk_score。聚合=min(三子分)。初始值=50。任意子分<20→对应级别暂停。min<10→全系统暂停。月度上限+20分。长期停滞扣减-5/月（最低0分）。

**A/B测试分组规则**（验收标准 1.12）：新铁规上线→10%代码提交走新门禁（影子回放），其余走旧门禁。分组随机（按提交日期奇偶均匀分布）。显著性检验：p<0.05，最小样本量=50次提交（Cohen's d≥0.3）。分组不可逆：同一提交批次不切换门禁版本。

**元进化递归上限**（验收标准 1.12）：修改铁规判定标准（如将BLOCK降级为WARN的阈值）视为元进化操作。最大递归深度=2（修改→新标准化→再次修改→禁止第三次修改）。年度上限=1次元进化。每次元进化后重新通过四级门禁验收（验收标准 4.4）。

## 第四层：北极星

### 4.1 三层检测信号

**正信号**（权重 20%）：门禁拦截到真实数据错误（不是误报）、新铁规有效防止了某类历史高发问题

**负信号**（权重 30%）：误报率上升（BLOCK 了很多但其实没问题的提交，导致开发效率下降）、门禁执行耗时超过 30 秒

**底线信号**（权重 50%）：门禁被绕过、铁规被私下修改、门禁脚本被替换

北极星检查不可绕过——任何进化操作前必须通过三层信号全检。

### 4.2 对端指标矩阵

| 优化指标 | 对端指标（不能降） |
|---------|------------------|
| 门禁严格度（BLOCK率） | 开发提交效率不得下降 >50% |
| 新铁规数量 | 误报率不得超过 5% |
| 门禁执行速度 | 执行耗时不得超过 30 秒 |

### 4.3 月度自检

检查模块四道防线约束。门禁脚本自身需通过签名校验——防止被替换。

**语义一致性声明**：本文档的技术描述与大白话描述已通过语义一致性检测。

### 大白话

覆盖门禁就是数据质量的「出厂质检」。10 条铁规是十道关卡——任何 KP 数据要上线，先过完这十关。R1 查分数是不是程序算的（不能手填），R2 查三个文件的编号能不能对上，R4 查每个错因有没有实际绑到知识点上，R8 查知识图谱的边是不是双向的——你连了 A→B 就必须有 B←A。

门禁不过就拦截提交，没商量。CI 管道自动跑，10/10 全绿才放行。门禁本身也要进化——从静态规则到自适应规则。比如连续 3 次 WARN 应该自动升级成 BLOCK（说明这个问题在积累），这是 R11 要做的事。

不可修改的底线：门禁规则不能被绕过。你不能说「这次 BLOCK 我先绕过去下次再修」。没有豁免通道——紧急情况产品经理可以手工签批跳过，但 72 小时内必须补齐门禁。门禁是假数据的最后防线，以前旧版 6 维体系 73.8% 数据是假的——就是因为没有程序化门禁。

门禁执行脚本 data_quality_gate.py 本身也受保护——改它等于改质检标准，必须走完整进化流程+人工签批。CI 管道配置里写死了门禁脚本的路径和接口，动了路径等于门禁静默失效。
