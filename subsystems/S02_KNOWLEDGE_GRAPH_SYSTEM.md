## 第一层：内功 — 知识图谱的自我进化能力

### 1.1 系统定义与边界

**这个系统负责什么**：知识图谱系统维护知识点之间的依赖关系网络。负责「学会这个需要先会什么」——两层架构，上下游边，双向一致性。

**这个系统不负责什么**：不负责定义知识点是什么（S01）、不负责判断学生哪里不会（S07）、不负责发现新的教材内容（S16，待建设）。

### 1.2 两层架构

**学习路径层（用户可见）**：单元内部按教材顺序串联链条，不跨单元连线。前端展示为线性进度条——「孩子在学什么，下一个是什么」。

**诊断依赖层（引擎内部）**：1-6 年级全量织网，所有硬依赖都连上，跨单元跨年级。被 S07 诊断引擎用于溯源——沿依赖链向上追查根因。用户完全不可见。

### 1.3 硬依赖判定标准

四条标准，满足任一即连边：

1. **定义包含**：B 的定义/公式直接使用 A
2. **概念层级**：A 是抽象概念，B 是具体应用——没有 A 就无法理解 B
3. **操作前置**：解 B 必须先执行 A
4. **教材刚性顺序**：必须先复习 A 才能教 B

**优先级规则**（当多条标准给出不同结论时，按此顺序裁决）：
- 定义包含 > 概念层级 > 操作前置 > 教材刚性顺序
- 「定义包含」优先级最高——如果 B 的定义里直接用了 A，无论其他标准怎么判断，必须连边
- 「教材刚性顺序」优先级最低——因为教材版本差异可能导致假边，仅在此标准命中时需额外标注

**教材版本差异处理**：如果一条边仅在某版本教材下满足教材刚性顺序（如人教版要求复习 A 再教 B，但苏教版不要求），该边必须标注版本限定——在 edge 记录中增加 `textbookVersion: ["人教版"]`。未标注版本限定的教材刚性顺序边，默认仅对人教版生效。

软依赖不连边：启发关系、顺带提及、平行主题。

### 1.4 跨内容类型的硬依赖调整

| contentType | 硬依赖标准 | 示例 |
|---|---|---|
| textbook_kp | 四条标准全适用 | 通分→异分母分数加减法 |
| course_concept | 概念层级+课程大纲顺序 | 「矩阵的秩」→「线性方程组的解」 |

### 1.7 第一层禁用清单

- 禁止将软依赖连为硬依赖（导致诊断引擎追错根因）
- 禁止删除已有边而不做双向一致性检查
- 禁止手动修改图谱 JSON 文件而不跑 build 脚本
- 禁止忽略跨图引用报错

**检测手段**：build 脚本在每次合法生成后写入签名字段（`buildSignature` = 文件内容的 SHA256 hash，使用项目级密钥加盐）。CI 流水线在每次 commit 时校验：
1. 检查图谱 JSON 中 `buildSignature` 字段是否存在
2. 用同样算法重新计算 hash，与存储值对比
3. 不匹配 → CI 阻断，报「图谱文件疑似手动修改，请跑 build 脚本」
4. 替代方案（不依赖签名）：CI 直接跑一次 build 脚本，比对输出是否与当前 commit 一致——任何差异都是手动修改的证据

> **设计理由**（约束）：图谱是诊断引擎溯源的唯一依据。如果依赖关系错误——软依赖连成硬依赖或漏了真正的硬依赖——诊断引擎会追到错误根因。任何修改图谱边的操作必须经过跨样本验证（≥3个独立数据源验证教学效果）。

> **Phase 1 过渡说明**：以上「禁止手动修改图谱 JSON 文件而不跑 build 脚本」规则在 build 脚本实现后生效。当前 Phase 1 过渡期间（build 脚本未就绪，见 §1.6 第 6 项），图谱 JSON 文件暂由人工通过知识管理系统界面手动维护边关系，签名检测暂不生效，改为人工代码审查（见 §1.8 退化场景处置表）。build 脚本实现后此过渡方案自动废止。

### 1.9 设计演化推理链

> 本文档的架构设计和演化路线选择建立在以下核心推演基础上。每条推理链记录了关键决策的「为什么这样选」——帮助审查者和后续维护者理解设计意图，避免在不知情的情况下推翻已有推理。

**推理链一：为什么选两层架构（学习路径层+诊断依赖层）？**

最初考虑过三层方案：教材依赖层（按教材章节顺序连边）、认知依赖层（按思维发展规律连边）、考试依赖层（按历年考试频次连边）。但三层方案有两个问题：(1) 教材依赖层和认知依赖层的边大量重合——90% 以上的「先学 A 再学 B」关系在两个层面都成立，区分成本远高于收益；(2) 考试依赖层不是「依赖」而是「关联」——两个知识点同时出现在同一道题里不等于其中一个必须先掌握。最终选择两层：学习路径层（用户可见的教材顺序）和诊断依赖层（引擎内部的全量硬依赖网）。用户看到的是清晰的学习路径，引擎内部有完整的根因追溯能力。

**推理链二：为什么用四条硬依赖判定标准？**

四条标准对应四种不同的「必须先会 A 才能学 B」场景：(1) 定义包含——B 的定义里直接出现了 A（最硬，优先级最高）；(2) 概念层级——A 是抽象概念，B 是具体应用（第二硬）；(3) 操作前置——解 B 必须先执行 A 这个操作（第三硬）；(4) 教材刚性顺序——教材编写者认为必须先复习 A 才能教 B（最软）。四条标准的优先级顺序本身就是设计决策：定义包含 > 概念层级 > 操作前置 > 教材刚性顺序。为什么教材顺序排最后？因为教材版本差异可能导致假边（人教版要求先复习这个再教那个，苏教版不要求），所以教材顺序的边需要额外标注版本限定。

**推理链三：为什么跨内容类型需要差异化标准？**

知识点的 contentType 不同（教材知识点/课程概念/技能/知识点合集/能力素养），「依赖」的定义不同。教材知识点可以适用全部四条标准；技能类知识点只有「操作前置」最相关（先会装 requests 才能学用 requests）；能力素养类知识点是递进关系（能读数据报告→能从数据中提建议）。如果用同一套标准处理所有 contentType，会产生大量假边。

**推理链四：为什么用双向一致性约束（R8）？**

单向边的后果：诊断引擎沿 prerequisites 链上溯到某节点后，发现该节点的 next 列表不含反向边——溯源路径断裂，S07 静默失败给出错误根因。R8 的核心价值不是「让边好看对称」而是「确保每条依赖关系在两个方向上都是可追溯的」。这是诊断引擎溯源可靠性的最低保障。

**推理链五：为什么信用分初始值设 50？**

0 分意味着第一周几乎不能做任何进化操作（信用分低于门槛→暂停），系统冷启动后至少浪费 2.5 周等待信用积累。100 分意味着系统一启动就拥有全部权限——出事故后才能扣分，等同于前置条件是「撞墙才知道痛」。50 分是折中：系统默认有基本信誉，但需要持续证明自己。与 S01 保持一致，避免子系统间信用分规则冲突。

**推理链六：为什么 A/B 测试豁免班级隔离？**

DOCUMENTATION_STANDARDS v2.1 原则七规定 A/B 分组以班级为最小单位（防止同班学生交叉影响——同一教师教 A 组和 B 组学生会产生溢出效应）。但图谱 A/B 测试验证的是「改了依赖边后，S07 诊断引擎的溯源准确率是否提升」——图谱是引擎内部件，边的修改不改变教师的教学行为、不出现在学生界面上，因此不存在同班学生之间的交叉影响。学生看不到自己走的是新图谱还是旧图谱，教师也看不到诊断引擎内部溯源路径的变化。此场景豁免班级隔离要求。

**推理链七：为什么阶段 1 和阶段 2 之间加一个 1.5？**

从「人工判定依赖」直接跳到「系统自动发现依赖」缺少一个关键前提——系统先能「看到」学生在未掌握前置知识的情况下尝试新知识时会怎样。阶段 1.5 就是做这个数据采集基础设施的：埋点记录「学生在未掌握前置 A 的情况下尝试 B」的行为数据。没有这个数据积累，阶段 2 的自动分析就是空中楼阁。这个阶段的触发条件是人工依赖（需 S07/S09 开发团队实现埋点能力），反映了基础设施先行的设计原则。

**推理链八：为什么禁止手动修改图谱 JSON 文件？**

图谱是诊断引擎溯源的唯一依据。如果依赖关系错误——软依赖连成硬依赖或漏了真正的硬依赖——诊断引擎会追到错误根因，导致学生收到错误的学习建议，家长收到错误的能力报告。手动修改 JSON 文件绕过了 build 脚本的所有校验（双向一致性检查、跨图引用完整性校验、签名检测）。因此引入 build 脚本签名机制：每次合法生成后写入 buildSignature，CI 在每次 commit 时校验签名一致性。Phase 1 过渡期间（build 脚本未就绪），改为人工代码审查。

### 2.10 安全执行管道

**三层管道**（原则十二 + 原则五安全约束）：

**第一层：影子模式**——在历史数据上回放图谱变更的影响。
- 数据脱敏：学生 ID 哈希化（SHA256 + 项目级盐）、去除设备信息/IP/地理位置、时间戳精确到教学周（周一 00:00）、单个学生数据量 < 阈值时自动合并到同年级匿名桶
- 前置检查：北极星底线信号（图谱完整性、跨图引用无悬空）通过后方可进入影子模式
- 失败处置：影子模式中检测到溯源准确率下降 > 5%→拒绝进入下一层

**第二层：A/B 测试**——小流量真实验证。
- 分组规则：实验组 5%（走新图谱）/对照组 5%（走旧图谱，用于与实验组对比统计），其余 90% 走旧图谱（不参与对比统计）。按 studentId hash 取模分流，同一学生不可同时出现在两组
- 最小样本量：每组 ≥50 个活跃学生（活跃 = 近 7 天内 ≥3 次答题）
- 观察周期：≥2 教学周（10 个教学日）
- 显著性检验：双尾 t 检验，p < 0.05 且效应量 Cohen's d > 0.2
- 前置检查：北极星全三层通过

> **设计理由——班级隔离豁免**：DOCUMENTATION_STANDARDS v2.1 原则七规定 A/B 分组以班级为最小单位（防止同班学生交叉影响——同一教师教 A 组和 B 组学生会产生溢出效应）。但图谱 A/B 测试验证的是「改了依赖边后，S07 诊断引擎的溯源准确率是否提升」——图谱是引擎内部件，边的修改不改变教师的教学行为、不出现在学生界面上，因此不存在同班学生之间的交叉影响。学生看不到自己走的是新图谱还是旧图谱，教师也看不到诊断引擎内部溯源路径的变化。此场景豁免班级隔离要求。

**第三层：全量上线**——替换正式图谱。
- 前置检查：PSI < 0.1 + 北极星无恶化 + A/B 测试通过
- 回滚准备：全量前保留旧图谱快照，上线后监控 30 天，任意指标恶化→自动回滚

- 混合进化：同时改边和算法→两套流程都过

**元进化递归上限**：图谱系统修改自身进化路线图（如新增第六维度、修改阶段触发方式）视为元进化——高风险操作。递归深度按动作计次：(1) 系统第一次修改路线图的参数/触发条件 = 深度 1，允许自动执行但需经安全管道（影子→A/B→全量）；(2) 修改「路线图修改规则」本身（即元进化的元进化——如让 AI 自己决定是否可以跳过安全管道中的某层）= 深度 2，永远禁止自动执行，仅限人工硬编码启动。G3 卡点在每次元进化触发前检查当前递归深度：深度 ≥ 2 → 拒绝执行。任何自动触发的元进化必须经过：北极星全三层通过 + 人工签批 + G3 卡点。

### 大白话

**系统职责**：知识图谱就是知识点之间的「谁先谁后」关系表。两层各有用处：学习路径层像教材目录——告诉你先学什么后学什么，学生和家长看到的就是这个顺序。诊断依赖层像医生看病的病因追溯链——你头疼，医生不会直接开头疼药，而是查血压、查颈椎，因为头疼可能是高血压引起的。诊断引擎也一样，学生卡在分数乘法，引擎沿依赖链往上查：是不是分数意义没搞懂？是不是通分有问题？但诊断引擎不会无限追下去，最多查三代——就像医生不会因为头疼去查你的 DNA 序列。再往上就不追了，而是建议安排专项基础测试。

**核心设计**：硬依赖就像盖房子的地基——没有地基不能盖一楼。软依赖就像两个相邻的店铺——挨着但不互相依赖。不要把两个挨着的知识点自动连一条线，除非满足那四条标准之一。四条标准之间有优先级：定义里直接用了对方→最高；概念层级→第二；操作前置→第三；教材顺序→第四。为什么教材顺序排最后？因为不同版本的教材顺序不一样——人教版要先复习这个再教那个，苏教版未必。这种「仅某个教材版本才有的依赖」需要标注清楚。

**进化路线图**：进化路线图是一个维度：从「人手工画线」到「系统自己发现隐藏的依赖关系」。但这中间缺一个关键步骤——系统得先能「看到」学生在没掌握前置知识的情况下尝试新知识时会怎样。这就像医生要先装好心电监护仪，才能判断某种药对心脏的影响。所以阶段 1 和阶段 2 之间加了一个 1.5：先把数据采集埋点做好。有了数据积累，系统才能分析——学了 A 的孩子学 B 是不是更快？没用 A 的学 B 是不是更容易失败？如果统计显著，标记为候选隐性依赖，人工确认后在教学实验中验证，验证有效就加入图谱。但系统也会犯错误——如果连续验证失败，就自动退回到更保守的阶段。

**不可修改边界**：不可修改的就是三件大事加四个防线。三件大事：一是「双向边必须一致」——你连了 A→B，就必须有 B←A，单边会让诊断引擎在溯源时走进死胡同。二是「跨图引用不能悬空」——高年级的知识点如果指向前置的低年级知识点，那个低年级知识点必须真的存在。三是「不能乱贴标签」——图谱只能按知识点逻辑关系连边，不能说「因为这个孩子 XX 属性所以这部分学不好」。四个防线：全局编号不能改、通信数据结构不能变、学生原始数据不能碰、质量门禁不能松。为了防止有人偷懒直接改 JSON 文件而不跑 build 脚本，build 脚本每次会留下一个数字签名（像文件的「防伪标识」）。CI 机器人在每次提交代码时检查签名——签名对不上就说明有人手动改过，直接拦下来。

**自我进化执行方法**：推动系统从一个阶段进化到下一个阶段，需要同时满足两个条件：时间够了 + 信誉够了。就像考驾照——不光要学满学时，还得考试通过。光耗时间但老出事故不行，光分数高但时间没学够也不行。系统操作分五步走：先攒两周数据、再生成方案、然后三级审核、接着三层管道上线（先在历史数据上回放看有没有问题→用 5% 学生做 A/B 测试→全量上线）、最后观察 30 天稳定期。五类角色各管各的：芽芽干低风险的、AI 编程工具只能看和提方案不能动手、产品经理管高风险决策、教研老师审内容、审查员标通过不通过。

**自我进化执行方法（续）**：A/B 测试就像新药临床试验——找两拨水平差不多的学生（按学号哈希分，保证公平），一拨用新图谱一拨用旧图谱，两周后看用新图谱的学生诊断是不是更准。但图谱验证不像课堂实验——改图谱边是发生在引擎内部的，学生看不到、教师感觉不到，所以按学生个体分就够了，不需要按班级隔离（不像课堂策略测试，同一教师教两组学生会有溢出效应）。影子模式更安全——在历史数据上「重放」一遍，看如果当初用了新图谱会怎样。但影子模式前必须做数据脱敏——学生真名换成哈希码、去掉地理位置和设备信息、时间只保留到教学周——就像病历去隐私后才能做研究。

**异常进化处置**：系统也会生病。连续三次失败（像连续发烧），自动冻结 60 天——给系统「静养」时间。长期停滞（同一个阶段待了 18 个月以上），信用分每月自动扣 5 分——像驾照长期不年审会降级。路线图矛盾（比如发现自己在往终局的相反方向走），整个维度冻结——像导航发现走错了先停下来，别越走越远。

**应急熔断**：三级熔断像电闸。轻度——只关高风险进化，低风险的继续（像跳了一个房间的电闸，其他房间还有电）。中度——图谱全部进化暂停（像跳了整个楼层的电闸）。重度——全系统停 + 回滚（像跳了整栋楼的电闸）。

**进化信用分**：信用分像三个独立的银行账户——低风险一个、中风险一个、高风险一个，各算各的。但取钱时看最少的那个——一个账户空了，所有账户都受限。这就是为什么初始化给 50 分——既不是不信任（0 分意味着至少一个月不能做任何事），也不是盲目信任（100 分意味着出事故才扣）。50 分 = 「系统默认有基本信誉，但需要持续证明自己」。

**内容与代码进化分离**：加一条边（内容）像在菜单上加一道菜——教研审核就行，不用过安检。改判定算法（代码）像改菜刀——要用安检仪（北极星检查）过一遍。同时改两样（混合）像换厨房——两套流程都得过。系统改自己的进化规则（元进化）是最高风险——像改宪法——递归深度不能超过两层，再深只能人工启动。
