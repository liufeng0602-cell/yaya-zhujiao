## 第一层：内功 — 知识图谱的自我进化能力

### 1.1 系统定义与边界

**这个系统负责什么**：知识图谱系统维护知识点之间的依赖关系网络。负责「学会这个需要先会什么」——两层架构，上下游边，双向一致性。

**这个系统不负责什么**：不负责定义知识点是什么（S01）、不负责判断学生哪里不会（S07）、不负责发现新的教材内容（S16）。

### 1.2 两层架构

**学习路径层（用户可见）**：单元内部按教材顺序串联链条，不跨单元连线。前端展示为线性进度条——「孩子在学什么，下一个是什么」。

**诊断依赖层（引擎内部）**：1-6 年级全量织网，所有硬依赖都连上，跨单元跨年级。被 S07 诊断引擎用于溯源——沿依赖链向上追查根因。用户完全不可见。

### 1.3 硬依赖判定标准

四条标准，满足任一即连边：

1. **定义包含**：B 的定义/公式直接使用 A
2. **教材刚性顺序**：必须先复习 A 才能教 B
3. **操作前置**：解 B 必须先执行 A
4. **概念层级**：A 是抽象概念，B 是具体应用——没有 A 就无法理解 B

「定义包含」优先于「概念层级」——如果 B 的定义里直接用了 A，归为定义包含。概念层级适用于 B 的定义没直接引用 A 但学习 B 必须先理解 A。

软依赖不连边：启发关系、顺带提及、平行主题。

### 1.4 跨内容类型的硬依赖调整

| contentType | 硬依赖标准 | 示例 |
|---|---|---|
| textbook_kp | 四条标准全适用 | 通分→异分母分数加减法 |
| course_concept | 概念层级+课程大纲顺序 | 「矩阵的秩」→「线性方程组的解」 |
| skill | 实操前置条件 | 「安装requests」→「用requests发GET请求」 |
| topic | 内容引用关系 | 「Transformer注意力机制」→「多头注意力变体」 |
| competency | 能力等级递进 | 「能读懂数据报告」→「能从数据中提业务建议」 |

### 1.5 当前数据规模

- G5-G6：174 节点有前置边，162 节点有后续边
- G1-G4：162 节点有前置边，156 节点有后续边
- 跨图引用正常（G5-G6 节点的前置指向 G1-G4 节点）

图谱构建脚本：`scripts/build_knowledge_graph.py`，四步全量重建：
1. 重置所有边，按 knowledge.json 单元排序建单元内链
2. 加跨单元硬依赖（约12条）
3. 加跨年级硬依赖（约48条，覆盖1-6年级）
4. 双向一致性检查（自动修复）

### 1.6 当前能力缺口（它还不会什么）

1. 不会自动发现隐性依赖——所有硬依赖靠人工四条标准判定
2. 不会感知知识点变化——S01 拆合 KP 后图谱需手动重建
3. 不会跨学科连线——数学和物理之间的空间推理传导关系未被发现
4. 图谱边没有权重——依赖强度（强/中/弱）未被量化

### 1.7 第一层禁用清单

- 禁止将软依赖连为硬依赖（导致诊断引擎追错根因）
- 禁止删除已有边而不做双向一致性检查
- 禁止手动修改图谱 JSON 文件而不跑 build 脚本
- 禁止忽略跨图引用报错

> **设计理由**（约束）：图谱是诊断引擎溯源的唯一依据。如果依赖关系错误——软依赖连成硬依赖或漏了真正的硬依赖——诊断引擎会追到错误根因。任何修改图谱边的操作必须经过跨样本验证（≥3个独立数据源验证教学效果）。

## 第二层：元能力 — 安全进化机制

### 2.1 自我进化路线图

图谱系统只有一个独立维度：从「人工判定依赖」到「系统自动发现依赖+自适应权重」。

| 阶段 | 触发条件 | 系统能力 |
|------|---------|---------|
| 1（当前） | — | 所有依赖由人工用四条标准逐对判定 |
| 2 | S01 维度一阶段2后（有 observations 数据） | 系统分析「掌握 A 之后 B 的掌握速度变化」+「未掌握 A 但尝试 B 的失败率」。统计显著（p < 0.01）标记为候选隐性依赖 |
| 3 | ≥5 条候选被人工确认+≥3 条在教学实验中验证有效（干预组 vs 对照组 p<0.05） | 自动将中置信度隐性依赖加入诊断依赖层（标记 source: auto_discovered）。高置信度直接加入学习路径层 |
| 4 | 阶段3稳定运行≥12个月 | 系统自己发现跨学科能力传导规律，记录为跨域依赖 |

**退化降级规则**：连续3次隐性依赖验证失败→回退到阶段2，p值阈值收紧20%。

**AI-MUTABLE 阈值**：

- d4_stage3_min_confirmed_deps = 5  <!-- AI-MUTABLE: d4_stage3_min_confirmed_deps, type=int, range=[3, 20] -->

### 2.2 不可修改边界

| 类型 | 项目 | 原因 | AI的直接后果 |
|------|------|------|------------|
| 数据完整性 | 图谱 JSON 的双向一致性约束 | R8 铁规——若A.prerequisites含B，则B.next必含A | 单向边导致诊断引擎溯源路径断裂 |
| 全局外键 | knowledgeCode | 图谱以 knowledgeCode 定位所有节点 | 全局数据不一致 |
| 质量门禁 | R8双向边一致、R10 scope一致性 | 数据腐败防线 | 假边通过校验 |

### 2.3 间接修改边界（量化标准）

**modificationHistory 五字段JSON结构**（每次边变更后写入）：
```json
{
  "modificationId": "edge_mod_001",
  "startedAt": "<ISO8601>",
  "endedAt": "<ISO8601>",
  "rollbackAt": null,
  "description": "新增M5S2→M4S3隐性依赖边",
  "confoundingEvents": []
}
```

### 2.4 间接修改边界（量化标准）

| 修改类型 | 差异阈值 | 处置 |
|---------|---------|------|
| 修改硬依赖判定标准 | <5% | 正常推进 |
| 修改四标准之一 | 5-15% | 审查员审核 |
| 放宽硬依赖范围 | 15-30% | 产品经理签批 |
| 移除某条标准 | >30% | 视为越界直接拒绝 |

### 2.4 异常进化处置

| 异常 | 检测 | 处置 |
|------|------|------|
| 连续失败 | 同维度连续3次回滚 | 60天冻结 |
| 长期停滞 | 同阶段≥18教学月 | 信用分-5/月（最低0分） |
| 路线图矛盾 | 终局冲突或阶段倒退 | 全维度冻结 |

### 2.5 应急熔断（三级）

- 轻量级：暂停高/中风险进化，低风险继续
- 子系统：图谱全部进化暂停
- 全系统：全停+回滚未完成的

### 2.6 环境差异化

| 环境 | 进化权限 | 北极星校验 |
|------|---------|-----------|
| 开发 | 全级别允许 | 仅底线信号 |
| 测试 | 低/中自动，高风险签批 | 全三层但负信号仅WARN |
| 影子 | 仅低风险 | 全三层但仅可验证信号 |
| 正式 | 低风险事后报告 | 全三层，任何恶化直接拦停 |

### 2.7 内容进化与代码进化

- 内容进化：新增/删除图谱边、调整依赖关系（教研审核→灰度→全量）
- 代码进化：修改硬依赖判定算法、修改 build 脚本（路线图触发→北极星→安全管道）
- 混合进化：同时改边和算法→两套流程都过

### 3.2 自我进化执行方法

五类角色权限台账：芽芽自己（低风险自主，无检查模块修改权）/ AI编程工具（仅读取+方案生成）/ 产品经理（最高权限+签批+熔断）/ 教研人员（中风险审核）/ 审查员（标记通过/疑似/不通过）。

五步流程：信号采集（≥2周教学日）→方案生成→审核（三级权限）→安全执行（影子→A/B→全量）→记录验证（30天稳定期）。

**信用分三子分**：low_risk_score / mid_risk_score / high_risk_score 三本独立账。聚合=min(三子分)。任意子分<20→对应级别暂停。min<10→全系统暂停。月度上限+20分。

## 第四层：北极星

### 4.1 三层检测信号

**正信号**：溯源准确率上升（干预组vs对照组p<0.05）、图谱覆盖率提升、隐性依赖被验证有效

**负信号**：过度发现（候选依赖>KP总数的30%）、溯源路径过长导致诊断延迟

**底线信号**：图谱边被篡改无审计、数据完整性破坏

| 优化指标 | 对端指标（不能降） |

| 图谱覆盖率 | 诊断引擎溯源耗时不得超过2秒 |
| 依赖强度量化精度 | 诊断准确率不得下降 |

|------|-----------|---------|

### 大白话

知识图谱就是知识点之间的「谁先谁后」关系表。两层各有用处：学习路径层像教材目录——告诉你先学什么后学什么，学生和家长看到的就是这个顺序。诊断依赖层像医生看病的病因追溯链——你头疼，医生不会直接开头疼药，而是查血压、查颈椎，因为头疼可能是高血压引起的。诊断引擎也一样，学生卡在分数乘法，引擎沿依赖链往上查：是不是分数意义没搞懂？是不是通分有问题？

硬依赖就像盖房子的地基——没有地基不能盖一楼。软依赖就像两个相邻的店铺——挨着但不互相依赖。不要把两个挨着的知识点自动连一条线，除非满足那四条标准之一。

进化路线图是一个维度：从「人手工画线」到「系统自己发现隐藏的依赖关系」。系统会观察数据——学了 A 的孩子学 B 是不是更快？没用 A 的学 B 是不是更容易失败？如果统计显著，标记为候选隐性依赖，人工确认后在教学实验中验证，验证有效就加入图谱。但系统也会犯错误——如果连续验证失败，就自动退回到更保守的阶段。

不可修改的就是「双向边必须一致」——你连了 A→B，就必须有 B←A。单边会让诊断引擎在溯源时走进死胡同。
