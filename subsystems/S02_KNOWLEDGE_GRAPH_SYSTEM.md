## 第一层：内功 — 知识图谱的自我进化能力

### 1.1 系统定义与边界

**这个系统负责什么**：知识图谱系统维护知识点之间的依赖关系网络。负责「学会这个需要先会什么」——两层架构，上下游边，双向一致性。

**这个系统不负责什么**：不负责定义知识点是什么（S01）、不负责判断学生哪里不会（S07）、不负责发现新的教材内容（S16）。

### 1.2 两层架构

**学习路径层（用户可见）**：单元内部按教材顺序串联链条，不跨单元连线。前端展示为线性进度条——「孩子在学什么，下一个是什么」。

**诊断依赖层（引擎内部）**：1-6 年级全量织网，所有硬依赖都连上，跨单元跨年级。被 S07 诊断引擎用于溯源——沿依赖链向上追查根因。用户完全不可见。

### 1.3 硬依赖判定标准

四条标准，满足任一即连边：

1. **定义包含**：B 的定义/公式直接使用 A
2. **概念层级**：A 是抽象概念，B 是具体应用——没有 A 就无法理解 B
3. **操作前置**：解 B 必须先执行 A
4. **教材刚性顺序**：必须先复习 A 才能教 B

**优先级规则**（当多条标准给出不同结论时，按此顺序裁决）：
- 定义包含 > 概念层级 > 操作前置 > 教材刚性顺序
- 「定义包含」优先级最高——如果 B 的定义里直接用了 A，无论其他标准怎么判断，必须连边
- 「教材刚性顺序」优先级最低——因为教材版本差异可能导致假边，仅在此标准命中时需额外标注

**教材版本差异处理**：如果一条边仅在某版本教材下满足教材刚性顺序（如人教版要求复习 A 再教 B，但苏教版不要求），该边必须标注版本限定——在 edge 记录中增加 `textbookVersion: ["人教版"]`。未标注版本限定的教材刚性顺序边，默认仅对人教版生效。

软依赖不连边：启发关系、顺带提及、平行主题。

### 1.4 跨内容类型的硬依赖调整

| contentType | 硬依赖标准 | 示例 |
|---|---|---|
| textbook_kp | 四条标准全适用 | 通分→异分母分数加减法 |
| course_concept | 概念层级+课程大纲顺序 | 「矩阵的秩」→「线性方程组的解」 |
| skill | 实操前置条件 | 「安装requests」→「用requests发GET请求」 |
| topic | 内容引用关系 | 「Transformer注意力机制」→「多头注意力变体」 |
| competency | 能力等级递进 | 「能读懂数据报告」→「能从数据中提业务建议」 |

### 1.5 当前数据规模

- G5-G6：174 节点有前置边，162 节点有后续边
- G1-G4：162 节点有前置边，156 节点有后续边

图谱构建脚本：`scripts/build_knowledge_graph.py`，四步全量重建：

2. 加跨单元硬依赖（约12条）
3. 加跨年级硬依赖（约48条，覆盖1-6年级）

### 1.6 当前能力缺口（它还不会什么）

1. 不会自动发现隐性依赖——所有硬依赖靠人工四条标准判定
2. 不会感知知识点变化——S01 拆合 KP 后图谱需手动重建
3. 不会跨学科连线——数学和物理之间的空间推理传导关系未被发现
4. 图谱边没有权重——依赖强度（强/中/弱）未被量化
5. 不会跨图引用完整性校验——当前 build 脚本的 R8 双向检查仅在单图内执行，G5-G6 指向 G1-G4 的边不校验目标节点是否存在。如果 G1-G4 删除了某个节点，G5-G6 的 prerequisites 就会变成悬空引用

### 1.7 第一层禁用清单

- 禁止将软依赖连为硬依赖（导致诊断引擎追错根因）
- 禁止删除已有边而不做双向一致性检查
- 禁止手动修改图谱 JSON 文件而不跑 build 脚本
- 禁止忽略跨图引用报错

**检测手段**：build 脚本在每次合法生成后写入签名字段（`buildSignature` = 文件内容的 SHA256 hash，使用项目级密钥加盐）。CI 流水线在每次 commit 时校验：
1. 检查图谱 JSON 中 `buildSignature` 字段是否存在
2. 用同样算法重新计算 hash，与存储值对比
3. 不匹配 → CI 阻断，报「图谱文件疑似手动修改，请跑 build 脚本」
4. 替代方案（不依赖签名）：CI 直接跑一次 build 脚本，比对输出是否与当前 commit 一致——任何差异都是手动修改的证据

> **设计理由**（约束）：图谱是诊断引擎溯源的唯一依据。如果依赖关系错误——软依赖连成硬依赖或漏了真正的硬依赖——诊断引擎会追到错误根因。任何修改图谱边的操作必须经过跨样本验证（≥3个独立数据源验证教学效果）。

## 第二层：元能力 — 安全进化机制

### 2.1 自我进化路线图

图谱系统只有一个独立维度：从「人工判定依赖」到「系统自动发现依赖+自适应权重」。

| 阶段 | 触发条件 | 系统能力 |
|------|---------|---------|
| 1（当前） | — | 所有依赖由人工用四条标准逐对判定 |
| 1.5 | S07诊断引擎和S09课堂生成引擎实现「前置条件检测埋点」——记录「学生在未掌握前置A的情况下尝试B」的行为数据（包括：A的mastery水平、B的尝试时间、B的失败率） | 埋点就绪——数据开始积累，但尚未用于自动发现 |
| 2 | 阶段1.5埋点积累≥2教学月数据 + S01维度一阶段2后 | 系统分析「掌握 A 之后 B 的掌握速度变化」+「未掌握 A 但尝试 B 的失败率」。统计显著（p < 0.01）标记为候选隐性依赖 |
| 3 | ≥5 条候选被人工确认+≥3 条在教学实验中验证有效（干预组 vs 对照组 p<0.05）+ 信用分≥60 | 自动将中置信度隐性依赖加入诊断依赖层（标记 source: auto_discovered）。高置信度直接加入学习路径层 |
| 4 | 阶段3稳定运行≥12个月 + 信用分≥120 | 系统自己发现跨学科能力传导规律，记录为跨域依赖 |

**退化降级规则**：连续3次隐性依赖验证失败→回退到阶段2，p值阈值收紧20%。

### 2.2 不可修改边界

| 类型 | 项目 | 原因 | AI的直接后果 |
|------|------|------|------------|
| 数据完整性 | 图谱 JSON 的双向一致性约束 | R8 铁规——若A.prerequisites含B，则B.next必含A | 单向边导致诊断引擎溯源路径断裂 |
| 全局外键 | knowledgeCode | 图谱以 knowledgeCode 定位所有节点 | 全局数据不一致 |
| 跨图引用完整性 | G5-G6图谱中所有prerequisites指向的G1-G4节点必须存在且未被标记为已删除 | 跨图悬空引用导致诊断引擎溯源中断但无报错 | 静默溯源失败→诊断引擎给错误根因 |
| 质量门禁 | R8双向边一致 | 数据腐败防线 | 假边通过校验 |

### 2.4 间接修改边界（量化标准）

| 修改类型 | 差异阈值 | 处置 |
|---------|---------|------|
| 修改硬依赖判定标准 | <5% | 正常推进 |
| 修改四标准之一 | 5-15% | 审查员审核 |
| 放宽硬依赖范围 | 15-30% | 产品经理签批 |
| 移除某条标准 | >30% | 视为越界直接拒绝 |

### 2.5 异常进化处置

| 异常 | 检测 | 处置 |
|------|------|------|
| 连续失败 | 同维度连续3次回滚 | 60天冻结 |
| 长期停滞 | 同阶段≥18教学月 | 信用分-5/月（最低0分） |
| 路线图矛盾 | 终局冲突或阶段倒退 | 全维度冻结 |

### 2.6 应急熔断（三级）

- 轻量级：暂停高/中风险进化，低风险继续
- 子系统：图谱全部进化暂停
- 全系统：全停+回滚未完成的

### 2.7 环境差异化

| 环境 | 进化权限 | 北极星校验 |

| 正式 | 低风险事后报告 | 全三层，任何恶化直接拦停 |

### 2.8 内容进化与代码进化

- 内容进化：新增/删除图谱边、调整依赖关系（教研审核→灰度→全量）
- 代码进化：修改硬依赖判定算法、修改 build 脚本（路线图触发→北极星→安全管道）

### 3.2 自我进化执行方法

五类角色权限台账：芽芽自己（低风险自主，无检查模块修改权）/ AI编程工具（仅读取+方案生成）/ 产品经理（最高权限+签批+熔断）/ 教研人员（中风险审核）/ 审查员（标记通过/疑似/不通过）。

五步流程：信号采集（≥2周教学日）→方案生成→审核（三级权限）→安全执行（影子→A/B→全量）→记录验证（30天稳定期）。

**信用分三子分**：low_risk_score / mid_risk_score / high_risk_score 三本独立账。聚合=min(三子分)。任意子分<20→对应级别暂停。min<10→全系统暂停。初始值=50（与S01一致）。月度上限+20分。长期停滞扣减-5/月（最低0分）。

**阶段推进信用分阈值**（确保与时间条件数值一致）：
- 阶段 1→2：信用分≥20（约1个月无故障运行）
- 阶段 2→3：信用分≥60（约3个月）+ ≥5条候选+≥3条教学验证
- 阶段 3→4：信用分≥120（约6个月）+ 稳定运行≥12个月

**设计理由**：12个月稳定运行期间如果出现连续失败导致信用分扣减，120分可能比12个月更难达到——双条件互为保险。信誉好但时间不够→等时间；时间够但信誉差→修问题。

### 大白话

知识图谱就是知识点之间的「谁先谁后」关系表。两层各有用处：学习路径层像教材目录——告诉你先学什么后学什么，学生和家长看到的就是这个顺序。诊断依赖层像医生看病的病因追溯链——你头疼，医生不会直接开头疼药，而是查血压、查颈椎，因为头疼可能是高血压引起的。诊断引擎也一样，学生卡在分数乘法，引擎沿依赖链往上查：是不是分数意义没搞懂？是不是通分有问题？但诊断引擎不会无限追下去，最多查三代——就像医生不会因为头疼去查你的DNA序列。再往上就不追了，而是建议安排专项基础测试。

硬依赖就像盖房子的地基——没有地基不能盖一楼。软依赖就像两个相邻的店铺——挨着但不互相依赖。不要把两个挨着的知识点自动连一条线，除非满足那四条标准之一。四条标准之间有优先级：定义里直接用了对方→最高；概念层级→第二；操作前置→第三；教材顺序→第四。为什么教材顺序排最后？因为不同版本的教材顺序不一样——人教版要先复习这个再教那个，苏教版未必。这种"仅某个教材版本才有的依赖"需要标注清楚，免得换了教材版本后图谱出现假依赖。

进化路线图是一个维度：从「人手工画线」到「系统自己发现隐藏的依赖关系」。但这中间缺一个关键步骤——系统得先能"看到"学生在没掌握前置知识的情况下尝试新知识时会怎样。这就像医生要先装好心电监护仪，才能判断某种药对心脏的影响。所以阶段1和阶段2之间加了一个1.5：先把数据采集埋点做好。有了数据积累，系统才能分析——学了 A 的孩子学 B 是不是更快？没用 A 的学 B 是不是更容易失败？如果统计显著，标记为候选隐性依赖，人工确认后在教学实验中验证，验证有效就加入图谱。但系统也会犯错误——如果连续验证失败，就自动退回到更保守的阶段。

不可修改的就是两件大事：一是「双向边必须一致」——你连了 A→B，就必须有 B←A，单边会让诊断引擎在溯源时走进死胡同。二是「跨图引用不能悬空」——高年级的知识点如果指向前置的低年级知识点，那个低年级知识点必须真的存在。如果低年级的知识点被删了，高年级的指针就变成"鬼箭头"——指着不存在的东西，系统不会报错但诊断结果全错。

为了防止有人偷懒直接改JSON文件而不跑build脚本，build脚本每次会留下一个数字签名（像文件的"防伪标识"）。CI机器人在每次提交代码时检查签名——签名对不上就说明有人手动改过，直接拦下来。

推动系统从一个阶段进化到下一个阶段，需要同时满足两个条件：时间够了+信誉够了。就像考驾照——不光要学满学时，还得考试通过。光耗时间但老出事故不行，光分数高但时间没学够也不行。
