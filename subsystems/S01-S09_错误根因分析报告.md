## 三、错误根因分类（6 类）

### A 类：生成后不跑自查清单 → 大规模遗漏

占比：约 40%（~96 项）

典型表现：
- 缺少输出 Schema、缺少消费清单接口定义、缺少事件触发时序
- 缺少 AI-MUTABLE 标注（S05-S09 反复出现）
- 缺少阶段表"人工角色"和"终局状态"列（S04-S07 全都缺）
- 缺少演化推理链（S04-S07 全都没有 §1.6）
- 缺少安全执行管道完整描述（S04-S07 只有一句话）
- 缺少大白话覆盖 10/10（S03-S07 首次都只有 5-6 段）
- 缺少信用分门槛（S04-S07 全缺）
- 缺少维度交互矩阵（S07/S09 首版缺）
- 缺少数据闭环声明（S01-S08 全缺）
- 缺少不可修改边界 7 类补全（S03-S07 首次只有 3-4 类）
- 缺少消费清单检测方式列（S03-S07 首次只有 6 列）
- 缺少间接修改量化表（S03-S07 首次只有一句话）
- 缺少越界告警表（S03-S07 首次没分三级）
- 缺少 A/B 测试规则+元进化递归上限（S04-S07 首版缺）
- 缺少提交前自查声明（S04-S07 首版缺）
- 缺少 lesson_generated 消费行（S09 首版缺）
- 缺少 Phase 1 默认画像表（S09 首版缺）
- 缺少首次生成延迟预估（S09 首版缺）
- 缺少 SP_G_D01 兜底课堂结构（S09 首版缺）
- 缺少阶段 2 退化时积木继承规则（S09 首版缺）

根因：生成完文档后，没有对照 DOC_ACCEPTANCE_STANDARD 逐条检查就直接输出。

### B 类：跨文档引用不验证 → 依赖断裂

占比：约 15%（~36 项）

典型表现：
- S09 引用 S03"第一个常见错因"但 S03 没定义排序规则
- S09 引用 S13 的 difficulty 字段但 S13 尚未定义
- S09 引用 09 号事件协议但缺少 lesson_generated 事件
- S09 引用 S07 的 diagnosis_updated 但推送方式未定义
- S09 引用 18 号画像文档但该文档不存在
- S02 信用分初始值与 S01 不对齐（0 vs 50）
- S08 引用 S06 字段但字段名不同
- S09 消费清单 S04 行写"待定义"但没实际读 S04
- 消费清单中 S08/S09/S12 引用未标注"待定义"

根因：写文档时凭记忆引用其他子系统，没有打开对应文档核实。

### C 类：多处出现同一概念不全局搜索 → 不一致

占比：约 20%（~48 项）

典型表现：
- 通用骨架定义了 4 个位置，1 处写 2 道题、3 处写 3 道题（或反之）
- 退化规则维度一和维度二互相矛盾
- G5 挑战偏好=balanced、G6 挑战偏好=中等（枚举不一致）
- 大白话"六个系统" vs 消费清单 10+ 行
- 阶段 1 终局 394 个知识点 vs Phase 1 实际 189 个
- 24 种粗粒度画像 vs Phase 1 实际 8 种（2 年级×4 级别）
- 延迟预估"200ms"和"10 秒超时"放在同一行
- masteryLevel 和 curriculumLevel 两个量表混用
- v1.1 和 v1.2 的变更记录几乎逐字相同
- sampleCount 和 studentAnswerCount 混用（S08）
- 维度交互矩阵与进化路线图退化规则不一致（S01/S09）
- 章节编号偏移（S08）
- 安全护栏引用顺序错乱（S08）

根因：在文档多处写同一概念时，没有用 grep 全局搜索检查一致性。修复时也没有做"该字符串在全文是否还有残留"的检查。

### D 类：不模拟端到端场景 → 逻辑缺陷

占比：约 15%（~36 项）

典型表现：
- S07 异步诊断 vs S09 实时需求——两条路径分开写了但没模拟"学生做错题当场要反馈"
- 通用骨架错因指向不精确——"第一个常见错因"不等于"跟本次错误匹配的错因"
- Phase 1 不启动积木沉淀——写了课堂级 effectScore 但没说关闭积木沉淀
- traced 链缺 excluded/confirmed_with_reservation/inconclusive 三个分支（S08 累计 3 轮修）
- 预热期信用分机制——"连续 30 天"在 30 天预热期内永远无法触发（S08）
- 3σ 冷启动阈值 10 vs 30——两个阈值用途不同但没解释（S08）
- 北极星正信号缺均值回归排除（S08）
- 回滚率分子分母未定义（S01）
- 维度四退化缺跨维度约束（S01）
- Phase 1 降级规则不完整——只写了 2 行受影响，没说其余 4 行（S08）
- 5000 条 vs 6 个月取 AND 不合理（S09）
- 连续退化未定义（S09）
- S05 检查不在六步管线里（S09）
- 通用骨架是否跳质检未定义（S09）

根因：写管道步骤时，"如果 X 没就绪，用户会经历什么"这种端到端走查没做。

### E 类：修复后不 grep 残留 → 残值

占比：约 5%（~12 项）

典型表现：
- G6"挑战偏好=中等"在 Fix3 后仍然残留（S09 v1.3）
- S08 多处 sampleCount 没有一次性全部重命名
- 章节编号调整后遗漏引用点

根因：修复一个字符串后，没有用 grep 搜索该字符串在全文是否还有残留。

### F 类：变更记录不做 diff 验证 → 版本混乱

占比：约 5%（~12 项）

典型表现：
- v1.1 和 v1.2 的变更记录几乎相同
- 变更记录日期不是 git 实际提交时间（反复出现）
- 修复内容列表和实际 diff 不一致

根因：写变更记录时凭印象写，没有跑 git diff 后对照实际改动写。

## 四、根因汇总

6 个根因按影响从大到小排列：

1. **不跑自检清单** → 40% 的错误（大规模遗漏：缺 Schema、缺接口、缺标注、缺章节）
2. **不全局搜索一致性** → 20% 的错误（多处写同一概念互相矛盾）
3. **不验证跨文档引用** → 15% 的错误（凭记忆引用其他文档导致字段名/定义不存在）
4. **不做端到端走查** → 15% 的错误（写的管线在实际场景中跑不通）
5. **不 grep 残留** → 5% 的错误（修复不彻底）
6. **不写真实变更记录** → 5% 的错误（版本历史不可信）

## 五、解决方案

### 方案：6 步强制自检流水线（生成或修改任何子系统文档后必须执行）

以下 6 步全部通过才能提交：

**第 1 步：对照 DOC_ACCEPTANCE_STANDARD 逐条检查**

检查清单：
- [ ] 1.1 版本号+变更记录
- [ ] 1.2 北极星对齐章节
- [ ] 1.3 大白话 10/10 覆盖（系统职责/核心设计/进化路线图/不可修改边界/消费清单/执行方法/异常处置/应急熔断/信用分/内容代码分离）
- [ ] 1.4 不可修改边界 7 类（外部法规定义/通信协议基础字段/学生原始数据/全局外键标识符/北极星底线信号/质量门禁最低标准/Shareable 状态阈值）
- [ ] 1.5 消费关系清单 7 列（被消费方/消费方式/故障类型/严重等级/检测方式/链路类型/备注）
- [ ] 1.6 自我进化执行方法（五类角色权限台账/间接修改边界量化表/越界告警三级/AB测试规则+元进化递归上限）
- [ ] 1.7 安全执行管道完整三层（影子回放/A/B测试分流/全量上线PSI）
- [ ] 1.8 信用分三子分（内容/代码/响应）+ 各阶段信用分门槛
- [ ] 1.9 AI-MUTABLE 全量标注（所有可进化阈值必须有注释）
- [ ] 1.10 数据闭环声明（集中段落，所有维度的 09 号事件依赖汇总）
- [ ] 1.11 维度交互矩阵（二维度以上的系统必须有）
- [ ] 1.12 设计演化推理链（为什么这样设计、为什么不是别的方式）
- [ ] 1.13 提交前自查声明（3.18 式）
- [ ] 1.14 演化路线图阶段表（含人工角色/终局状态两列 + 阶段触发条件含信用分门槛）

**第 2 步：提取所有 S\d\d 引用 → 打开对应文档核实**

用 grep 提取文档中所有 `S\d\d` 引用，逐个打开目标文件核对：
- 字段名是否一致
- 接口定义是否存在
- 数据格式是否匹配
- 标注"待定义"如果目标系统尚未就绪

**第 3 步：提取关键概念 → grep 全文档检查一致性**

提取文档中所有关键定义（如"通用骨架""题量""默认值""兜底""退化"），
每个概念在全文中搜索，确认所有出现位置的描述一致。

**第 4 步：端到端走查 3 个场景**

选 3 个真实使用场景，脑中模拟完整路径：
- 场景 1：Phase 1 首次使用（所有依赖系统未就绪时的降级路径）
- 场景 2：Phase 2 已有历史数据（正常路径）
- 场景 3：退化/异常场景（某个依赖系统出问题时的回退路径）

每步问："这一步的输入从哪来？如果输入不存在怎么办？"

**第 5 步：修复后 grep 残留**

修改任何字符串后，用 grep 搜索该字符串在全文是否还有残留。
特别是：字段名改名、枚举值修改、数字调整、章节号调整。

**第 6 步：git diff 后写变更记录**

跑 `git diff --cached` 看实际改动，逐条写入变更记录。
变更记录日期从 `git log -1 --format="%ai"` 取实际提交时间。
严禁复制粘贴上一版本的变更记录。
