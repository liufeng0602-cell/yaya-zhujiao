## 北极星对齐

策略引擎不直接和学生交互——它通过选策略间接影响学习体验。因此北极星校验走「策略中断熔断」通道：任何策略组合变更都必须证明不会让底线信号恶化。

| 优先级 | 信号层 | 判定规则 |
|--------|--------|---------|
| 最高 | 底线信号 | 任何一项触碰 → 策略进化暂停，回滚到上一个稳定版本 |
| 中等 | 负信号 | 任一恶化 → 策略选择不升级，维持当前版本 |
| 基础 | 正信号 | 仅当底线/负信号均无恶化时，正信号至少 2 项改善 → 策略升级通过 |

**正信号**（策略质量上升）：
- 策略干预后学生后续答题正确率上升（同一 KP 间隔掌握率提升）<!-- AI-MUTABLE: 上升阈值 5%, type=float, range=[0.03, 0.10] -->
  > **排除回归均值规则**：正确率上升必须排除回归均值——将学生前 5 次该 KP 的答题正确率均值作为基线，策略后正确率必须高于基线 + 1 个标准差才算有效提升。基线自动衰减（时间越久远权重越低）。
- 学生求助后不再重复求助（同一 KP 再次 hint_requested 比例下降）<!-- AI-MUTABLE: 下降阈值 10%, type=float, range=[0.05, 0.15] -->
- 策略覆盖率提升（新错因有对应策略可查）<!-- AI-MUTABLE: 覆盖率目标 95%, type=float, range=[0.90, 0.99] -->

**负信号**（策略方向偏离）：
- 强策略占比上升但整体正确率持平或下降（策略在「打地鼠」——换着花样无效干预）
- 学生连续 3 次收到同一天 2 个以上策略仍未见效（策略选择失效）
- 轻量策略使用频率异常上升（诊断置信度整体下降，全员降级为保守策略）

**底线信号**（绝对不可触碰）：
- 策略导致学生放弃率上升<!-- AI-MUTABLE: 放弃率基线+2σ, type=float, range=[0, 3.0] -->
- 强策略连续触发 3 次学生仍答错——系统未降级（安全护栏失效）
- 策略参数直接操作学校课表/家庭作业时间（侵入非教学域）

**北极星检查不可绕过性声明**：策略引擎自我进化中的任何策略新增/淘汰/权重调整生效前，必须跑三轮北极星检查（新→旧→新二次确认）。三轮检查必须全部通过。任何一环失败→回滚到进化前版本，记录失败原因到策略进化日志。

## 一、定义与职责

### 这个系统负责什么

策略引擎是芽芽的「治疗师」——诊断引擎（S07）输出诊断结果后，策略引擎决定「怎么治」。具体职责：

1. **消费诊断事件**：监听 `diagnosis_updated` 事件（09 号事件 8），读取六态诊断结果
2. **六态调度决策**：根据 `diagnosisStatus`（confirmed/confirmed_with_reservation/excluded/inconclusive/in_progress/traced）走不同分支
3. **策略选择**：从 S03 错因库和 S04 策略包库中匹配最佳策略，叠加上学生历史效果
4. **体验保护**：在策略选择前先检查学生是否处于「需要抢救」的状态（连续错误、挣扎、退出倾向）
5. **安全护栏**：强策略条件检查、频率限制、冷启动保护
6. **输出策略事件**：发出 `strategy_applied` 事件（09 号事件 9）→ S09 课堂引擎消费、S12 仪表盘消费
7. **效果回写**：消费 `strategy_completed` 事件（09 号事件 10），计算 `effectScore`，回写到 S04 策略效果表

### 这个系统不负责什么

- 不负责诊断学生错在哪（S07 诊断引擎）
- 不负责定义策略包的内容（S04 策略包体系）
- 不负责定义错因（S03 错因体系）
- 不负责生成课堂内容/话术（S09 课堂生成引擎）
- 不负责向家长解释（S12 家长摘要系统）
- 不负责决定知识点有多难（S06 分级系统）

### 大白话定义

策略引擎就像医院里的「开药系统」——S07 诊断引擎是体检报告（「右腿骨折、中度」），S08 策略引擎就是根据报告开处方的（「打石膏 4 周、止痛药每天 1 次、两周后复查」）。它不看片子（那是 S07 的活），只看报告，然后从药房（S04 策略包库）里挑最合适的药。

## 二、核心设计

### 2.1 输入：diagnosis_updated 事件

策略引擎监听 09 号事件的 `diagnosis_updated`（事件 8）。消费以下字段：

| 字段 | 用途 | 来源 |
|------|------|------|
| `diagnosisStatus` | **调度分支的开关**。六态：confirmed / confirmed_with_reservation / excluded / inconclusive / in_progress / traced | S07 诊断引擎 |
| `mainHypothesis.errorCode` | 主错因编码。用于选择对应的策略包 | S07 → S03 错因库 |
| `mainHypothesis.probability` | 错因概率（0-1）。决定策略强度 | S07 诊断引擎 |
| `mainHypothesis.confidence` | 置信度（0-1）。决定是否允许强策略 | S07 诊断引擎 |
| `mainHypothesis.recommendedAction` | 诊断引擎的动作建议（如 reduce_difficulty / provide_scaffolding） | S07 诊断引擎 |
| `hypotheses[]` | 完整假设集。如果主假设的强制策略有风险，回退到次假设 | S07 诊断引擎 |
| `diagnosisId` | 用于关联后续 strategy_applied/strategy_completed | S07 诊断引擎 |
| `chainId` | 用于追踪完整事件链 | 09 号事件协议 |
| `studentId` | 查历史策略效果 | S07 → S01 学生画像 |
| `knowledgeCode` | 查学科专属策略包 + 历史有效率 | S07 → S01 KP 库 |
| `errorDiagnosis.traceDepth` | 溯源深度。traced 态时判断是否达到 3 层上限 | S07 诊断引擎 |

**fallback 逻辑**（老版本数据兼容）：
- 如果 `diagnosis_updated` 不存在但 `diagnosis_attempted` 存在 → 仅 `inconclusive` 态触发轻量策略，其他态等待 `diagnosis_updated`
- 如果两者都不存在 → 等待 `diagnosis_updated` 到达，不做策略选择

### 2.2 调度顺序

策略引擎的执行顺序不是「诊断→策略」一步到位。在选策略之前，先查体验保护——学生可能正在挣扎、连续错误、想退出。这时候先抢救状态，再谈策略。

```
体验保护检查（§2.3）→ 六态决策（§2.4）→ 安全护栏检查（§2.5）→ 策略选择算法（§2.6）→ 发出 strategy_applied
```

体验保护触发后跳过六态决策，直接覆盖策略选择。

### 2.3 体验保护覆盖规则

在策略选择之前，先检查以下条件：

| 条件 | 覆盖动作 | 优先级 |
|------|----------|--------|
| 同一知识点连续错 ≥ 3 道 + 学习能量状态 = "low" | 能量恢复包 SP_G_D03 | 最高 |
| 同一知识点连续错 ≥ 3 道 + 学习能量正常 | 舒缓讲解包 SP_G_D01 | 高 |
| 答题时间 > 历史均值 3σ（挣扎）<!-- AI-MUTABLE: 3σ, type=float, range=[2.0, 4.0] --> | 能量恢复包 SP_G_D03 + difficultyAdjustment = -0.3 | 高 |
| 上一策略 effectScore = -2 | 基础巩固包 SP_G_C01（安全兜底） | 高 |
| 同一 lessonId 内连续 3 次 answer_abandoned<!-- AI-MUTABLE: 3次, type=int, range=[2, 5] --> | 能量恢复包 SP_G_D03 或舒缓讲解包 SP_G_D01 | 高 |
| 5 分钟内连续退出<!-- AI-MUTABLE: 5分钟, type=int, range=[3, 10] --> | 下次进入时预设能量恢复包 | 中 |

> **连续错误计数范围**：同一 lessonId 内。跨 lesson 不累计——每节课开始时连续错误计数器重置为 0。这样避免学生在第二节课第一道题就被判定为「挣扎」，也避免在单节课内连错三道才补救。

> **effectScore = -2 的触发时序**：基于当前 chainId 内的 strategy_completed 事件即时判断，不等 StrategyEffect 表异步写入。触发一次兜底后同时标记该策略待禁用——7 天禁用规则接管后续保护。
>
> **学习能量状态来源**：由画像系统 GrowthMemory 表存储。
>
> **Phase 1 降级规则**：Phase 1 如未接入学习能量信号，仅前两行（学习能量状态相关）降级——连续错误 >= 3 一律使用舒缓讲解包 SP_G_D01。其余 4 行（答题时间 > 3sigma、effectScore = -2、连续 3 次 answer_abandoned、5 分钟内连续退出）不受学习能量信号缺失影响，正常生效。
>
> **连续退出定义**：同一 lessonId 内连续 2 次 answer_abandoned，或 lesson_completed.completionStatus = interrupted。触发后写入 GrowthMemory.pendingAction，下次 lesson_started 时策略引擎读取并预设能量恢复包，读取后清除。

### 2.4 六态调度决策树

根据 `diagnosisStatus` 走不同分支：

#### 2.4.1 confirmed（确诊）

```
mainHypothesis.probability ≥ 0.85：
  confidence ≥ 0.7 → 允许强策略（变式特训/错因专项）
  confidence < 0.7 → 只用中策略（分步引航/审题侦探/精准计算）

mainHypothesis.probability < 0.85：
  只用中策略

选策略：
  从 error_taxonomy.json 读该 errorCode 的 recommendedStrategies
  映射到策略包编码（strategy_mapping.json）
  叠加学生历史策略效果（有效升权、无效/有害降权）
  检查策略滥用防护
  输出 strategy_applied
```

> **probability 阈值**：0.85<!-- AI-MUTABLE: confirmed概率阈值, type=float, range=[0.75, 0.95] -->
> **confidence 阈值**：0.7<!-- AI-MUTABLE: 强策略confidence阈值, type=float, range=[0.60, 0.85] -->
> **中策略定义**：介于强策略和轻量策略之间，包括分步引航包 SP_G_B01、审题侦探包 SP_G_B04、精准计算包 SP_G_C05。中策略触发门槛比强策略宽松：probability ≥ 0.5 即可<!-- AI-MUTABLE: 中策略概率阈值, type=float, range=[0.40, 0.60] -->，不要求 confidence ≥ 0.7。

#### 2.4.2 confirmed_with_reservation（有保留确诊）

```
probability 在 0.5-0.85，满 3 轮验证：
  只触发轻量策略：
    - 基础巩固包 SP_G_C01
    - 记忆唤醒包 SP_G_C06
  不触发强策略
  不触发中策略
  参数设置：
    questionCount = 2（低题量）
    hintLevel = 1（提供提示）
    difficultyAdjustment = -0.1（略降难度）
```

#### 2.4.3 excluded（假设被排除）

```
等待下一个 diagnosis_updated（诊断引擎正在切换假设）
  如果下一个 diagnosisStatus = "traced" → 见 2.4.6
  如果下一个 diagnosisStatus = "in_progress" → 不动作，继续等待
  如果超过 120 秒未收到新 diagnosis_updated → 降级：发轻量策略（基础巩固包），记录 badcase
```

#### 2.4.4 inconclusive（无法判断）

```
触发轻量安全策略：
  1. 基础巩固包 SP_G_C01（降低该知识点难度 + 减题量）
  2. 能量恢复包 SP_G_D03 或 舒缓讲解包 SP_G_D01（取决于学习能量状态）：
     - 连续错误 ≥ 3 且学习能量正常 → 舒缓讲解包
     - 连续错误 ≥ 3 且学习能量低或出现退出倾向 → 能量恢复包
  参数设置：
    questionCount = 1~2
    difficultyAdjustment = -0.2
    hintLevel = 1
```

> **连续错误阈值**：3<!-- AI-MUTABLE: inconclusive连续错误阈值, type=int, range=[2, 5] -->

#### 2.4.5 in_progress（验证进行中）

```
不动作。策略引擎继续等待下一个 diagnosis_updated。
不记录策略事件。
```

#### 2.4.6 traced（切入溯源）

```
诊断引擎已对前置知识点发起新诊断。
策略引擎通过同一 chainId 下的下一个 diagnosis_updated 事件识别溯源结果——如果该事件的 knowledgeCode 不同于原始诊断的 knowledgeCode（说明是对前置知识点的诊断），且 diagnosisStatus = 'confirmed'，则对该前置知识点触发策略（回到 2.4.1）
如果溯源 3 层仍 inconclusive → 对原始知识点触发轻量安全策略（回到 2.4.4）
判断方式：策略引擎收到 diagnosisStatus='inconclusive' 且 errorDiagnosis.traceDepth=3，两个条件同时满足则触发。
```

### 2.5 安全护栏

#### 强策略触发条件

以下策略为「强策略」：变式特训包 SP_G_C02 / 错因专项包 SP_G_C03<!-- HUMAN-SIGNED: 强策略列表, 签字人=教研人员 -->

必须同时满足 5 个条件：
1. diagnosisStatus = confirmed
2. mainHypothesis.probability ≥ 0.85<!-- AI-MUTABLE: 强策略概率门槛, type=float, range=[0.75, 0.95] -->
3. mainHypothesis.confidence ≥ 0.7<!-- AI-MUTABLE: 强策略置信度门槛, type=float, range=[0.60, 0.85] -->
4. sampleCount ≥ 30<!-- AI-MUTABLE: 强策略最小样本量, type=int, range=[20, 50] -->
5. 该策略历史 effectScore 不低于 -1

#### 策略频率限制

| 限制 | 说明 |
|------|------|
| 同策略同天 ≤ 3 次<!-- AI-MUTABLE: 同策略日上限, type=int, range=[2, 5] --> | 超过后当天禁用 |
| 同策略连续使用 ≤ 5 次<!-- AI-MUTABLE: 连续使用上限, type=int, range=[3, 7] --> | 第 6 次强制轮换为备选策略包 |
| 强策略同天 ≤ 2 次<!-- AI-MUTABLE: 强策略日上限, type=int, range=[1, 3] --> | 同天内强策略总触发次数，不受策略包种类限制 |
| effectScore = -2 的策略 7 天内禁用<!-- AI-MUTABLE: 冷却天数, type=int, range=[3, 14] --> | 给策略冷却期 |

#### 冷启动保护

sampleCount < 30<!-- AI-MUTABLE: 冷启动样本门槛, type=int, range=[10, 50] --> 时：
- 所有强策略关闭
- 策略参数自动降级：questionCount 减半，hintLevel + 1，difficultyAdjustment - 0.2
- 优先使用轻量安全策略（基础巩固包、记忆唤醒包、能量恢复包）<!-- HUMAN-SIGNED: 轻量安全策略列表, 签字人=教研人员 -->

### 2.6 策略选择算法

```
function selectStrategy(diagnosisUpdated, studentHistory):
  errorCode = diagnosisUpdated.mainHypothesis.errorCode
  knowledgeCode = diagnosisUpdated.knowledgeCode

  // Step 1: 从 taxonomy 取候选策略
  candidateStrategies = errorTaxonomy[errorCode].recommendedStrategies
  // 通过 strategy_mapping.json 映射到策略包编码

  // Step 2: 读取诊断引擎的 recommendedAction
  recommendedAction = diagnosisUpdated.mainHypothesis.recommendedAction
  if recommendedAction:
    if recommendedAction == "reduce_difficulty": params.difficultyAdjustment = -0.2
    if recommendedAction == "provide_scaffolding": params.hintLevel += 1

  // Step 3: 根据 diagnosisStatus 过滤强度
  if diagnosisStatus in ("confirmed_with_reservation", "inconclusive"):
    candidateStrategies = filterLightOnly(candidateStrategies)

  // Step 4: 叠加知识点专属策略（SP_M_*）
  // 如果 taxonomy 映射后已包含 SP_M_* 策略包（该错因的 recommendedStrategies 已映射到 SP_M_*），则保留。
  // 如果映射结果中没有任何 SP_M_*，则从该错因的 error_taxonomy 条目中读取 neighborErrors 字段，
  // 查找相邻错因是否有 SP_M_* 映射，有则加入候选。
  if candidateStrategies 中不含 SP_M_*:
    neighborSp = 从 neighborErrors 查找相邻错因的 SP_M_* 映射
    if neighborSp 非空: candidateStrategies += neighborSp

  // Step 5: 历史效果调整权重
  // 注意：Step 5 查询的是“该策略对该学生该知识点的历史执行次数”（用于判断效果置信度），
  // 区别于 §2.5 强策略/冷启动使用的 sampleCount——“该学生该知识点的总答题次数”。
  // 两者统计口径不同，开发实现时不可共用同一个变量。
  for each strategy in candidateStrategies:
    historyEffect = queryStrategyEffect(
      studentId, strategy, knowledgeCode,
      minSamples=3,
      fallbackScope="subject"
    )
    if historyEffect >= 1: weight × 1.3
    if historyEffect == -1: weight × 0.5
    if historyEffect == -2: remove(strategy)

  // Step 6: 滥用防护
  removeOverusedStrategies(candidateStrategies)

  // Step 7: 选最高权重
  selectedStrategy = topWeighted(candidateStrategies)

  // Step 8: 强策略最终安检
  if isStrongStrategy(selectedStrategy):
    if not checkStrongConditions(selectedStrategy, diagnosisUpdated):
      selectedStrategy = fallbackToMedium(selectedStrategy.errorCode)

  return selectedStrategy
```

## 四、设计演化推理链

### 为什么是「六态调度」而不是「诊断→策略」一步到位？

旧版设计（v1.0）是：诊断引擎输出「确诊错因 X」→ 策略引擎直接发策略。3 个致命问题：

1. **假确诊**：诊断引擎说「80% 概率是小数点位错」但实际是审题问题。策略选错→学生收到无效干预→放弃率上升
2. **信息浪费**：诊断引擎有 6 种状态（确诊/有保留/排除/无法判断/验证中/溯源中），但旧版只用了「确诊/非确诊」2 种——4 种状态的信息全扔了
3. **没有安全阀**：学生连续错 5 道题，策略引擎还在「选策略」——应该先检查学生是否已经崩溃

**演化信号**：从学生行为数据中发现——(a) 非确诊状态的策略选择正确率比确诊低 40%；(b) 连续错误 3+ 次后仍发策略的学生放弃率飙升；(c) 溯源场景中策略发错对象（对原始 KP 发策略而非前置 KP）

**结论**：六态调度树不是设计选择，而是数据驱动的必然——每种状态需要完全不同的处理方式。体验保护放在决策树之前——先抢救再治疗。

### 为什么策略选择用「权重叠加」而不是「规则引擎」？

规则引擎（if A then B）的问题是：
- 规则数量爆炸：87 个错因 × 14 个策略包 = 1218 条规则
- 规则冲突无法处理：taxonomy 推荐策略 A，但学生历史上策略 A 无效→规则引擎不知道怎么办
- 新数据无法让系统变聪明：effectScore 积累了 1000 条数据，但规则引擎不会自动调整

权重叠加解决这三个问题：taxonomy 推荐 = 基准权重 × 1.0，历史效果好的调高权重，历史效果差的调低或移除。规则引擎是静态的「如果忘了进位就发进位特训」，权重叠加是动态的「上次进位特训对他没用，这次换审题侦探吧」。

**演化信号**：从 S04 策略包历史 effectScore 分布中发现——同一策略对不同学生的 effectScore 分布差异极大（标准差 > 1.5），说明「同一错因→同一策略」的规则模式是错的。

### 为什么强策略要五条件限制？

早期设计是「确诊就发强策略」。数据发现：确诊但 confidence < 0.7 的场景中，强策略的 effectScore = -2 比例高达 35%——等于每发 3 个强策略就有 1 个伤害学生。五条件限制把伤害比例降到 5% 以下。

**演化信号**：effectScore = -2 的事件聚类分析发现，70% 的「有害策略」出现在 confidence < 0.7 或 sampleCount < 30 的场景中。

### 6.2 信用分体系

策略引擎的进化信用分独立于 S01 全局信用分，专门衡量策略选择的可靠性：

- **初始值**：50
- **上升信号**：策略 effectScore 均值 > 0.5 连续 30 天 → +5
- **下降信号**：强策略触发后 effectScore = -2 比例 > 10% → -10
- **底线信号**：策略导致放弃率连续 7 天上升 → -20
- **上限**：100
- **下限**：0（触发全系统熔断——策略引擎降级为仅发基础巩固包 SP_G_C01）

> **信用分熔断与安全护栏的层叠关系**：信用分触发熔断时，安全护栏的频率限制和冷启动保护仍然生效，但强策略条件被覆盖——即使满足五条件，也不发强策略。熔断期间唯一输出 = 基础巩固包 SP_G_C01。
- **信用分影响**：信用分 < 20 → 所有强策略关闭<!-- AI-MUTABLE: 强策略关闭信用分阈值, type=int, range=[10, 30] -->；信用分 < 10 → 仅保留体验保护覆盖逻辑，关闭权重选择（所有策略包等权随机选）<!-- AI-MUTABLE: 等权随机信用分阈值, type=int, range=[5, 15] -->

### 6.3 Phase 1 初始启动状态

策略引擎上线后立即开始六态调度和策略选择（不设纯观测期）。信用分从上线第一天开始计算。

- **前 30 天为信用分预热期**——期间信用分只涨不跌（不上涨->不下跌->不回滚、不熔断），用于积累 baseline 数据。
- **预热期结束后正式启用完整信用分体系**——信用分可涨可跌，20 分以下关闭强策略，10 分以下仅等权随机选，0 分熔断为仅基础巩固包。
- 预热期期间的数据自然积累 strategy_completed 事件——30 天后至少有一定数据量供信用分初始校准。

### 6.4 安全执行管道

策略引擎的任何自我进化（算法修改、阈值调整、权重因子变更）走三层管道：

**第一层 — 影子模式**：新策略选择逻辑在后台并行运行（不实际发出策略），shadowEffectScore 与当前策略的 effectScore 对比 30 天。shadowEffectScore 高于当前至少 5%<!-- AI-MUTABLE: 影子模式提升阈值, type=float, range=[0.03, 0.10] --> 且无底线信号触发 → 进入 A/B。

**第二层 — A/B 测试**：5% 学生进实验组使用新逻辑<!-- AI-MUTABLE: A/B实验组比例, type=float, range=[0.03, 0.10] -->，其余用现有逻辑。分组随机分层（按年级 + knowledgeCode 均匀分布）。p < 0.05，最小样本量 = 30<!-- AI-MUTABLE: A/B最小样本量, type=int, range=[20, 50] -->（Cohen's d ≥ 0.3）。A/B 测试期间实验组 effectScore 必须高于对照组，且底线信号无恶化。

**第三层 — 全量上线**：A/B 通过 + 信用分 ≥ 50<!-- AI-MUTABLE: 全量上线信用分阈值, type=int, range=[40, 60] --> → 全量上线。上线后持续监控 30 天，如 effectScore 下降 → 自动回滚。

## 九、大白话（十类必配）

### 1. 系统职责定义
策略引擎就是医院里的「开药系统」。S07 诊断引擎给了体检报告（「右腿骨折、中度」），S08 根据报告从药房（S04 策略包库）里选最合适的药。它不看片子，只看报告。

### 2. 核心设计
这个系统的核心设计是「六态调度树」——确诊就上猛药（强策略），有保留确诊就用温和的药（轻量策略），还在检查中就等着，查不出来就先用最安全的药兜底。不是「诊断→策略」一步到位，而是根据诊断的可信度分级处理。就像医生不是每个病人都开抗生素——100% 确定是细菌感染才开，80% 确定就先开退烧药观察。

### 3. 自我进化路线图
策略引擎会越用越聪明——第 1 阶段是人工编排的策略库（14 个策略包），第 2 阶段开始看家长反馈（「这个策略不适合我家孩子」），第 3 阶段系统自己从几十万条效果数据里学习哪个策略对哪种学生最有效。最终策略选择全自动——系统不再「选策略」，而是「赛策略」（同时试多个策略，留下效果最好的）。

### 4. 不可修改边界
有六类东西绝对不能碰。最硬的两类：策略包编码不能改（全系统的 Key，改了等于把所有药房的标签换掉，开药系统找不到药）；effectScore 计算公式不能改（改了等于把体温计的刻度换了，过去所有的体温数据都没法比了）。还有教育伦理底线——策略不能诱导学生上瘾、不能歧视、不能入侵学校课表。

### 5. 接口约定
策略引擎只和三个系统有「硬连接」：S07 给它诊断报告、S04 给它药房目录、S09 拿它的处方去配药。接口断了任何一个，策略引擎就没法工作——就像医院的药房进不去、化验单收不到、医生拿不到处方单。

### 6. 自我进化执行方法
任何让策略引擎变聪明的改动（改权重、改阈值、加新策略选择逻辑）必须走「影子→小范围试→全量推广」三步。影子是让新逻辑在后台偷偷跑一个月，看效果是不是比旧的更好。小范围试是只给 5% 的学生用新逻辑，确认没有害处才全量推广。跳过任何一步的直接后果是：新逻辑可能伤害学生。

### 7. 异常进化处置
如果策略引擎自己变聪明后反而更糟糕了（比如 effectScore 下降、放弃率上升），系统会自动回滚到上一个「安全版本」。就像手机系统升级后卡死了，自动重启回旧版本。回滚后策略引擎降级为最保守模式——只发最安全的策略（基础巩固包），等教研人员查清楚原因再恢复。

### 8. 应急熔断
策略引擎有两层保险。第一层是安全护栏——强策略必须同时满足 5 个条件（确诊、高概率、高置信度、样本量够、历史没有害过学生），缺一不可。第二层是信用分熔断——策略引擎的信用分掉到 0 分时，整个系统降级为只发基础巩固包（最安全的策略），等于放弃所有聪明做法、回到最保守模式。

### 9. 进化信用分
策略引擎有自己的「成绩单」——信用分。初始 50 分。策略效果好（学生后续正确率上升）→ 加分；策略导致学生放弃率高 → 扣分。分数低于 20 → 强策略全部关闭；低于 10 → 不再分析策略效果，所有策略随机选（最保守模式）。信用分的意义是：系统越值得信赖，就能用越强的策略；系统不可信时，只走最安全的路。

### 10. 内容与代码进化分离
策略引擎的进化分两条路：内容是「策略选择精度」（权重怎么调、策略怎么排序），代码是「策略选择效率」（选择算法跑多快、缓存怎么设计）。两者独立进化——策略选得好不好和策略选得快不快是两回事。就像医院的药房：药选得对不对（内容）和药抓得快不快（代码），应该分开优化。
