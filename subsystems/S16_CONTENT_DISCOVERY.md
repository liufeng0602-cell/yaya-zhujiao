## 第一层：内功 — 内容发现与录入的自我进化能力

### 1.1 系统定义与边界

**这个系统负责什么**：S16 是芽芽系统的「内容雷达和入库通道」。它负责发现、解析、校验、录入所有从外部进入芽芽知识体系的新内容——新教材、新题型、新错因模式、教研反馈中涌现的新知识结构。它不负责管理已经入库的知识（那是 S01），而是负责管「什么东西可以入库、怎么入库」。

**核心流程**：外部来源接入 → 文本解析与结构化 → 教研标注与审核 → 数据质量校验（过 S05 门禁）→ 结构化入库（写入 S01 knowledge.json / S03 error_taxonomy.json / S13 教材映射表）。

**这个系统不负责什么**：
- 不负责定义知识点的认知结构（S01 知识点管理系统）
- 不负责构建知识点之间的依赖关系（S02 知识图谱系统）
- 不负责定义错因的分类体系（S03 错因体系——S16 只录入新发现的错因实例）
- 不负责生成课堂内容（S09 课堂生成引擎）
- 不负责判断教材大纲来源规则的制定（S13——S16 消费 S13 的规则来校验录入内容的来源合法性）
- 不负责学生行为模式画像（S09 中引用的 S16「行为模式系统」指数据消费方向——S16 录入的错因/题型数据被 S09 画像系统消费）

> **设计理由**（约束）：S16 和 S01 的分界线是「入库前 vs 入库后」。S16 管管道——从外部世界到芽芽知识库的通道。S01 管库房——知识入库后的组织、演化、拆合。混在一起会导致两个问题：(1) 管道逻辑（OCR校验/格式转换/人工标注流程）污染知识管理逻辑（认知质发现/自动分级）；(2) 新内容录入的审批流程和已有内容的自我进化流程互相阻塞。

> **设计历史**（参考）：v1.0 首版。此前 S01 v3.1.x 消费清单中已预留 S16 定位（当前 S01 版本为 v3.1.14，具体消费关系以 S01 最新版为准）——`cognitive_primitives[].shareable` 被 S16 消费用于跨校复用验证。S01 维度五依赖 S13+S16+content_imported 事件完成教材变化感知的数据闭环。

### 大白话

S16 是芽芽的「海关和质检通道」。外面的东西要进芽芽的知识库——新教材、新题型、老师反馈的新错因——不能直接扔进去。得先过海关：验明正身（这题确实来自人教版五年级下册）、开箱检查（OCR 有没有识别错）、分类贴标签（这是计算题还是应用题）、质量检测（数据格式对不对、必填字段全不全）、最后入库（写入 knowledge.json 或错因库）。S01 是仓库管理员——管货架上怎么摆、要不要拆开重新打包。S16 是海关——管什么东西能进来、进来前要查什么。

### 1.2 当前核心能力

#### 内容发现来源（四类入口）

| 来源类型 | 触发方式 | 输入形态 | 目标输出 |
|---------|---------|---------|---------|
| 新教材/新版本 | 人工触发：教研人员上传新教材 PDF 或目录结构文件 | PDF 扫描件/EPUB/结构化目录 JSON | 解析后的章节结构 + 知识点候选列表（含 knowledgeCode 预分配） |
| 新教学大纲/课程标准 | 人工触发：教育部发布新课程标准后教研人员录入 | 结构化文档（章节/级别/目标描述） | curriculumLevel 映射更新建议 + 新增 KP 候选 |
| 教研反馈 | 半自动：教研人员通过标注工具提交「此处需要新增知识点」「此题代表新题型」 | 标注工具的 JSON 输出（含原始题目/学生错误样本/建议错因描述） | 待审核的新增知识点/错因/题型候选 |
| 错因聚类发现 | 自动触发：S03 错因体系的未分类错误占比超过阈值 → S16 接收聚类分析结果 | S03 输出的 `unclassified_error_cluster`（含错误样本集+统计特征） | 经教研确认后新增错因类型写入 error_taxonomy.json |

> **设计理由**（约束）：四类入口覆盖了「自上而下」（教材/课标变化）和「自下而上」（数据驱动的错因/题型发现）两条通道。如果只做自上而下，系统永远慢于教学实践——老师已经发现了学生的新错误模式但系统不认识。如果只做自下而上，系统无法感知教材改版这种结构性变化。

#### 录入流程（四步管道）

```
外部来源接入
  ↓
第一步：解析与结构化
  ├─ PDF/EPUB → 章节树 + 知识点原文提取
  ├─ 标注工具 JSON → 标准化录入表单
  └─ 错因聚类数据 → 候选错因 Schema
  ↓
第二步：教研标注与审核
  ├─ 知识点：分配 knowledgeCode（不透明标识符）、标注 curriculumLevel、填写 teachingGoal/keyPoints/commonMistakes
  ├─ 错因：命名 errorCode、定义 parentExplanation、绑定到 knowledgeCode
  └─ 题型：标注 questionType、绑定 skeletonType、标记 difficulty
  ↓
第三步：数据质量校验（对接 S05 门禁）
  ├─ R1-R9 铁规全量过检（新增内容触发增量校验而非全量重跑）。R10 为 S05 规划中的规则，当前 data_quality_gate.py 仅实现 R1-R9
  ├─ 新增专用校验：OCR 置信度阈值、来源合法性（S13 sourceType 合规）、重复检测（与已有 KP/错因去重）
  └─ BLOCK → 退回第二步修正；PASS → 进入第四步
  ↓
第四步：结构化入库
  ├─ 写入 S01 knowledge.json（新增 KP）或 error_taxonomy.json（新增错因）
  ├─ 发出 content_imported 事件（⚠️ 该事件待 S10 吸收定义——当前 S10 未定义此事件，实施前需先提交事件 Schema 到 S10。S10 吸收本事件 Schema 后，权威定义迁移至 S10_EVENT_AND_DATA_MODEL.md。本地的内嵌定义降级为「实现参考副本（authority: S10）」。S10 修改 Schema 后，S16 通过消费关系清单实时校验感知变更。事件被吸收前，S16 降级为直接调用下游接口写入）
  └─ 通知下游：S01 维度五感知新内容、S02 图谱需要增量重建、S09 课堂引擎缓存失效
```

**增量校验说明**：S05 门禁默认全量跑（acceptance-artifacts/data_quality_gate.py 扫描全部 KP（截至 2026-05-17 共 394 条——G1-G4: 205 条 + G5-G6: 189 条 = 394 条，来源：knowledge.json 的 knowledgePoints 数组长度）。KP 总数动态变化，具体数字以 knowledge.json 为准）。S16 录入新内容时触发增量模式——仅校验新增/修改的记录，耗时从 O(n) 降为 O(1)。但 R2（三文件交叉对账）和 R8（双向边一致性）仍须全量校验——新增一条 KP 可能破坏全局一致性。

### 1.3 它还不会什么（当前缺口）

1. **不会自动解析 PDF 中的数学公式**：当前 OCR 对数学符号（分数、根号、几何图形标注）的识别准确率不足以自动入库——需人工校对或等待专用数学 OCR 模型。
2. **不会自动对齐新旧教材的章节结构**：新旧教材的章节映射（新教材第三章第二节 → 旧教材第四章第一节）需人工建立一级目录映射——这是 S01 维度五阶段 2 的前置条件（见 S01 §2.1 维度五）。
3. **不会自动判断「这是一个新知识点还是一个已有知识点的变体」**：当教材改版后某个章节的内容有 60% 变化时，系统无法自动判断该不该新建 KP 还是修改已有 KP——需教研人员判定。
4. **不会自动为错因聚类结果命名**：S03 输出的 `unclassified_error_cluster` 是统计聚类——系统能发现「有一群错题的模式很像」但说不出「这群错题的共同本质是什么」——命名和定义需要人。
5. **不会跨语言处理**：仅支持中文教材。英文/其他语言教材需人工翻译后录入。
6. **标注工具尚未实现（Web UI 待建设）**——当前阶段标注流程依赖教研人员直接编辑 JSON 文件，生产效率低、易出错。此缺口对应进化路线图阶段 1 的「标注工具就绪」触发条件（详见 §2.1）。替代方案 SLA：若标注工具在 6 个月内未交付，启动 JSON Schema 自校验工具作为过渡方案。当前基准：JSON 直编模式下，熟练教研人员日均可处理约[待实测]条内容录入（含标注+校验）。

### 1.4 设计演化推理链

**为什么需要独立的内容发现管道，而不是让 S01 直接「吃」外部数据？** S01 的职责是管理和进化已有知识——它的核心算法（认知质发现、自动分级、隐性依赖发现）假设输入数据已经是干净、规范、经过校验的。如果把原始 PDF、教研随手标注的半成品直接灌给 S01，相当于让仓库管理员同时做海关检查——职责混乱、错误放大。

**为什么是四步管道而不是「AI 一把梭」？** 内容录入最危险的不是效率低，而是「悄悄错了」。一个 OCR 识别错误的知识点定义如果直接入库，S09 课堂引擎会基于错误定义生成数千堂课的讲解——错误被指数级放大。四步管道的每一步都是「检查点」：解析完检查 OCR 质量 → 标注完检查教研合理性 → 校验过门禁 → 入库发事件通知下游。每步可以独立回退，不污染下一步。

**为什么错因聚类发现走 S16 而不是 S03 直接写入？** S03 错因体系负责错因的定义、分类、绑定——它可以说「有一群错误看起来很像，可能是一个新错因类型」。但它不能自己定义新错因——因为定义一个错因需要教育语义判断（这到底是「概念混淆」还是「计算粗心」），这是人的领域。S16 在此扮演「经纪人的角色」：S03 发现线索→S16 组织人工确认→确认后写入 S03。

**为什么 S16 录入走内容进化流程而非代码进化流程？** 新增一道题、新增一个错因描述是内容增删——不影响算法、不改变数据结构。内容进化不需要北极星检查（加题不会让北极星偏离），不走信用分体系。但 S16 的质量校验（第三步）共享 S05 门禁的底线信号检测——如果新录入的错因文案含诱导性/歧视性语言，同样触发底线拦截。

> **设计理由**（约束）：以上四条推理链定义了 S16 在整个芽芽系统中的「管道」角色——不是生产者、不是消费者、不是管理者，是守门人。守门人的核心能力不是「多聪明」，而是「多可靠」。

### 1.5 第一层禁用清单

- 禁止绕过质量校验直接入库（跳过第三步→数据腐败防线失效）
- 禁止在 OCR 解析未达标（字符准确率 < 99.5%）时进入自动标注（错上加错）
- 禁止未经教研人员确认的错因聚类结果自动写入 error_taxonomy.json
- 禁止重复录入已存在的 KP（不检查去重→知识库冗余膨胀）
- 禁止修改 S13 的 sourceType 枚举值来绕过来源合法性校验

**检测手段**：第四步入库前，CI 管道自动执行增量质量校验脚本。校验失败→入库事务回滚，content_imported 事件不发。

### 1.6 退化场景处置

| 场景 | 处置 |
|------|------|
| OCR 服务不可用（PDF 解析依赖的 OCR 引擎宕机） | 教材录入降级为「人工逐条录入」模式——教研人员在标注工具中手动输入知识点原文。content_imported 事件正常发出但标记 `ocrFallback: "manual"` |
| S05 门禁脚本不可用 | S16 入库操作暂停——所有新内容标记为「待门禁恢复后校验」，不入正式库。已入临时库的内容对外不可见 |
| S01 knowledge.json 不可写（文件锁定/权限问题） | S16 入库失败→content_imported 事件不发→重试队列（指数退避，最大 3 次，间隔 30s/5min/30min）→3 次均失败→仪表盘红色告警。重试期间数据暂存于 `{data_dir}/s16_pending_queue.jsonl`（追加写，每次重试从队头消费）。3次均失败后数据保留在队列中，通过仪表盘告警通知人工处理。进程重启不丢数据（磁盘持久化）。队列文件需定期清理（≥30天未处理的条目归档或人工处置）。 |
| S13 来源规则未就绪 | 来源合法性校验降级为「仅检查 sourceType 枚举值不为空」——不验证教材版本/课标编号的格式合法性 |

### 大白话（进化路线图）

S16 的进化路线是让海关从「人工逐件开箱」变成「智能安检机」。现在是阶段 0——安检机还没装好，全靠人工手检（教研人员手动编辑 JSON）。阶段 1 把安检机装上（OCR 自动提取 + 标注工具上线 + 数据开始积累）。阶段 2 安检机开始帮忙——机器先扫一遍预填表单，人只确认不通过的地方（标注一致率 85%）。阶段 3 大部分包裹免检——80% 的字段机器自动填且多个教研员的标注结果高度一致，人只确认教学目标和关键点。阶段 4 低风险货物自动放行——教材改版中的顺序调整、非核心知识点删除标记系统自己处理，只有新增知识点和新增错因才需要人签字。

但机器不是一直进步的——如果 AI 标注的被拒绝率超过 10% 或自动入库的东西被回退超过 3 次/月，系统自动退回更保守的阶段。宁可多让人看一眼，也不能放进去错误的知识定义。

### 2.3 内容进化与代码进化分离治理（原则十二）

| 类型 | 定义 | S16 场景示例 | 流程 |
|------|------|------------|------|
| 内容进化 | 新 KP 录入、错因描述增删改、题型分类调整、教材章节映射更新 | 教研反馈中确认的新错因写入 error_taxonomy.json；新教材知识点写入 knowledge.json | 教研审核→S05 质量校验→灰度发布（5% 学生，1 教学周期）→质量监控→全量 or 回滚 |
| 代码进化 | OCR 解析算法、去重检测算法、AI 自动标注模型、增量校验脚本 | 修改 OCR 引擎参数；调整重复检测的文本相似度算法；升级自动标注的模板匹配逻辑 | 路线图触发→北极星检查→信用分→安全执行管道（影子→A/B→全量）→回滚 |
| 混合进化 | 新增内容类型（如从仅支持「知识点/错因/题型」扩展到支持「教学视频/互动素材」） | 新内容类型需要新增数据结构字段 + 新解析算法 + 新标注表单 | 两套流程都过。任一套不通过→整体搁置 |

**关键约束**：
- 内容进化不经过北极星检查（加几道题不会让北极星偏离），不走信用分体系
- 但内容进化共用底线信号检测——若新增错因文案含诱导性/歧视性语言，触发底线拦截
- 代码进化走完整十二条原则流程
- S16 进化路线图中「阶段 4 自动入库低风险内容」属于混合进化——低风险判定算法（代码进化）+ 实际写入操作（内容进化）。自动判定算法走代码进化流程（北极星+信用分+安全管道），写入动作走内容进化流程（教研审核+S05 校验）。任一类失败→该次自动入库搁置

> **设计理由**（约束）：S16 最频繁的变更就是录入新内容——新教材每 3-5 年才改一次，但教研反馈可能每周都有。如果每次录入都走北极星检查 + 信用分，教研人员的反馈永远堵在管道里。

### 大白话（内容/代码分离）

S16 的日常操作就跟海关放行货物一样——每天都有新包裹（教研反馈的新错因、新题型），每个都过安检（S05 门禁）但不需要每次都惊动总统（北极星检查）。安检机坏了要修（改 OCR 算法）→走重大工程审批（代码进化流程）。开了新业务线（比如增加「教学视频」这种新内容类型）→既加柜台又改安检流程（混合进化），两套审批都过。

### 2.5 应急熔断（原则十一）

| 级别 | 触发条件 | 效果 |
|------|---------|------|

### 大白话（核心设计）

S16 内容发现系统就像出版社的「选题编辑」——它负责发现值得教的新知识、新错因、新题型，然后把它们审核、整理、入库。四步管道就是编辑部的标准流程：外部来源接入（相当于收到投稿）→文本解析（相当于初审——OCR 就是「用机器认字」）→教研标注（相当于专家审稿——教研人员判断这个知识点值不值得教）→结构化入库（相当于出版——知识正式进入芽芽的知识库）。每一步都有质量把关——S05 门禁就是「校对室」，过不了校对的内容不能出版。

### 大白话（不可修改边界）

S16 有七道不能碰的红线，最核心的几条：1) curriculumLevel（了解/理解/掌握/运用）只能原样引用新课标，不能自己造——这是教育部定的标准，不是系统能自己决定的。2) knowledgeCode 的编码规则不能改——改了全系统不认识这个知识点了。3) content_imported 事件的 schema 结构不能变——这是 S16 和全系统通信的唯一语言。4) 已经入库的学生原始数据不能改——那是「病历」，不能篡改。5) S05 门禁的核心规则（R1-R9）不能绕过——质量关是死规矩。

### 大白话（接口约定）

S16 对外有三条主要接口：1) 消费关系清单定义了 S16 提供给下游的「content_imported 事件」——相当于告诉 S01/S02/S09/S10「有新东西入库了，你们看着办」。2) 消费 S05 的门禁接口——每次录入都要过质量校验。3) 消费 S01/S02/S03 的写入接口——新知识点写入 knowledge.json，新错因写入 error_taxonomy.json。如果下游服务不可用，S16 有自己的降级策略——排队重试，不丢数据。

### 大白话（异常进化处置）

S16 的进化不是一帆风顺的——如果系统试图绕过 S05 门禁、试图修改 curriculumLevel、试图自己决定 knowledgeCode 编码，就是异常进化，必须拦截。越界告警分三级：轻（只是想了一下）→退回记录日志；中（写了方案但没执行）→冻结 7 天扣信用分；重（真的执行了）→永久封禁+全量回滚。异常进化的处置有一条铁律：重级越界必须通知人工管理员，不能悄悄放过。

### 大白话（应急熔断）

S16 有三道熔断开关：1) 轻量级熔断——某个来源连续 5 次被 S05 拒绝→停掉这个来源，其他来源照常。OCR 连续 3 次不达标→停 OCR，改人工模式。2) 子系统熔断——教研标注一致性跌破 60%→整个 S16 暂停录入，别让低质量内容污染知识库。3) 全系统熔断——北极星底线被触碰→全部停止，人工接管。熔断后要恢复正常也不容易——连续 2 次正常校验通过 + 根因修复 + 产品经理签批，三条件缺一不可。

### 大白话（进化信用分）

S16 的信用分就是系统的「信用记录」——录入的内容越准、教研越认可、北极星越安全，信用分越高。信用分低就不能升级（比如不能从 AI 辅助标注升级到 AI 自动标注）。三条独立账本：低风险（调个阈值之类的）、中风险（改 OCR 参数）、高风险（改自动入库判定算法）。最低的那个账本分数决定了整体信用——就像木桶最短的那块板。违规扣分很重（绕过门禁直接扣 30 分），但做好事也涨分（连续 200 次准确录入加一次分）。破产了也不怕——连续 30 天不犯错就能从 10 分重新开始。重置规则：大版本升级、群体模型重建、产品经理手动重置——这三种情况全部归零重来，14 天冷静期后再慢慢积累。

### 大白话（消费关系清单）

S16 是大楼的「进货通道」——它消费上游（S05 门禁=质检员、S13=来源规范、S03=错因线索、OCR=扫描仪），产出下游（S01=仓库、S02=货架图、S03=错因分类本、S09=厨师）。最重要的是 content_imported 事件——就像进货单，所有下游都凭这张单子知道「新货到了」。如果进货单丢了（事件总线宕机），货进了仓库但没人知道——24 小时内没人来领，系统就报警。P0 级别的故障是全系统数据不一致——比如 knowledgeCode 冲突导致两个完全不同的知识点用了同一个编号。

### 3.2 自我进化执行方法（原则七）

#### 五类角色权限台账

| 消费者 | 读什么 | 权限 | 权限回收规则 |
|--------|--------|------|------------|
| 芽芽自己 | 触发条件+AI-MUTABLE 阈值+教研一致率统计 | 低风险自主执行——自动标注预填、重复检测、一致率统计。无检查模块修改权 | 越界 3 次→降级 30 天（仅监控）；北极星失败 3 次→仅监控 60 天 |
| AI 编程工具 | 路线图+消费清单+不可修改边界 | 仅读取+方案生成，无执行权 | 越界 3 次→永久冻结 |
| 产品经理（liufeng） | 各维度阶段+高风险方案+管道健康度 | 最高权限——可签批批量入库/熔断/一票否决 | 信用分重置每年每子系统 ≤ 1 次 |
| 教研人员 | 标注工具+审核任务+中风险方案+内容进化边界 | 标注新内容、审核 AI 建议、确认错因类型。不可签批代码进化变更 | 连续 30 次标注被门禁退回→标注工具锁定，需产品经理解锁 |
| 审查员（人或AI） | 修改方案+消费清单+入库内容抽检 | 标记通过/疑似/不通过，不可批准执行 | 连续 5 次 override 后 30 天内否决不可 override |

#### 五步执行流程

**第一步：信号采集与触发判定**
- 数据来源：content_imported 事件累积 + S03 错因聚类阈值 + 教研反馈队列长度
- 噪音过滤：≥2 周教学日持续满足才算有效触发
- 外部事件识别：教材改版公告（教育部发布）→ 人工录入→自动标记 confoundingEvents

**第二步：方案生成**
- 读边界→读影响面（遍历消费清单）→生成修改方案和影响评估报告→停住等审核
- 全局注册表宕机：30 秒无响应→持锁进化暂停保留现场→恢复后检查→完好的继续/不完好的回滚→24 小时不恢复→自动回滚

**第三步：审核**
- 三级权限：低风险自主执行→中风险人工审核（7 天超时暂缓）→高风险人工签批（绝不超时自动）
- 中风险二次申诉：超时驳回后系统可基于新数据（≥3 个月额外观察）重新提交

**第四步：安全执行**
- 影子模式：在脱敏后的历史数据上回放入库影响（studentId 哈希化、去除设备信息、时间戳精确到教学周），通过 G6 北极星底线信号检查（底线无触碰→正信号至少 1 项改善→通过）
- A/B 测试：新录入内容仅对 5% 学生可见，通过 G7 效果指标监控（PSI<0.1 + 北极星无恶化）
- 全量上线：A/B 通过→人工终审→全量上线，监控 30 天稳定期
- 回滚条件：PSI≥0.1 / 超基线 95% 区间 / 北极星底线触发 / 越界 / 下游 BLOCK

**第五步：记录验证**
- 30 天稳定期：成功→降低下次同类修改审核门槛+涨信用分。失败→收紧条件+回滚复盘
- 回滚复盘含整改绑定迭代版本+上线时间，超期未落地→自动冻结

### 大白话（进化执行方法）

S16 的进化执行跟其他子系统一样——五类角色各管各的、信用分三本账、五步流程从信号到验证。区别在于 S16 的「低风险自主执行」范围很窄——自动标注预填、重复检测、一致率统计——这些不涉及修改数据库。真正写库的操作至少是「中风险」（AI 辅助标注 → 人确认）或「高风险」（自动入库 → 人签批）。越界告警对 S16 特别敏感——如果 AI 试图绕过 S05 门禁直接把 OCR 结果写进 knowledge.json，三个子分各扣 30，相当于驾照扣满吊销。

## 第四层：北极星 — 终极约束

### 4.1 核心使命与本系统的关系

芽芽核心使命：**让每个孩子，都拥有最适合自己的学习旅程。**

S16 是芽芽的「感知器官」——如果感知器官出错，孩子学的东西可能就是错的。错误来源有三：
- **来源层面**：录入了非正规教材的内容→教的是课标外的东西
- **解析层面**：OCR 识别错误→知识点的定义本身就是错的
- **校验层面**：门禁被绕过→脏数据污染整个知识库

### 4.2 三层检测信号与权重配比

| 优先级 | 信号层 | 判定规则 |
|--------|--------|---------|
| 最高 | 底线信号 | 任何一项触碰 → 直接 BLOCK |
| 中等 | 负信号 | 任一恶化 → BLOCK。正信号上涨不能抵消 |
| 基础 | 正信号 | 仅当底线/负信号均无恶化时，正信号至少 2 项改善或持平 → 通过 |

**正信号（方向对时出现）**：

| 信号 | 指标 | 对端指标（不能降） | 数据来源 |
|------|------|-------------------|---------|
| 录入效率提升 | 平均单条录入耗时下降（人工耗时 + OCR 处理时间） | 门禁 BLOCK 率不得上升（不能为了快牺牲质量） | content_imported 事件的 totalDurationMs 字段 |
| 教研一致性提升 | 多教研标注一致率上升 | AI 自动标注拒绝率不得上升（不能用简单内容刷一致率） | content_imported 事件的教研标注结果对比 |
| 下游消费及时性 | content_imported 事件发出后 24h 内消费者完成处理的比例上升 | 消费者处理失败率不得上升 | 事件总线消费延迟监控 |
| 去重准确率 | 重复检测的准确率（人工确认的重复标记 vs 系统判断） | 漏检率不得上升（宁可多报疑似重复，不能漏掉真正的重复） | 教研标注工具中的「标记为重复」确认率 |

**负信号（方向偏时出现）**：

| 信号 | 指标 | 触发条件 | 严重程度 |
|------|------|----------|---------|
| 门禁 BLOCK 率飙升 | S05 门禁对 S16 来源的内容 BLOCK 率 > 20%（连续 30 次录入） | 任一来源类型触发 | 中等——暂停该来源的自动预填，回退全人工 |
| 教研一致性下降 | 标注一致率连续下降 ≥ 3 个月 | 全来源触发 | 轻度——发出教研对齐培训建议 |
| AI 自动标注质量恶化 | 自动标注的教研拒绝率连续上升 ≥ 2 个月 | AI 辅助来源触发 | 轻度——收紧自动标注的置信度阈值 |
| 录入积压 | 教研反馈队列长度 > 50 条且等待时间 > 7 天 | 任一来源类型触发 | 轻度——告警通知产品经理追加教研人力 |

**底线信号（绝对不可触碰，三级违规分级处置）**：

| 信号 | 判定规则 | 违规分级 | 触发后果 |
|------|----------|---------|---------|
| 篡改课标级别 | AI 在新录入的 KP 中修改了 curriculumLevel 为与原课标定义不符的值 | 一律重 | 立即回滚+违规记录+管理员告警 |
| 污染学生原始数据 | S16 的错因录入过程中读取或修改了 answer_submitted 原始数据 | 一律重 | 进化流程阻断+数据完整性告警 |
| 录入歧视性/诱导性内容 | 新录入的错因描述或知识点定义含负面标签/诱导性语言 | 轻/中/重 | 轻度=拦截该条；中度=拦截本批次+信用分清零；重度=全系统熔断 |
| 绕过质量门禁 | AI 或人工绕过 S05 门禁直接写入 knowledge.json | 一律重 | 全量回滚+信用分清零+管理员告警 |
| 录入非教学必需的学生信息 | 标注表单中收集了超出教学需要的学生个人数据 | 轻/中/重 | 见总纲分级处置 |

答：content_imported 事件在 S10 中尚未定义——S16 发出的事件无人监听，下游（S01 维度五/S02/S03/S09）感知不到新内容入库。解决路径：(1) 先将 content_imported 事件 Schema 提交给 S10 维护者吸收；(2) 在事件被吸收前，S16 降级为直接调用下游接口（S01 knowledge.json 文件写入→S02 重跑 build 脚本→S03 错因库直接写入）。注意：S02 当前无「增量 rebuild API」，降级时需重跑全量 build 脚本（scripts/build_knowledge_graph.py），绕过事件总线。此降级方案已记录在 §3.1 S16 内部消费关系表格中。

## 大白话（全文总结）

S16 是芽芽的「海关和进货通道」——管什么新东西能进芽芽的知识库、怎么进、进之前要查什么。

四类东西会触发进货：教育部发新教材了（自上而下）、新课程标准下来了（政策驱动）、教研老师说「这个学生错误模式以前的分类里没有」（一线反馈）、系统自己发现「有一群错误很像但我们的错因库不认识」（数据驱动）。

进货流程四步走：先进安检机扫描（OCR 解析 PDF 或读取标注工具的 JSON）→然后海关人员开箱检查（教研人员标注知识点属性、确认错因定义）→过质检线（S05 门禁自动跑 9 条铁规）→合格盖上「已验」章入库（写入 knowledge.json 或错因库，发出 content_imported 进货单通知所有人）。

机器的进化路线是这样的——现在是纯人工（教研人员手动编辑 JSON 文件）。以后装上传送带（标注工具 + OCR 自动提取）。再以后机器帮你预填表单——人只要确认不一致的地方。再再以后大部分低风险货物免检——教材改版里的顺序调整、非核心知识点删除机器自己处理——只有真正的新知识点和之前没见过的新错因才需要人签字。

但机器的权限有边界——不能改课标级别（那是国家定的）、不能绕过质检线（那是最后一道锁）、不能动学生的原始答卷（那是病历）。机器越界分轻重——只是想想→警告；写了方案→冻结 7 天；真改了→永久封号回滚。

最关键的一句——S16 是守门人，不是生产者。它的核心能力不是「多聪明」，而是「多可靠」。
