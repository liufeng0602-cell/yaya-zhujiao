## 一、定义与职责

### 1.1 这个系统负责什么

知识点管理系统是整个芽芽引擎的「食材清单」。它的唯一职责是：定义「芽芽要教什么」。

具体包括：
- 知识点的定义、编号、年级归属、教材出处
- 每个知识点的认知级别（新课标 A/B/C/D）
- 知识点之间的前后依赖关系（学习路径层 + 诊断依赖层）
- 每个知识点的教材页码、教参教学目标、关键点、常见错误
- 所有知识点的质量校验（必填字段完整、级别标签一致、双向边一致）

### 1.2 这个系统不负责什么

- 不负责诊断学生哪里不会（那是诊断引擎的事）
- 不负责决定怎么教（那是策略引擎的事）
- 不负责生成具体的课堂内容（那是课堂生成引擎的事）
- 不负责解释学生为什么犯错（那是错因体系的事）

### 1.3 KP 的本质局限

KP（Knowledge Point，知识点）是人类的「认知拐杖」。

教材编者为了大规模标准化教学，把数学切成一块一块的：今天学分数的意义，明天学通分，后天学分数的加减法。有了 KP，一个老师才能同时教五十个孩子。

但它不是真实的认知结构。真实的大脑学「异分母分数加减法」时，同时发生着六件事：
- 理解「分母不同」意味着什么（概念识别）
- 调用通分技能（程序执行）
- 检索最小公倍数（记忆提取）
- 执行分子相加（算术运算）
- 判断结果要不要约分（规则判断）
- 从应用题中提取分数信息（情境翻译）

这六个认知动作分属六个不同的认知维度。但在传统 KP 体系里，它们被捆在同一个标签下，叫「分数加减法」。

当系统说「这孩子分数加减法没掌握」——等于没说。因为他可能前三步很流畅，只在第五步「判断约分」上犹豫。

这就是 KP 的本质局限：它在「教学管理」层面极其高效，但在「认知诊断」层面是精度的天花板。芽芽的终局，不是管理 394 个 KP，而是理解每个孩子大脑里所有认知维度的状态图谱。

| `confidence` | float | 置信度 |
| `status` | string | 状态。当前全员为 `needs_textbook_body_review`（教材原文核对未完成），核对完成后更新为 `verified`。非教材类内容使用 `needs_review` / `verified` / `needs_update` |

| 字段 | 类型 | 说明 |
|------|------|------|
| `textbookPage` | object | 教材页码（含 `pageStart`、`pageEnd`、`lesson`、`source`）。非教材类内容此字段不存在 |

#### 2.2.3 文件级元数据字段（每个 knowledge.json 的顶层）

文件级字段描述「这一批数据属于什么范围」，与 KP 级字段形成双重校验——文件声明了范围，KP 不得超出。

| 字段 | 类型 | 说明 | 当前值 |
|------|------|------|--------|
| `scope.subject` | string | 本文件内容所属学科代码 | `MA` |
| `scope.stage` | string | 学段代码 | `P` |
| `scope.grades` | int[] | 年级范围，支持不连续数组，如 `[5,6]` | G5-G6 文件为 `[5,6]`，G1-G4 文件为 `[1,2,3,4]` |
| `scope.version` | string | 教材版本代码。PEP=人教版 / BNU=北师大版 / JIANGSU=苏教版 | `PEP` |
| `scope.contentType` | string | 本文件的内容类型，与 KP 级 `contentType` 一致 | `textbook_kp` |
| `scope.levelingFramework` | string | 本文件使用的分级体系 | `curriculum_2022_math` |

质量门禁应校验：`scope` 字段与每个 KP 的对应字段一致——文件说自己是 PEP 小学数学，KP 就不能声明自己是 BNU 初中语文。

### 2.3 分级规则：多框架共存设计

芽芽完全对齐《义务教育数学课程标准（2022年版）》的四级认知分类。**原封不动，不做翻译、不合并、不重新映射。**

| 芽芽级 | 新课标 | 含义 | 得分 |
|--------|--------|------|------|
| A | 了解 | 能识别、说出、举例 | 3 |
| B | 理解 | 能解释、推导、说明原因 | 6 |
| C | 掌握 | 能熟练计算、规范解题 | 9 |
| D | 运用 | 能解决新情境问题、迁移应用 | 12 |

`levelScore` 由 `curriculumLevel` 程序推导（A=3, B=6, C=9, D=12），手工填写被 R1 铁规禁止。

394 个 KP 分级分布：A=46 / B=106 / C=204 / D=38。

**旧版 6 维体系已废止**（73.8% 数据为虚假值，2/6 维度为死维度）。`levelDims` 字段在 `knowledge.json` 中保留仅用于向后兼容，不再作为权威分级依据。权威字段为 `curriculumLevel`。

**多框架共存设计**：A/B/C/D 体系是 2022 版数学课标的产物。其他学科和国家有不同的分级体系——语文课标分「识字与写字/阅读/表达与交流」三个实践领域，每个领域内再分层；英语课标分「语言技能/语言知识/文化意识/学习策略」四个维度；IB 课程用 1-7 分制；A-Level 用 A*-E。

系统通过 `levelingFramework` 字段区分框架，不把不同标准塞进同一套 A/B/C/D 里。新增学科时，定义该学科的 `levelingFramework` 标识 + 级别枚举 + levelScore 映射规则，不与现有数学框架冲突。如果一个 KP 同时属于多个框架（如 IB 数学与中国数学共用部分内容），`levelingFramework` 使用数组，`curriculumLevel` 同步扩展为按框架索引的映射。

**无课标内容的 fallback 机制**：对于没有国家课标体系的 `contentType`（如 `skill`、`topic`、`course_concept`、部分 `competency`），`levelingFramework` 填写 `"generic"`，`curriculumLevel` 填写 `"NA"`（Not Applicable），`levelScore` 填写 `0`。通用分级框架 `generic` 不包含 A/B/C/D 级别，仅通过 `observedDifficulty` 字段（Phase 2 后启用）提供数据驱动的难度感知。策略引擎读取 `generic` 框架时，跳过课标级别的难度调节，完全依赖 `observedDifficulty` 做决策。

### 2.4 知识图谱：两层架构

图谱是 KP 之间的依赖关系网络，包含两层：

**学习路径层**（用户可见）：单元内部的顺序链 + 跨册过渡。遵循教材排列顺序，在 UI 中展示为线性进度条。

**诊断依赖层**（引擎内部）：所有硬依赖，包括跨单元和跨年级。被诊断引擎用于根因溯源——如果学生不会 B，引擎沿依赖链向上追查是否 A 没掌握。

硬依赖判定标准（四条满足任一）：

1. **定义包含**：B 的定义/公式直接使用 A
2. **教材刚性顺序**：必须先复习 A 才能教 B
3. **操作前置**：解 B 必须先执行 A
4. **概念层级**：A 是抽象概念，B 是具体应用

软依赖不连边：启发关系、顺带提及、平行主题。

**非教材内容类型的硬依赖判定标准**：以上四条标准为 `textbook_kp` 设计。对于其他 `contentType`，硬依赖按以下规则调整：

|| contentType | 硬依赖标准 | 示例 |
||---|---|---|
|| `course_concept` | 「概念层级」+「课程大纲顺序」。不使用「教材刚性顺序」（无统一教材） | 「矩阵的秩」→「线性方程组的解」 |
|| `skill` | 「实操前置条件」——必须先学会 A 才能执行 B | 「安装 requests 库」→「用 requests 发送 GET 请求」 |
|| `topic` | 「内容引用关系」——B 的理解直接依赖 A 的概念定义 | 「Transformer 注意力机制」→「多头注意力变体」 |
|| `competency` | 「能力等级递进」——低等级能力是高等级的前置 | 「能读懂数据报告」→「能从数据中提出业务建议」 |

诊断依赖层的溯源逻辑对所有 `contentType` 一致——沿前置链向上追查根因。学习路径层的展示方式则随 `contentType` 变化：`textbook_kp` 展示为教材章节顺序链，`skill` 展示为技能树，`topic` 展示为概念地图。

### 2.5 图谱构建方法

由 `scripts/build_knowledge_graph.py` 全量重建，四步完成：

1. **重置所有边**，按 `knowledge.json` 的单元排序建单元内链
2. **加跨单元硬依赖**（人工审定的约 12 条依赖）
3. **加跨年级硬依赖**（人工审定的约 48 条依赖，覆盖 1-6 年级）
4. **双向一致性检查**（自动修复，处理跨图引用）

跨图引用（如 G5-G6 节点的前置指向 G1-G4 节点）属于正常现象，不在同一图内检查反向边。

### 2.6 教材来源规则

知识点从以下来源按优先级提取：

1. 人教版学生教材（文本 PDF，`pymupdf` 提取）
2. 人教版教师教学用书（扫描版 PDF，OCR 识别）
3. 《义务教育数学课程标准（2022年版）》

教学目标的权威出处是教师教学用书，原文录入 `teachingGoal` 字段，来源标注在 `teachingGoalSource` 字段。AI 生成的占位文本必须替换。

### 2.7 质量门禁

任何对 `knowledge.json`、`levels.json` 或 `knowledge_graph.json` 的改动，必须通过 `data_quality_gate.py` 的 9 条铁规（R1-R9）校验。返回 BLOCK 则禁止提交。

与知识点管理系统直接相关的铁规：

| 铁规 | 名称 | 检查内容 |
|------|------|----------|
| R1 | 算分程序计算 | `levelScore == sum(levelDims)` 主校验，fallback 到 `curriculumLevel` 映射 |
| R2 | 三文件交叉对账 | `knowledge.json` ↔ `knowledge_graph.json` ↔ `levels.json` 代码集完全一致 |
| R5 | 必填字段完整 | 15 个必填字段全部存在 |
| R6 | 级别标签一致 | `knowledge.json` 的 `level` 与 `levels.json` 的 `level` 完全匹配 |
| R8 | 双向边一致 | 若 A.prerequisites 含 B，则 B.next 必含 A |
| R10 | scope一致性 | 每个 KP 的 `subjectCode`/`stage`/`contentType`/`textbookVersion`/`levelingFramework` 与文件 `scope` 中的对应字段一致 |

### 2.8 内容类型体系

芽芽管理的「内容单元」不限于 K12 教材知识点。以下是系统设计支持的完整内容类型谱系：

| contentType | 中文名 | 例子 | 有教材？ | 有课标？ | 有固定学习路径？ | grade 值 |
|---|---|---|---|---|---|---|
| `textbook_kp` | 教材知识点 | 「异分母分数加减法」 | 是 | 是 | 是 | 1-12 |
| `course_concept` | 大学课程概念 | 「矩阵的特征值分解」 | 参考教材（非统一） | 否（无国家课标） | 弱（课程大纲，因人而异） | 0 |
| `skill` | 技能 | 「用 Python 写一个 web scraper」 | 否 | 否 | 否（高度个性化） | 0 |
| `competency` | 能力框架条目 | 「能从数据中发现趋势并提出业务建议」（数据分析师能力框架） | 否 | 否（有行业标准） | 弱（能力等级递进） | 0 |
| `topic` | 自由专题 | 「AI 提示词工程的思维链技巧」 | 否 | 否 | 否（完全自由） | 0 |

**设计原则**：同一套数据结构承载不同类型的「内容单元」。认知质模型不绑定 `contentType`——「异分母识别」这个认知动作在小学分数加减法和大学矩阵运算中都可能出现。诊断引擎和课堂生成引擎通过 `contentType` 判断可用策略——教材 KP 用教材骨架模板，技能类内容用项目式学习骨架，专题类内容用探究式骨架。

**认知质是跨内容类型的**：同一个认知质可以在不同 `contentType` 中复用。「从情境中提取信息」在数学应用题（`textbook_kp`）和语文阅读理解（`textbook_kp`，不同学科）中是同一个认知动作；「将问题分解为子步骤」在编程技能（`skill`）和数学解题（`textbook_kp`）中是同一个认知动作。诊断引擎的错因匹配逻辑对教材 KP 和大学课程概念使用同一套认知质体系——`contentType` 只影响课堂生成引擎的骨架选择和策略包的适用范围，不影响认知质本身的定义和诊断逻辑。

**skill 与 topic 的精确边界**：`skill`（技能）是可操作、可练习、可熟练化的动作序列——学生需要亲手做，系统需要出实操题；`topic`（专题）是知识性、概念性的内容，需要理解和思考但不一定需要亲手操作——系统出理解题和讨论题。判断标准：能否设计一个「动手练习」？能 → `skill`，不能 → `topic`。

**大学阶段的特殊设计**：不试图为大学建立「教材→单元→课时」映射（全国无统一教材），不试图给大学课程贴 A/B/C/D 课标级别（无统一课标）。大学内容以 `course_concept` 类型存储，保留 `name`、`keyPoints`、`commonMistakes`、`tags`、`domainTags` 等通用字段，`textbookPage`、`textbookVersion`、`teachingGoalSource` 等教材专属字段留空。大学图谱的依赖关系以「课程大纲顺序」和「概念层级」为依据，不使用教材刚性顺序标准。

**非学科教育的特殊设计**：AI 学习、编程、职业技能等内容以 `skill` 或 `topic` 类型存储。无课标、无教材、内容更新快（半年过时）。数据结构保持不变——`confidence` 字段天然适合表示「此内容的新鲜度」，`status` 字段可以标记 `needs_update`。知识图谱的依赖关系以「实操前置条件」为依据——如「会用 API 调试工具」→「用 LLM API 构建聊天机器人」。

## 三、它还不会什么（当前缺口）

### 缺口 1：不会自己发现该拆还是该合

**具体例子**：系统用了三年，几百个学生在「分数加减法」上反复卡住。数据分析明确显示——学生总是在「异分母通分」那一步崩，同分母部分完全没问题。这意味着「分数加减法」其实应该是两个独立的知识点。但目前系统只存了一个静态 JSON 条目，它看得见这个信号，却没有能力说「我建议拆开」。

反过来也一样：掌握了「长方体表面积」的学生 95% 自动掌握了「正方体表面积」。系统该能提出合并建议。

### 缺口 2：不会自己重新分级

「小数乘法」这个 KP 在 2025 年的孩子身上可能比 2015 年学得快——因为生活中接触小数更早了。但它的 `curriculumLevel` 是静态标注的 C 级。如果实际数据显示学生平均 20 分钟就掌握了一个 C 级知识点，系统不会自己说「建议降为 B 级」。

这不是质疑新课标——课标是法规不能改。但系统应该在「实际教学难度」这个维度上做自适应标记，让策略引擎知道「虽然课标说这是 C 级，但数据显示实际难度接近 B 级」。

### 缺口 3：不会自己发现隐藏的依赖关系

现在图谱里的硬依赖是人手工判定的（用那四条标准逐对判断）。但存在人没发现的隐性关系。比如学了某个几何知识点的学生，在另一个看似无关的代数知识点上速度明显快——因为几何训练了某种空间推理能力，对代数有帮助。这种跨域的能力传导规律，人类教研员几乎不可能发现。

### 缺口 4：不会自己感知教材变化

如果人教版出新版教材，知识点变了、章节顺序变了、新增了内容——需要手工重来一遍。系统不会自动对比新旧教材、自动发现变化、自动生成更新建议。

## 四、设计演化推理链

### 4.1 为什么 KP 是必要的起点

KP 是人类教育体系的「原子单位」——教材以此划分章节、课标以此定义目标、考试以此命题。芽芽必须从这里起步，因为它面对的是一个已经用 KP 语言运行了几十年的真实教育世界。不是芽芽选择了 KP，是 KP 已经在那里。

### 4.2 为什么 KP 最终会被消解

但 KP 是「教学管理」维度的最优单元，不是「认知诊断」维度的最优单元。同一个 KP 下的学生，错因分布常常稳定地分成两个互不重叠的集群——这表明同一个 KP 下存在两个不同的认知维度。如果诊断引擎继续以 KP 为最小诊断单元，它对学生的理解永远不会超越「分数加减法没掌握」这个粒度。

从 KP 到认知质的进化，不是放弃 KP——KP 仍然是教材编排、课标对齐、进度管理的必要容器——而是在 KP 之下新增一层更细粒度的认知维度，让诊断发生在对的颗粒度上。

### 4.3 为什么自适应分级不会质疑课标

课标的 A/B/C/D 分级的定义权在国家标准，这是一条不可逾越的边界。但课标的级别是「应然」——这个知识点在国家标准中应该属于哪个认知层次。实际教学中出现的是「实然」——这个知识点在真实学生群体中的难度感知。

系统不修改 `curriculumLevel`，但新增一个独立字段 `observedDifficulty`，描述数据看到的真实难度。两个维度共存：课标级别是法规锚点，观察难度是教学信号。

### 4.4 为什么教材感知能力必须从第一天开始积累

教材不是每年都变，但一定会在某个时间点变。当它变的时候，如果没有历史数据参照，系统只能从零重新标注。但如果系统从第一天就开始记录「这个 KP 在现行教材中的精确位置、典型例题、教学目标」，当新教材出现时，它有参照系——可以自动发现「这些 KP 的章节顺序变了」「这两个 KP 被合并了」「这是新增内容」。这种感知能力不是某一天突然长出来的，是从第一天开始沉默积累的。

### 4.5 为什么可扩展性预处理必须现在做

现在只有 394 个 KP，小学数学人教版。但芽芽的目标是 K12 全学科 → 大学 → 终身学习。如果不在数据量最小时把扩展通道开好，后果不是「将来加字段」，是「将来从零再建」。

具体来说：

**学科代码现在不加的代价**：语文有 3000+ 常用汉字、500+ 阅读知识点、200+ 写作知识点。等数学 2500 条 KP 入库后，再加 `subject` 和 `subjectCode` 字段，每条都要回填。现在 394 条，加字段就是设默认值；5000 条时加字段，等于把数据录入工作重做一遍。

**知识域分类现在不统一的代价**：G1-G4 文件有 `l1_domain`/`l2_module`/`l3_topic` 三层结构，G5-G6 文件没有。如果不趁现在规范化，将来语文、英语各自长出自己的分类体系，跨学科搜索「哪些内容训练了空间推理」变成不可能。

**分级框架现在不标注的代价**：A/B/C/D 当前只对数学有效。加语文时如果没有 `levelingFramework` 字段加以区分，就会出现「用数学的 C 级去标语文的某个掌握程度」这种语义错误。不同学科的课标体系完全不同——语文分实践领域、英语分维度、IB 用数字——把不同标准混在一起就是灾难。

**contentType 现在不区分的代价**：大学和 AI 学习是真实需求。如果系统只认「教材 → 单元 → 课时」这一条路，大学概念和技能类内容连入库都入不了。但它们完全可以用同一套认知质模型——因为认知质描述的是「人脑里的认知动作」，不绑定教材体系。

**为什么选现在这个时间点**：394 条数据是一个微小的时间窗口。数据量小到可以一键迁移，大到已经暴露了真实数据结构的所有坑——文件不一致、图谱绑定单版本、质量门禁硬编码路径。现在不修，以后修的成本不是加法是乘法。

## 五、自我进化路线图

本系统的进化分五个独立维度，每个维度的阶段数、触发条件和终局状态各不相同。

### 维度一：KP → 认知质消解（5 阶段）

这是知识点管理系统最深层的进化——从「管理 KP」到「理解认知维度」。

| 阶段 | 名称 | 触发条件 | 系统能力 | 人工角色 |
|------|------|----------|----------|----------|
| 1 | 静态 KP | 当前状态 | KP 是人类定义的固定 JSON 列表 | 人定义一切 |
| 2 | 带观察层的 KP | Phase 2 启动（有 ≥ 100 个学生的 ≥ 3 个月行为数据） | 每个 KP 下累积 `observations`：实际掌握时间、错误率分布、同级别 KP 对比数据。系统输出差异报告，但不修改 KP | 人审核报告，手动决定是否调整 |
| 3 | 自动发现候选认知质 | 连续 6 个月的数据显示某个 KP 下存在 ≥ 2 个稳定的、互不重叠的错因集群（聚类稳定性指标 > 0.7，具体阈值 Phase 1 数据积累后校准） | 系统自动在 `cognitive_primitives` 数组中写入候选认知质（含暂定名称、支撑数据摘要、置信度） | 人审核后确认或驳回 |
| 4 | 认知质驱动的诊断 | ≥ 60% 的 KP 拥有 ≥ 1 个已确认的认知质，且认知质级别的诊断准确率不低于 KP 级别的诊断准确率 | 诊断引擎切换到认知质为最小诊断单元。KP 降级为「容器」——仅用于教学进度管理和课标对齐 | 人监督切换过程，处理边界异常 |
| 5 | 自我生长 | 系统稳定运行于阶段 4 后自动进入 | 系统自己发现新的认知质、合并冗余的认知质、弃用不再有效的认知质。KP 概念在引擎内部完全消解 | 人仅处理系统无法自动决策的边缘案例 |

### 维度二：自拆合 KP（4 阶段）

系统自主判断 KP 该拆成多个还是多个合并为一个。

| 阶段 | 触发条件 | 系统能力 |
|------|----------|----------|
| 1（当前） | — | 所有拆合由人手工完成 |
| 2 | 与维度一的阶段 2 同步触发（有观察层数据后） | 系统基于 `observations` 中的错因聚类和掌握曲线同步性，输出拆合建议报告。**不自动执行。** |
| 3 | 连续 3 次人工审核中 ≥ 80% 被确认为正确，且连续两次审核中零严重错误（建议把完全不同类型的 KP 拆合） | 系统自动执行低风险拆合（合并掌握曲线高度同步的 KP）。高风险拆合（涉及 C 级以上 KP 的拆分）仍需人工审核 |
| 4 | 阶段 3 稳定运行 ≥ 6 个月，自动拆合的准确率 ≥ 95% | 全自动拆合。人工只看系统标记为「不确定」的案例 |

### 维度三：自适应分级（3 阶段）

系统基于实际学习数据，为每个 KP 增加「观察难度」标记。

| 阶段 | 触发条件 | 系统能力 |
|------|----------|----------|
| 1（当前） | — | `curriculumLevel` 由人工参照新课标标注，静态不变 |
| 2 | 与维度一的阶段 2 同步触发 | 系统为每个 KP 计算 `observedDifficulty`（基于平均掌握时间、错误率、前置依赖强度），与该 KP 的课标级别对比。偏差超过阈值（如一个 C 级 KP 的平均掌握时间低于所有 B 级 KP 的中位数）时，系统标记建议复核 |
| 3 | 阶段 2 稳定运行后，人工确认 observedDifficulty 计算方法可靠（与资深教研员的直觉判断一致性 > 80%） | `observedDifficulty` 自动更新，供策略引擎和课堂生成引擎做难度调节参考。`curriculumLevel` 始终不变——课标级别是法规锚点，observations 是教学信号 |

### 维度四：隐性依赖发现（4 阶段）

系统自己发现人未能察觉的知识点之间的依赖关系。

| 阶段 | 触发条件 | 系统能力 |
|------|----------|----------|
| 1（当前） | — | 所有依赖由人工用四条硬依赖标准逐对判定 |
| 2 | 有 ≥ 50 个学生 × ≥ 30 个 KP 的完整学习轨迹数据 | 系统分析「掌握 A 之后 B 的掌握速度变化」和「未掌握 A 但尝试 B 的学生失败率」。发现统计显著（p < 0.01）但未被定义为依赖的关系时，标记为「候选隐性依赖」 |
| 3 | 连续 5 条候选依赖被人工确认后，≥ 3 条在教学实验中验证有效（干预组掌握速度显著优于对照组） | 系统自动将中置信度的隐性依赖加入诊断依赖层（标记 `source: auto_discovered`，区别于人工依赖）。高置信度依赖直接加入学习路径层 |
| 4 | 阶段 3 稳定运行 ≥ 12 个月 | 系统自己发现跨学科的能力传导规律（如「几何空间推理 → 代数符号操作」），并记录为跨域依赖。这类依赖不强制进入学习路径，但被诊断引擎用于更精准的溯源 |

### 维度五：教材变化感知（3 阶段）

系统自动检测教材版本变化，提供更新建议。

| 阶段 | 触发条件 | 系统能力 |
|------|----------|----------|
| 1（当前） | — | 新教材导入完全依赖人工 |
| 2 | 现行教材的 KP 标注完成度 ≥ 95%（所有 KP 有 `textbookPage` 和 `textbookPageSource`） | 新教材 PDF 导入后，系统自动对比：知识点覆盖变化、章节顺序变化、例题变化、教学目标文字变化。输出差异报告。不自动修改 KP |
| 3 | 连续 2 次人工审核差异报告，≥ 90% 的系统建议被直接采纳 | 系统自动处理低风险变化（章节顺序调整、非核心 KP 的删除标记）。高风险变化（新增 KP 的分级、合并后 KP 的重新标注）生成建议供人工确认 |

`cognitive_primitives` 的填充责任同样在诊断引擎——诊断引擎在每次诊断完成后运行旁路分析，对同一 KP 的错因做聚类。如果连续 6 个月的聚类稳定性超过阈值，写入 `cognitive_primitives` 候选。

## 七、与其他子系统的接口约定

| 子系统 | 关系 |
|--------|------|
| S02 知识图谱系统 | 知识点管理系统提供 KP 列表和图谱结构。图谱独立维护自己的边和双向一致性验证。两者通过 `knowledgeCode` 关联。图谱系统需通过 `subjectCode`/`stage`/`contentType` 过滤关联域——数学 KP 的依赖不连接到语文 KP |
| S03 错因体系 | 错因绑定以 KP 为锚点（`error_kp_bindings.json`）。KP 拆分合并后，绑定须同步更新。错因编码通过 `subjectCode` 隔离——数学错因 `decimal_point_error` 与语文错因 `word_confusion` 不在同一池中 |
| S04 策略包体系 | 学科专属策略包的适用知识点领域定义依赖 KP。KP 变化后，策略包的知识点领域声明须同步复核。策略包通过 `subjectCode` + `domainTags` 声明适用范围 |
| S07 诊断引擎 | 诊断以 KP 为最小单元运行。维度一阶段 4 切换到认知质后，诊断引擎的最小单元随之切换。诊断引擎负责填充 `cognitive_primitives`。诊断引擎读取 `contentType` 以选择诊断策略——教材 KP 用错因匹配诊断，技能类用项目完成度诊断 |
| S08 策略引擎 | 策略引擎读取 KP 的 `curriculumLevel` 和 `observedDifficulty`（阶段 2 后）做难度调节。通过 `levelingFramework` 区分不同框架的级别含义 |
| S09 个性化课堂生成引擎 | 课堂生成以 KP 为内容边界。KP 的 `teachingGoal` 和 `keyPoints` 是课堂骨架的关键输入。课堂生成引擎通过 `contentType` 选择骨架模板——教材 KP 用教材骨架，技能类用项目骨架，专题类用探究骨架 |
| S13 教材大纲来源规则 | 知识点管理系统的教材来源逻辑引用 S13 的提取流程和优先级规则。仅对 `contentType=textbook_kp` 的内容生效 |
| S14 知识点分级规则 | 知识点管理系统的分级逻辑引用 S14 的新课标对齐规则和 levelScore 计算规则。仅对 `levelingFramework=curriculum_2022_math` 的内容生效。其他框架对应各自的分级规则 |
| S00 项目总览 | 通过 S00 索引定位本文档和旧版归档文档的对应关系 |

| 预留编号 | 系统 | 触发场景 |
|----------|------|----------|
| S15 | 跨版本教材映射系统 | 当存在 ≥ 2 个 `textbookVersion` 时，维护不同版本间 KP 的复用/隐藏/年级映射关系。当前 `cross_version_mapping.json` 为其 Phase 1 原型 |
