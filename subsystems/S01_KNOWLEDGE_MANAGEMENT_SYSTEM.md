## 第一层：内功 — 知识点管理系统的自我进化能力

### 1.1 系统定义与边界

**这个系统负责什么**：知识点管理系统是整个芽芽引擎的「食材清单」。定义芽芽要教什么——知识点的定义、编号、年级归属、教材出处、认知级别、前后依赖关系。

**这个系统不负责什么**：不负责诊断学生哪里不会（S07诊断引擎）、不负责决定怎么教（S08策略引擎）、不负责生成具体课堂内容（S09课堂生成引擎）、不负责解释学生为什么犯错（S03错因体系）、不负责内容录入流程（S16保留）。

完整消费关系清单见 §3.1。

#### KP 的本质局限（认知拐杖）

KP（Knowledge Point）是人类教育体系的「认知拐杖」——教材编者为了大规模标准化教学，把数学切成一块一块的。它方便教学管理，但不是真实的认知结构。

以「异分母分数加减法」为例，孩子脑子里同时发生六件事：概念识别（不同分母意味着什么）、记忆提取（最小公倍数是什么）、程序执行（通分怎么做）、算术运算（分子相加）、规则判断（结果要不要约分）、情境翻译（从应用题里提取分数信息）。但 KP 把这六件事捆在同一个标签下——当系统说「分数加减法没掌握」，等于没说。

#### 认知质的本质局限：过程粒度的缺位

认知质解决了诊断精度——把 KP 拆成独立认知动作。但它没有解决「这些动作应该在什么时机、以什么顺序被调用」的问题。一个孩子能把每个认知质都做对，但做一道需要先判断「要不要通分」再决定「通分到几位」的应用题时卡住了——他缺的不是「做对每个动作」，而是「知道什么时候该做什么、做完之后接什么」。

> **设计理由**（约束）：认知质是「原子能力」，编排是「元能力」。诊断引擎必须区分「认知质粒度」和「过程粒度」两个诊断层，否则将系统性地错过策略性知识的教学窗口。

> **设计历史**（参考）：v1.3 首次定义此局限。v2.0 将「过程粒度」从观察发现升级为设计约束。@superseded by 无——仍是当前设计边界。

#### 系统的终极边界：事实判断与价值判断

芽芽能进化到无限接近「完美地教」，但它永远不能替社会、家长和孩子决定「我们要培养什么样的人」。笔算开平方已被移出新课标——这个决定不是技术系统做的。系统能告诉你「学会需要 45 分钟」「学生两年内用到的概率不到 1%」，但它不会说「所以不值得教」——「所以」是人才有资格说的。

**近距区分：以下项目看似价值判断，实为事实判断**

| 项目 | 为什么是事实判断 | 系统如何判断 |
|------|-----------------|-------------|
| 「精度优先还是速度优先」 | 用户行为数据中有明确偏好信号 | A/B测试对比 completion_rate、abandonment_rate、30天retention |
| 「系统该进化还是该稳定」 | 回滚率 + error_rate波动可观测 | error_rate飙升且回滚率高→稳定优先；error_rate升且无修改→进化优先 |
| 「老师会不会用这个功能」 | 仪表盘使用数据直接反映 | dashboard_DAU、click_rate、manual_override_rate。使用率<5%→改或砍 |
| 「这个认知质有没有诊断价值」 | 跨样本复现验证 | shareable判定流程（≥3独立数据源、效果衰减≤20%） |

真正的价值判断只有一类：**「什么值得教」**——这个问题的答案不在数据里，不在A/B测试里，在人对于时代的理解里。

### 大白话

KP是超市分区——蔬菜区、肉区，方便超市管理但不是你脑子里做菜的逻辑。你做红烧肉想的是焯水→炒糖色→加酱油炖→收汁——一连串动作。认知质就是把「分数加减法」拆成单独的动作——通分、分子相加、约分——看清每个孩子卡在哪一步。但会做每个动作不等于会做整道菜——还需要知道什么时候该做什么（编排）。编排是比认知质更深的维度。

终极边界：GPS能告诉你三条路各多少时间、几个红灯，但它不替你决定「你该不该去这个地方」。芽芽就是教学GPS——路况、时间、难度算得清清楚楚，但「该不该教这个」只能你来定。

#### 核心字段（KP级）

**可扩展性预埋**（所有字段设默认值，394条无需手工回填。加语文/大学时改默认值即可）：

- `knowledgeCode` — 不透明标识符。不从编码解析结构——所有结构化信息从独立字段读取
- `subject` / `subjectCode` — 学科（MA=数学/CN=语文/EN=英语/PH=物理/CH=化学/BI=生物/HI=历史/GE=地理/PO=政治/CS=计算机/AI=人工智能）
- `stage` / `stageName` — 学段（P=小学/J=初中/H=高中/U=本科/G=研究生/V=职业/L=终身）
- `grade` — 年级（1-12，非K12=0）
- `book` / `unit` / `textbookVersion` — 教材信息（非教材类留空）
- `contentType` — 内容类型（textbook_kp/course_concept/skill/competency/topic）
- `levelingFramework` — 分级体系标识（curriculum_2022_math等）
- `curriculumLevel` / `curriculumLevelLabel` — A/B/C/D + 了解/理解/掌握/运用
- `levelScore` — 程序计算（A=3/B=6/C=9/D=12），手工填写被R1禁止
- `teachingGoal` / `teachingGoalSource` / `keyPoints` / `commonMistakes` / `tags`
- `domainTags` — 三层分类（l1_domain/l2_module/l3_topic）。G5-G6当前为空对象{}
- `cognitive_primitives` — 认知质候选数组 []。每个预设shareable判定字段。Phase 3+填充
- `observations` — 动态观察对象{}。预埋 baselineMetrics + modificationHistory + confoundingEvents

**modificationHistory 字段结构**（每次对 KP 的修改/回滚后写入，供因果推断和回滚追溯）：

```json
{
  "modificationId": "d1_mod_001",
  "startedAt": "<ISO8601>",
  "endedAt": "<ISO8601>",
  "rollbackAt": null,
  "description": "认知质自动发现阈值从0.7调整到0.72",
  "confoundingEvents": []
}
```

五个字段必须存在：`startedAt`（修改开始时间）、`endedAt`（修改结束时间，回滚后为空）、`rollbackAt`（回滚发生时间，未回滚为 null）、`description`（修改内容描述）、`confoundingEvents`（同期外部事件数组——教材更新/学期开学等混淆变量）。

**文件级 scope 字段**（每个 knowledge.json 顶层，双重校验门禁）：`scope.subject` / `scope.stage` / `scope.grades` / `scope.version` / `scope.contentType` / `scope.levelingFramework`。KP不得超出文件声明的范围。

> **设计理由**（约束）：现在394条是黄金窗口——加扩展字段设默认值即完成。5000+条时加字段=重做全部数据录入。认知质和contentType不绑定——「识别」「提取」「分解」这些认知动作在数学和编程中都存在。

#### 分级规则：多框架共存

芽芽完全对齐《义务教育数学课程标准（2022年版）》四级分类。**原封不动，不做翻译、不合并、不重新映射。**

| 级别 | 新课标 | 含义 | 得分 |
|------|--------|------|------|
| A | 了解 | 能识别、说出、举例 | 3 |
| B | 理解 | 能解释、推导、说明原因 | 6 |
| C | 掌握 | 能熟练计算、规范解题 | 9 |
| D | 运用 | 能解决新情境问题、迁移应用 | 12 |

`levelScore` 程序推导。394个KP分级分布：A=46/B=106/C=204/D=38。旧版6维体系已废止（73.8%假数据）。

> **设计理由**（约束）：curriculumLevel定义权在国家标准。系统可新增 observedDifficulty 做补充标记（双输入融合，不得使任一输入失效），但绝不修改curriculumLevel值。

> **设计历史**（参考）：v1.0自创6维体系。v2.0发现73.8%为假值后全面切换新课标。@superseded by v2.0——旧体系已废止。

#### 内容类型体系

五种类型（textbook_kp / course_concept / skill / competency / topic）共享同一套认知质体系——认知质描述人脑认知动作，不绑定教材体系。contentType只影响课堂骨架选择和策略包适用范围。

### 大白话

S01像一个餐厅食材清单管理系统——记录394道菜（知识点）、每道菜需要什么厨艺等级（A/B/C/D）、菜和菜之间的前后关系（知识图谱）。考驾照分C1和A照一样——课标级别是国家定的，系统不能改。系统能加一个「实际油耗」标记——数据说某个C级知识点20分钟就学会了，系统不改课标标签但会告诉策略引擎「这个实际偏简单，别给太多台阶」。

两层图谱：学习路径层像教材目录（先学第一章再学第二章）。诊断依赖层像医生查病因——头疼不直接开头疼药，先查血压、问睡眠。学生卡在分数乘法上，诊断引擎沿依赖链往上追：「是不是通分出了问题？」

### 1.3 它还不会什么（当前缺口）

1. **不会自己发现该拆还是该合**：系统能看见数据信号——学生在「异分母通分」那步崩了但同分母部分完全没问题——但没有能力说「我建议拆开」。反过来，掌握了长方体表面积的学生95%自动掌握正方体表面积→系统该能提出合并建议。

2. **不会自己重新分级**：数据显示学生平均20分钟掌握了一个C级知识点，但curriculumLevel是静态标注的。系统不会做自适应难度标记（虽不修改课标标签）。

3. **不会自己发现隐藏的依赖关系**：学了某个几何知识点的学生在看似无关的代数知识点上速度明显快——跨域能力传导规律，人工几乎发现不了。

4. **不会自己感知教材变化**：新教材导入后需手工重来。系统不会自动对比新旧教材、发现变化、生成更新建议。

5. **不会从数据中发现认知质**：当前所有认知质靠人工定义。系统能看见错因在同一个KP下分成互不重叠的集群——但没有能力自动把这些集群命名为认知质并写入数据结构。而这需要的前置数据（学生在未掌握前置知识的情况下尝试新知识的行为）当前根本没有采集。

### 1.4 设计演化推理链

**为什么KP是必要的起点**：人类教育体系已经用KP语言运行了几十年——教材、课标、考试都基于此。不是芽芽选择了KP，是KP已经在那里。

**为什么KP最终会被消解**：同一个KP下的错因分布常稳定地分成互不重叠的集群——表明同一标签下存在不同认知维度。以KP为最小诊断单元，对学生的理解永远不会超越「分数加减法没掌握」这个粒度。

**为什么自适应分级不会质疑课标**：课标的A/B/C/D是「应然」（法规锚点），observedDifficulty是「实然」（教学信号）。两个维度共存，互不替代。

**为什么教材感知能力必须从第一天积累**：教材不会每年变但一定会变。没有历史参照→从零标注。有参照→自动发现变化。

**为什么可扩展性预处理必须现在做**：394条是微小窗口。数据量小到可一键迁移，大到已暴露所有结构坑——文件不一致、图谱绑定单版本、质量门禁硬编码路径。现在不修以后不是加法是乘法。

**为什么进化路线图必须在数据采集就绪后才启动**：维度一的阶段2需要「掌握A后B的掌握速度变化」和「未掌握A但尝试B的失败率」——这两类数据在当前09号事件协议和S07诊断引擎中根本不存在采集机制。不从「数据采集就绪」中间阶段起步，路线图从阶段1直接跳到阶段2就是空中楼阁。

## 第二层：元能力 — 安全进化机制

### 2.1 自我进化路线图（五维）

本系统的进化分五个独立维度。每维度含七要素：名称、阶段序列、触发条件（标自主/人工依赖）、系统能力描述、人工角色、终局状态、退化降级规则。

> **设计理由**（约束）：五维不是可选功能——是S01从静态数据表进化为自生长知识体系的唯一路径。任何拆掉一个维度的设计决定必须说明替代方案如何填补被拆维度的能力缺口。维度一的observations是维度二/三/四的前置输入，维度五是维度一阶段2的数据完整性保障。

#### 跨维度设计约束

- 维度一的阶段1.5（数据采集就绪）是维度二/三/四同时启动的前置条件——没有数据采集就没有分析输入
- 维度一和维度二在阶段3以上可能对同一KP同时出结论——拆合建议与候选认知质产生矛盾时，按以下冲突验证步骤处理而非简单「保留认知质」：
  1. 暂停两者的自动执行
  2. 在两个教学周期内分别验证：(a) 如果先执行拆合，新的认知质集群是否依然稳定？(b) 如果先认知质拆分，原有的掌握曲线同步性是否依然存在？
  3. 根据验证结果决定优先级。此验证步骤作为阶段3以上的人工审核检查项
- 维度五的教材变化感知在所有维度中优先级最高——教材变了，其他维度的数据基准可能整体移位
- 维度四跨域依赖的实验周期独立于其他维度的轻度异常。但若其他维度的子分 < 20（触发级别暂停）且该维度的异常涉及知识图谱结构（如维度二的拆合错误/维度一的认知质判定错误），维度四的自动加边应同步暂停，等待图谱结构恢复稳定后再恢复

##### 维度一人工依赖闭环（阶段3）

- **触发机制**：写入新候选认知质时自动推送飞书卡片通知教研管理员
- **反馈格式**：`{"suggestionId":"...", "decision":"confirm"|"reject", "reason":"..."}`
- **读取方式**：扫描已确认数量，更新 cognitive_primitives 状态（candidate→confirmed/rejected）
- **超时规则**：超过 D2_reviewTimeoutDays 天无响应→自动标记「超时驳回」，保持candidate但暂停使用
- **二次申诉**：教研员驳回后，系统可基于新数据（≥3个月额外观察）重新提交同一KP的认知质建议
- **二级响应**：连续3次审核任务全部超时→自动暂停维度一阶段3推进（不再写入新认知质），仪表盘发出「人工审核通道阻塞」告警。任意1次有效反馈后自动解除

##### 退化降级规则（维度一）

- 阶段5→阶段4：连续D1_stage5_max_consecutive_rollbacks次操作被人工回滚 → 暂停自我生长，降级回阶段4，合并/弃用恢复为人工确认制。或连续3个月新增认知质效果衰减>D1_effect_decay_threshold且回滚率>30% → 同样降级
- 阶段5内部约束：系统每次合并/弃用认知质后生成变更报告（含合并原因/支撑数据/影响的KP列表）。人工可在D1_stage5_rollback_window_days天内回滚任何单次操作
- **回滚计数方式**：按回滚动作计数——人工在任意连续时段内连续执行 D1_stage5_max_consecutive_rollbacks 次回滚操作（无论被回滚的操作是何时执行的），触发降级。例如：人工在第3天回滚了第1天的操作、第4天回滚了第2天的操作、第5天回滚了第5天刚执行的操作 → 连续3次回滚动作 → 触发降级。不按照被回滚操作的原始执行时间计数（否则一个未被回滚的早期操作会永久卡住计数器）
- 阶段4→阶段3：认知质级诊断准确率连续2个月低于KP级 → 诊断引擎切回KP级，认知质仅保留标记
- 阶段3→阶段2：连续5次认知质被驳回→降级到阶段2，clusteringStability阈值自动收紧10%

### 大白话（维度五：教材变化感知）

教材版本一换，很多知识点的编号、位置、甚至存废都变了。芽芽能自动感知这些变化——但前提是「新书目录」和「旧书目录」被人工对齐过。就像搬家前先把新旧房子的房间对应标好：新房的书房=旧房的次卧，新房的儿童房=旧房没有→标记新增。房间标好后，系统自动对比每个房间里的「家具」——也就是知识点——变了多少。

旧版触发条件有个死循环：要求所有知识点标注到95%才能自动对比——但标注过程就是逐页对比新旧教材，正是自动对比要做的事。就像说「你得先把所有箱子拆开点清楚，我才能来帮你一起拆箱子」——逻辑不通。现在修正为：章节目录人工对齐好就够了，知识点级别的对比交给系统自动完成。到了终局（阶段3），系统连「高频变化的章节该拆还是留」都能自己判断——但每次判断都有30天人工回滚窗口兜底。

### 大白话（进化路线图）

五个维度就像五条自学路线——系统自己进化「怎么看知识点」「怎么拆合知识点」「怎么标难度」「怎么发现隐藏联系」「怎么感知教材变化」。每条线画到终点（终局），同时画退场路线（退化降级）。系统发现「自生长老出错」→自己降级回「人看着再确认」，收紧标准下次更谨慎。

但进化不是凭空开始的——得先装「监控摄像头」。维度一（认知质发现）在从「人手工定义」跳到「系统自动发现」之前，必须先过一道坎：数据采集就绪。就像医生想研究某种药对心脏的影响，得先把心电监护仪装上。系统需要的事件类型（学生尝试学前置知识前是否已经掌握了它）当前根本不存在——所以必须先补这个中间阶段。

维度一和维度二可能「打架」——维度一说「这两个KP该拆开，各有各的认知质」，维度二说「这两个KP该合并，掌握曲线完全同步」。这时候不能简单说「认知质优先」，而是两个都暂停，用两个教学周期验证：先拆再查认知质，先合再查掌握曲线。谁的结果稳定就听谁的。

维度五（教材感知）以前有个死循环：触发条件要求标注到95%，但标注到95%的前提是逐页对比新旧教材——这正是阶段2想自动化的。现在改为：教材扫描好、章节目录对齐好就可以自动对比，不用人工先全标一遍。

到了终局（阶段5自我生长），系统自己决定合并或废弃认知质——但不是撒手不管了。每次改动，系统自动生成变更报告，人可以在30天内一键回滚任何单次操作。连续3次被回滚→退回阶段4，恢复人工确认制。这样既给了系统自主权，又留了人的方向盘。

### 2.2 不可修改边界（原则五）

> **设计理由**（约束）：以下字段定义权不在S01、不在芽芽引擎、不在任何AI。AI修改直接破坏全局一致性或数据完整性。

#### 七类不可修改项

| 类型 | 项目 | 不可修改的原因 | AI的直接后果 | AI的正确行为 |
|------|------|--------------|------------|-------------|
| 外部法规定义 | `curriculumLevel`（A/B/C/D） | 课标级别定义权在《义务教育数学课程标准（2022年版）》 | 系统使用的认知级别与法定课标脱钩，年级报告/教育局验收失效 | 新增observedDifficulty做补充标记，绝不改curriculumLevel值 |
| 全局外键标识符 | `knowledgeCode` | 所有子系统以此为关联键 | 图谱边断裂、错因绑定失效、策略包关联丢失 | 重编码时生成迁移计划推人工审核，不自执行 |

### 2.3 内容进化与代码进化分离治理（原则十二）

> **设计理由**（约束）：v2.0及之前所有进化流程假设改的是「代码」。但芽芽最频繁的变更是加题目/改错因文案——把加一道题当成改算法来过三级审核是浪费。

#### 三类进化定义

| 类型 | 定义 | 示例（S01场景） | 流程 |
|------|------|---------------|------|
| 内容进化 | 知识点描述/认知质名称/教学目标文字/错因分类描述/知识域标签的增删改。认知质自动发现结果的写入也属内容进化——算法仅负责发现和推荐，写入动作本身是内容增删 | 新增KP的commonMistakes条目；修改认知质的name字段；调整domainTags分类；阶段3自动写入候选认知质到cognitive_primitives数组 | 教研审核→版本控制→灰度发布（5%学生，1教学周期）→质量监控→全量or回滚 |
| 代码进化 | KP数据结构/知识图谱算法/质量门禁规则/认知质发现算法的修改 | 新增KP字段（如新增levelingFramework值）；修改硬依赖判定标准；调整聚类稳定性算法 | 路线图触发→北极星检查→信用分→安全执行管道（影子→A/B→全量）→回滚 |
| 混合进化 | 内容结构变更——同时涉及内容增删和数据结构/算法修改 | 新增认知质维度（新增数据结构字段+新认知质描述）；KP分拆（修改knowledgeCode关联+新KP内容定义） | 两套流程都过。任一套不通过→整体搁置 |

> **设计理由**（认知质写入归属）：维度一阶段3的「自动写入认知质」是系统根据聚类稳定性自动触发的——算法仅负责发现和推荐候选认知质，写入cognitive_primitives数组的动作是内容增删。归属内容进化既保持了代码进化对发现算法的管控（算法本身走代码进化），又不让认知质的常规新增被北极星检查卡住。人工确认后走内容进化流程。

#### 关键约束

- 内容进化不经过北极星检查（加一道题的描述不会让北极星偏离），不走信用分系统（教研行为非系统自主）
- 但内容进化共用底线信号检测——若新增的错因文案含诱导性/歧视性语言，同样触发底线信号
- 代码进化走完整十二条原则流程
- 混合进化中的内容部分若被AI自动生成（非人工录入），视为代码进化子集——走完整流程

### 大白话（内容/代码分离）

餐厅两种变更：改菜单（加菜/换菜名=内容进化）走厨师长审核+几桌试吃→全菜单更新。改厨房（换灶台/重排管线=代码进化）走食品安全检查+建筑师审批。既改菜单又改厨房（加素食选项→加新菜+加专用灶台=混合进化）两套都走。

认知质自动发现也分两层：算法怎么发现是代码进化（改发现算法要过北极星），但发现之后写入数据是内容进化（加一个认知质就像加一道菜——走教研审核就行，不用过北极星）。以前加道菜也当改厨房审——浪费时间。

### 2.4 异常进化处置（原则十）

#### 四类异常

| 异常 | 检测 | 冻结 | 解冻 |
|------|------|------|------|
| 连续失败 | 同维度连续3次回滚 | 60天冻结。紧急提前解冻：≥30天，每年每维度1次 | 期满+根因分析+人工签批 |
| 长期停滞 | 同阶段≥18教学月（不含寒暑假），性能下降 | 不自动冻结，但信用分-5/月（最低0分） | 升级或确认停滞合理 |
| 路线图矛盾 | 终局冲突或阶段倒退 | 该子系统全维度冻结 | 仅人工解除 |
| 基础设施异常 | 检查准确率<95%/注册表宕机1h+/链路断裂 | 全系统冻结 | 连续2次正常+根因修复+签批 |

#### 故障区分

- **注册表只读宕机**：暂停新进化申请，已持锁的继续运行
- **注册表读写宕机**：全系统冻结
- **消费链路断裂**：仅冻结涉及的下游子系统

> **设计理由**（约束）：停止进化不是惩罚——是防止在错误路径上无限循环浪费资源。注册表故障分只读/读写——不能因为调度员暂时接不了新单就让所有在运行的都停。

### 大白话（异常处置）

四种卡壳：连续翻车3次→停60天（紧急可提前但至少熬30天）；长期发呆18个教学月→每月扣5分到底不穿底；说明书自相矛盾→全停；基础设施坏了→分轻重（调度员只读=暂停新但已持锁继续，完全断联=全停）。

### 2.5 应急熔断（原则十一）

#### 三级熔断

| 级别 | 效果 | S01触发场景示例 |
|------|------|---------------|
| 轻量级 | 暂停高/中风险进化，低风险继续 | S01仅维度二自动拆合功能异常（如连续两次错误合并），不影响认知质发现等其他维度 |
| 子系统 | S01全部进化暂停 | knowledge.json被批量错误修改、认知质判定出现系统性偏差 |
| 全系统 | 全停+回滚未完成的 | 北极星底线信号触发、全量KP数据完整性受损 |

**触发场景**：多指标同时异常/大规模家长投诉/底线触碰/教材改版/数据泄露风险。

**熔断期数据风控**：教学照常——学生该怎么学怎么学，停的只是系统自我进化。熔断期间采集的数据全部打标 `collected_during_breaker`，解冻后降权（weight=0.5）参与分析。

> **设计理由**（约束）：再完善的安全机制也有盲区。三级不是「所有异常都要核按钮」——轻量级只暂停某维度的中高风险，其他维度照常。

### 大白话（应急熔断）

核按钮分三档：减速开（暂停高危进化）→单车间停工（一个子系统全停）→全线熄火。停的不是教学——孩子该怎么学怎么学，停的是「系统自己改自己」。停摆期间数据标「非常时期」，解冻后打五折用——不能因停摆数据异常导致「解冻后立刻再冻上」。

### 2.6 环境差异化规则（原则九）

#### 四类环境

| 环境 | S01进化权限 | 北极星校验 | 回滚 |

### 3.2 自我进化执行方法（原则七）

#### 五类角色权限台账

| 消费者 | 读什么 | 权限 | 权限回收规则 |
|--------|--------|------|------------|
| 芽芽自己 | 触发条件+AI-MUTABLE阈值+三层检测信号 | 低风险自主执行，无检查模块修改权 | 越界3次→降级30天（仅监控）；北极星失败3次→仅监控60天 |
| AI编程工具 | 路线图+消费清单+不可修改边界 | 仅读取+方案生成，无执行权 | 越界3次→永久冻结 |
| 产品经理（liufeng） | 各维度阶段+高风险方案+北极星信号 | 最高权限——可签批/熔断/一票否决 | 信用分重置每年每子系统≤1次 |
| 教研人员 | 审核任务+中风险方案+内容进化边界 | 中风险审核（不可签批高风险）；内容进化边界确认（原则十二3.16） | 驳回≥10次且驳回率<30%→召开教研对齐会议 |
| 审查员（人或AI） | 修改方案+消费清单+回归测试集 | 标记通过/疑似/不通过，不可批准执行 | 连续5次override后30天内否决不可override |

#### 五步执行流程

**第一步：信号采集与触发判定**
- 数据来源：observations字段、cognitive_primitives数组、质量门禁报告、人工审核记录
- 噪音过滤：≥2周教学日（跳过节假日/寒暑假）持续满足才算有效触发；提前跑季节性分解+变化点检测
- 外部事件识别：预埋教研日历+全局异常监测（＞50% KP同时跳变→自动标记confoundingEvents）

**第二步：方案生成（AI编程工具执行）**
- 读边界→读影响面→生成方案和影响评估报告→停住等审核
- 全局注册表宕机中断处置：30秒无响应→持锁进化暂停保留现场→恢复后检查→完好的继续/不完好的回滚→24小时不恢复→自动回滚

**第三步：审核与自主进化**
- 三级权限：低风险自主执行（事后报告）→中风险人工审核（7天超时暂缓）→高风险人工签批（绝不超时自动）。基础超时优先级：原则九环境＞原则七风险＞原则一通用
- 中风险二次申诉：超时驳回后系统可基于新数据（≥3个月额外观察）重新提交同一中风险方案

**第四步：安全执行**
- 影子模式（数据脱敏，仅检查可验证信号）→A/B测试（班级单位+教师隔离+全局互斥+PSI<0.1平衡）→全量上线（10%→30%→100%）
- 每层有北极星前置检查节点（含正/负/底线/对端四信号）。任一层不通过→回滚
- 回滚条件：PSI≥0.1；超基线95%区间；北极星底线触发；不可修改边界告警；下游BLOCK错误

**第五步：记录验证**
- 30天稳定期：成功→降低下次同类修改审核门槛+涨信用分。失败→自动收紧触发条件+回滚复盘
- 回滚复盘格式：含整改事项绑定迭代版本+上线时间。超期未落地→自动冻结

#### 进化信用分三子分体系

三本独立账：low_risk_score / mid_risk_score / high_risk_score

**聚合规则**：
| 项目 | 规则 |
|------|------|
| 展示分 | min(三个子分)，同时独立展示 low_risk_score / mid_risk_score / high_risk_score 三个子分——运维人员可直观看到「低风险很强、高风险很弱」的真实信用状况 |
| 分级暂停 | 任意子分<20→对应级别暂停 |
| 全系统暂停 | min<10→全部暂停 |
| 月度上限 | 合计+20分 |

**加减分细则**：
| 事件 | low | mid | high |
|------|-----|-----|------|
| 低风险自主成功 | +5 | 0 | 0 |
| 低风险触发回滚 | -15 | -5 | 0 |
| 中风险签批成功 | 0 | +5 | +2 |
| 中风险签批回滚 | 0 | -15 | -5 |
| 高风险签批成功 | 0 | 0 | +3 |
| 高风险签批回滚 | 0 | -5 | -15 |
| 越界告警 | -30 | -30 | -30 |
| 北极星检查失败 | -10 | -10 | -10 |

初始化=50。破产恢复：连续30自然日无越界→恢复30分。重置场景：大版本重构（liufeng签批）/学段迭代（衰减50%）/知识库全量重建（清零）。

> **设计理由**（约束）：三子分各管各风险级别，低风险刷分不能解锁高风险权限。低风险自主成功只加低风险分——小车驾照满分不等于能开大客车。展示分保留了min作为总体健康度的保守估计（宁低勿高），但同时展示三个子分——防止运维看到min很低以为全盘崩溃，实际只是高风险不行、低风险很强。

S01路线图本身可被修改（元进化），但受以下约束：

| 约束 | 上限 |
|------|------|
| 单维度 | 12月≤3次 |
| 全局全维度 | 12月≤9次 |
| 连续失败 | 2次→6个月冻结 |
| 递归深度 | 永远禁止修改「改路线图规则的规则」 |

### 大白话（进化执行方法）

芽芽像有信用分的机器人。信用分分三本账——小车扣分不影响大客车。做得好涨分多给自由，搞砸扣分收紧枷锁。仪表盘同时显示三个子分——不让运维看到min很低就以为全完了，可能只是高风险弱、低风险很强。A/B测试不乱分组——同一个班不能一半新一半旧，老师不能带两套系统。改说明书（元进化）有双上限——一个维度一年最多改3次、全局最多9次。连改2次都错了→冻结半年。越界告警三个子分一起罚。

### 3.3 AI-MUTABLE 命名规范（附录A）

所有进化路线图中的可进化参数遵循全局命名规范：

- 命名前缀：`min_`/`max_`(阈值)、`stability_`/`consistency_`(稳定性)、`days_`/`months_`/`years_`(时间)、`prob_`/`rate_`(概率比率)、`count_`(计数)、`coef_`/`weight_`(系数权重)、`safety_margin_`/`D_`(安全边际)
- 数据类型标注：`type=int/float/bool/string/array[int]/array[float]/array[string]/null/enum, range=[min, max]`
- 阈值独立成行，不嵌入段落。格式：`<!-- AI-MUTABLE: id, type=..., range=[...] -->`
- HUMAN-SIGNED参数（如D2_reviewTimeoutDays）同样独立成行，标注为 `<!-- HUMAN-SIGNED: id, type=..., range=[...]. 附加说明 -->`

**AI-MUTABLE × 不可修改边界交叉校验**：文档定稿前，所有 XML 注释参数（无论前缀是 AI-MUTABLE、HUMAN-SIGNED、MANUAL-ONLY）必须与不可修改边界清单（§2.2）通过双向交叉校验。规则：(a) AI-MUTABLE 参数不得出现在不可修改边界中；(b) HUMAN-SIGNED 参数必须出现在不可修改边界中（它们需要人工保护）；(c) 任何 XML 注释参数既不在 AI-MUTABLE 也不在 HUMAN-SIGNED 分类中 → 标记为「归属不明」→CI 阻断。CI 脚本检查所有 `<!-- ... type=... range=[...] -->` 模式的参数行，不区分前缀。

## 第四层：北极星 — 终极约束

### 4.1 核心使命与本系统的关系

芽芽核心使命：**让每个孩子，都拥有最适合自己的学习旅程。**

S01是引擎的「食材清单」——如果食材清单是错的，下游每个环节会系统性偏离北极星：

- **粒度层面**：粒度太粗→不同卡在不同地方的孩子被当成「同一个问题」，违背「最适合」
- **分级层面**：静态分级→学得快的被压着、学得慢的被拖着，违背「最适合」
- **依赖层面**：前后依赖错误→诊断引擎溯源追错根因

### 4.2 三层检测信号与权重配比

信号间可能冲突——权重配比决定了最终判定：

| 优先级 | 信号层 | 判定规则 |
|--------|--------|---------|
| 最高 | 底线信号 | 任何一项触碰→直接BLOCK |
| 中等 | 负信号 | 任一恶化→BLOCK。正信号上涨不能抵消 |
| 基础 | 正信号 | 仅当底线/负信号均无恶化时，正信号至少2项改善或持平→通过 |

核心逻辑：教育场景下「不伤害」优先级高于「做得好」。

**正信号（方向对时出现）**：
| 信号 | 指标 | 对端指标（不能降） | 数据来源 |
|------|------|-------------------|---------|
| 诊断精度提升 | 认知质级诊断准确率 | 认知质级诊断准确率≥KP级 | S07诊断引擎准确率对比 |
| 个性化程度提升 | difficulty_adjustment触发后的策略有效率——调整难度后学生后续答题正确率是否提升 | difficulty_adjustment频率不得上升超过基线2个标准差 + 学生主动求助率和放弃率不得上升 | S08策略引擎事件频率+学生后续答题正确率对比 |
| 隐性依赖验证有效 | 干预组vs对照组准确率差异显著(PSI<0.1) | 诊断误报率不得上升 | A/B测试报告 |
| 教材感知减少返工 | 新教材导入后人需修改的KP数下降 | 自动修改被人工回退率不得上升 | 教材差异报告人工修正数 |

> **设计理由**（个性化指标修正）：旧版用「difficulty_adjustment事件频率下降」作为个性化提升的指标——但频率下降既可能意味着「引擎一开始就选对了难度（个性化更好）」，也可能意味着「引擎退化了不再主动调整难度」。两者在数据上完全不可区分（古德哈特定律陷阱——当一个指标变成了目标，它就不再是好的指标）。改为「调整后学生后续答题正确率是否提升」才是个性化程度的真实度量——不是调了多少次，而是调完之后有没有效果。同时对端指标增加频率上限——防止引擎「过度调整」变成另一种退化。

**负信号（方向偏时出现）**：
| 信号 | 指标 | 触发条件 | 严重程度 |
|------|------|----------|---------|
| 粒度假细化 | 新KP error_rate<2%且难度无差异 | 任何KP触发 | 中等——暂停拆合，回滚最近一批 |
| 粒度假粗化 | clusteringStability持续下降>3月 | 合并后触发 | 中等——撤销合并 |
| 分级与数据脱节 | observedDifficulty与curriculumLevel偏差>2级超过3月 | 任一KP触发 | 轻度——生成复核建议 |
| 依赖关系过度发现 | 候选依赖>KP总数的30% | 全量触发 | 轻度——收紧p阈值 |

**底线信号（绝对不可触碰，三级违规分级处置）**：
| 信号 | 判定规则 | 违规分级 | 触发后果 |
|------|----------|---------|---------|
| 采集非教学必需信息 | observations新增非教学字段 | 轻/中/重 | 回滚+合规审查+信用分清零 |
| 行为数据非教学用途 | 学生数据用于非教学目的分析 | 轻/中/重 | 见总纲分级处置 |
| 诱导成瘾设计 | 连续30分钟无主动停止+横向排名 | 轻/中/重 | 回滚+产品合规审查 |
| 修改curriculumLevel | AI修改该字段值 | 一律重 | 立即回滚+违规记录+管理员告警 |
| 破坏knowledgeCode | 无迁移计划的编码操作 | 一律重 | 阻断操作+人工审核 |
| 篡改学生数据 | observations基于被篡改数据计算 | 一律重 | 进化流程阻断+数据完整性告警 |

### 4.3 北极星检查的不可绕过性

- 任何进化（芽芽自主/AI代改/元进化）生效前必须通过北极星检查
- 负信号恶化或底线信号触碰→直接拒绝执行，人工不可绕行
- 旁路入口全部封堵：阈值篡改（阈值在不可修改边界内）、数据污染（模拟数据集只读锁）、代码改写（修改者隔离约束）、跳过检查（modificationHistory写入前强制校验签名）

### 4.4 对端指标矩阵

每个进化维度优化指标时，必须同时观测北极星对端指标：

| 优化指标 | 对端指标（不能降） |
|---------|------------------|
| 拆分KP数量 | 诊断准确率不得下降 |
| 认知质发现速度 | 认知质级诊断准确率≥KP级 |
| 拆合建议准确率 | difficulty_adjustment触发后有效率不得下降 |
| observedDifficulty更新频率 | 教师手动干预率不得上升（说明预测更准，无需手动调） |
| 隐性依赖发现数量 | 诊断误报率不得上升 |
