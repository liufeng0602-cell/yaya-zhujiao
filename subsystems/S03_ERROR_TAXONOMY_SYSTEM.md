## 第一层：内功 — 错因体系的自我进化能力

### 1.1 系统定义与边界

**这个系统负责什么**：错因体系定义学生可能犯的所有错误类型。两层架构——L1通用错因（跨学科）11 类 + L2 数学专属错因 76 类，总计 87 种错因（11 L1 + 76 L2）。每个错因有 errorCode、中文描述、家长端中性解释（parentExplanation）。通过 error_kp_bindings.json 将错因绑定到知识点。

L1 第 11 类 `unknown_error` 为兜底错因——当诊断引擎无法将学生错误匹配到任何已知错因时使用。数据状态：L2 当前仅覆盖 G5-G6（scope: primary_math_g5_g6_pep, doc_maturity: draft, 数据完成度：76/76 类已定义但尚未通过正式验收）。

**这个系统不负责什么**：不负责诊断哪个学生犯了哪个错（S07诊断引擎）、不负责选择干预策略（S08策略引擎）、不负责向家长解释（S12家长摘要系统仅引用 parentExplanation）。

### 1.2 L1 通用错因 11 类

| # | 中文名 | errorCode | 典型表现 |
|---|--------|-----------|---------|
| 1 | 前置知识缺失 | prerequisite_gap | 需要用旧知识时完全想不起来 |
| 2 | 概念混淆 | concept_confusion | 把相似概念搞混（如周长/面积） |
| 3 | 题意理解偏差 | comprehension_bias | 应用题读错意思 |
| 4 | 关键信息遗漏 | info_omission | 做题时漏看题目条件 |
| 5 | 计算/操作失误 | calculation_error | 思路对但算错了 |
| 6 | 解题策略缺失 | strategy_missing | 面对题型不知道从哪下手 |
| 7 | 迁移失败 | transfer_failure | 标准题能做，换场景就不会 |
| 8 | 题型不熟 | question_type_unfamiliar | 对这类题的答题方式不熟 |
| 9 | 表达/书写/格式问题 | expression_format_error | 能做对但书写不规范 |
| 10 | 记忆提取失败 | memory_retrieval_failure | 公式/方法忘了 |
| 11 | 未知错因（兜底） | unknown_error | 不符合上述任何已知错因模式 |

L1 错因跨学科通用——语文、英语、物理等学科均可复用。L2 错因各学科独立创建。

> **设计理由**（约束）：错因分类是诊断引擎匹配的唯一语言。如果错因定义模糊——两个错因边界重叠——诊断引擎会在两个假设间摇摆。L1/L2 分层确保通用错因不重复定义，学科专属错因不遗漏。新增 unknown_error 兜底——诊断引擎匹配失败时不再发空字符串或 null，而是发明确的 unknown_error 信号，为阶段 2 的自动聚类分析提供数据来源。

> **unknown_error 即时行为**：S07 诊断引擎输出 errorCode='unknown_error' 时→S08 策略引擎使用兜底策略包（如 SP_G_C01 基础巩固包、SP_G_D01 舒缓讲解包等，由 S08 根据学生上下文选择）→同时该次诊断标记为 `classification=inconclusive` 供后续聚类分析。诊断后该学生对应当前 KP 进入「观察列表」，若同一 KP 累计 3 次 unknown_error 触发人工教研介入。

> **设计历史**（参考）：v1.0 沿用旧版 07 号的 10+76 分类。旧版在 2026-05-14 经过地狱审计修正了 parentSafeExplanation→parentExplanation 字段名。v1.2 新增 unknown_error 兜底错因——三方审计发现诊断引擎在无法匹配时没有约定信号格式，导致聚类分析无数据。@superseded by 无。

### 1.3 L2 数学专属错因 76 类

全量定义在 `acceptance-artifacts/error_taxonomy.json` 的 `l2_math_specific` 中，覆盖数与运算、图形与几何、统计与概率、综合与实践。数据范围：当前仅覆盖 primary_math_g5_g6_pep，doc_maturity: draft。示例：

- 分数单位理解错误（fraction_unit_misunderstanding）
- 通分规则错误（common_denominator_error）
- 小数点定位错误（decimal_point_error）
- 面积公式混淆（area_formula_confusion）
- 几何图形识别错误（shape_identification_error）

### 1.4 错因绑定

`acceptance-artifacts/error_kp_bindings.json`：1390 条绑定，394 个 KP 全覆盖。G5-G6 平均 3.2 绑定/KP，G1-G4 平均 3.8 绑定/KP（以上数据由 scripts/count_bindings.py 于 2026-05-18 统计自 error_kp_bindings.json——统计方法：逐 KP 计数 bindings 数组长度，按 G5-G6/G1-G4 分组取均值。已验证，脚本结果与文档数字一致）。A 级 KP 要求 ≥ 3 绑定（R7 铁规）。

> **G1-G4 绑定来源说明**：G1-G4 绑定主要由 L1 通用错因（通过默认生成规则——每 KP 的 prerequisite_gap + calculation_error，及按题型自动补充 comprehension_bias/info_omission/memory_retrieval_failure/strategy_missing）组成，辅以 G1-G4 适用的跨年级 L2 错因（如分数基础、面积/周长、单位换算、方向与位置等）。L2 错因当前仅覆盖 G5-G6 完整定义，G1-G4 的 L2 覆盖不完整——多数 G1-G4 KP 仅依赖 L1 通用错因。

绑定生成规则：每 KP 默认加 prerequisite_gap + calculation_error。应用题加 comprehension_bias + info_omission。公式记忆类加 memory_retrieval_failure。解题流程类加 strategy_missing。

> **已知限制**：10 个 L1 错因（#1-#10）不与特定 KP 做定制化绑定——它们通过默认生成规则（每 KP 默认加 prerequisite_gap + calculation_error 等，见上文绑定生成规则）以统一规则覆盖所有 KP，不存储为逐 KP 的独立绑定记录。在 error_kp_bindings.json 中，L1 错因通过生成脚本批量添加，而非逐 KP 手工维护。第 11 个 L1（unknown_error）同样无绑定，diagnosis 到 unknown_error 时由 S08 根据上下文选择兜底策略包（如 SP_G_C01/SP_G_D01 等）。

### 1.5 当前能力缺口

1. 不会自动发现新错因——76 个 L2 是人工定义的，系统无法从数据中自己发现新的错误模式
2. 不会自动调整错因绑定——KP 拆合后绑定需手动更新
3. 没有错因级别的掌握度追踪——只知道学生犯了这个错因，不知道他对这个错因的掌握程度是否在改善
4. 跨学科 L2 错因体系未建立——语文、英语的 L2 还在规划中，当前仅数学 G5-G6
5. ~~诊断引擎匹配失败时的信号格式未在协议层定义~~（v1.2 已通过新增 unknown_error 兜底错因解决——unknown_error 命中后的即时诊断动作已在 §1.2 中定义：S07→S08 兜底策略包→标记 inconclusive→观察列表）


### 1.6 设计演化推理链

以下记录错因分类体系每个核心设计决策的"为什么"——让后来的开发者和 AI 理解设计意图，判断未来的修改是否违背初始约束。

**为什么选择两层（L1/L2）而非平铺或三层？**

平铺 87 个无结构错因→无法维护，无法跨学科复用。三层（学科/通用/学科专属）→小学数学单学科时过度设计，分层成本多于收益。两层是在当前数学单学科和未来全学科之间的最佳折中——L1 的 10 类通用错因（prerequisite_gap / concept_confusion / comprehension_bias / calculation_error 等）是所有学科共享的认知失败模式，语文阅读理解的"审题偏差"和数学应用题的"题意理解偏差"是同一个 L1（comprehension_bias）。L2 各学科独立管理。

设计决策：两层固定，L1 跨学科共享，L2 按学科独立扩展。未来新增语文 L2 时只需新增 `l2_chinese_specific`，不动现有 L1。

**为什么 L1 是 10+1（含 unknown_error）而非更少或更多？**

认知错误分类文献通常识别 8-12 个根类别——10 类覆盖了所有常见模式（从"前置知识缺失"到"表达格式问题")但不过度细分。将 11 类再缩小（如合并为 6 类）会丢失诊断精度——calculation_error 和 strategy_missing 的干预策略完全不同。将 11 类再扩大（如 15-20 类）会导致 L1/L2 边界模糊，违背"通用"设计原则。第 11 类 unknown_error 是 v1.2 根据 DeepSeek 三方地狱审计新增——解决 S07 诊断引擎匹配失败时无约定信号的问题。设计原则：unknown_error 只用于"兜底"不是"凑数"——所有诊断必须落在 10+1 范围内，超出范围应触发聚类分析产出新候选错因。

**为什么 errorCode 是不透明字符串而非编号层级？**

编号层级（如 E01-01-001）隐含层级语义，会诱使开发者"从 code 猜含义"——长代码维护时，E01-01-001 和 E01-01-002 的区别没人记得住。不透明字符串（如 calculation_error）是自描述的，任何开发者无需查表即可理解。另一个设计约束：跨学科兼容——数学用 calculation_error，语文学科可能用 character_composition_error，两者的 L1 归属都在 comprehension_bias 下但不共享编码体系。若用编号层级，语文和数学的 L2 代码空间会冲突。

**为什么 parentExplanation 必须经人工审核才能入库？**

parentExplanation 是写给家长看的"孩子为什么错"的唯一文本，在 S12 家长摘要中直接展示。AI 生成的文本可能在语法上正确但情感上伤害——将"未掌握异分母通分"解释为"孩子比较粗心"就是态度归因，触犯北极星底线信号。设计决策：AI 可以生成草稿（source: auto_generated）但不能入库——教研人员必须逐条审核措辞。这道闸门在所有进化阶段不拆除（§2.8 显式声明）。这是教育伦理底线，不妥协。

**为什么 L1 没有逐 KP 的定制化绑定？**

10 个 L1 错因是通用模式——每类 L1 几乎在所有 KP 都可能出现，逐 KP 维护绑定是 394×10=3940 条冗余记录。设计采用"默认生成规则"：每 KP 自动获得 prerequisite_gap + calculation_error；应用题追加 comprehension_bias + info_omission；公式记忆类追加 memory_retrieval_failure；解题流程类追加 strategy_missing。这些规则确保 R4/R7 铁规通过，同时避免数据冗余。L2 保持逐 KP 手工绑定——因为 L2 是学科专属的精细诊断，需要人工判断。

**为什么错因不设"严重程度"或"分数扣减"属性？**

同一种错因在不同 KP 的严重性天差地别——"计算错误"在口算题和小数应用题中的扣分权重不同。严重性取决于知识点级别和出题分值，不是错因本身。设计原则：S03 只管"是什么错误"，S07 诊断引擎结合上下文判定"这个错误有多严重"，S08 策略引擎决定"用什么策略纠正"。S03 不存储任何与上下文相关的属性。

**为什么绑定密度不是越高越好？**

每个 KP 的绑定密度设定在平均 3.2（G5-G6）~3.8（G1-G4）是经过平衡的——过低（<2）导致 S07 无法精准匹配；过高（>10）导致诊断引擎在多个候选错因间摇摆，匹配耗时增加（S07 要求在 200ms 内完成）。R7 对 A 级 KP 要求 ≥3 绑定是对最小密度做下限约束，不是上限。

**为什么 L2 错因目前仅覆盖 G5-G6？**

S03 的 L2 错因与 G5-G6 知识点同步构建，G5-G6 是 MVP 首发范围（S01 初始 189 个 KP）。G1-G4 的 L2 错因定义尚未启动——G1-G4 的知识点（20 以内加减法/乘法口诀/基本几何）的错误模式与 G5-G6（分数/小数/方程/立体几何）完全不同，不能简单迁移。Phase 1 计划在 G5-G6 稳定运行后启动 G1-G4 L2 构建。当前 G1-G4 通过 L1 通用错因和跨年级 L2（如分数基础、单位换算）提供基本诊断能力——粒度粗但可用。

### 1.7 第一层禁用清单

- 禁止两个错因边界重叠（导致诊断引擎匹配摇摆）
- 禁止修改 errorCode 字符串（全局外键）
- 禁止删除已被 R4 铁规引用的错因而不更新绑定

### 3.2 自我进化执行方法

五类角色权限台账（含权限回收规则）：

五步流程：信号采集（≥2周教学日）→方案生成→审核（三级权限）→安全执行（影子→A/B→全量）→记录验证（30天稳定期）。

**A/B 测试分组规则**：实验组与对照组按 studentId 哈希值取模分配，流量切分默认 10%/90%（实验/对照）<!-- AI-MUTABLE: AB测试_实验组比例, type=threshold, range=[5%,50%], current=10% -->。显著性检验标准：p<0.05 且效应量 Cohen's d≥0.2。实验组北极星底线信号恶化→立即终止实验。

### 大白话

错因体系就是「学生为什么错」的词典。就像医院的疾病分类——不是所有肚子疼都是胃病，可能是肠、肝、胆。L1 通用错因 11 种跨学科通用（计算粗心/看错题/忘了公式），L2 数学专属 76 种（通分规则错/小数点定位错/面积公式混）。第 11 个 L1 叫 unknown_error——诊断引擎实在对不上号的时候，用这个兜底。

这个词典被三个系统用：诊断引擎拿它匹配「孩子这次错属于哪个类型」，策略包体系拿它选「这个错误应该用什么策略纠正」，家长摘要系统拿 parentExplanation 生成「给家长看的解释」。

进化路线图：从人工定义错因到系统自己发现错因。诊断引擎运行久了会发现——某个「非标准错因」频繁出现、聚类稳定——系统提交候选新错因供教研员确认。自动生成的家长解释必须教研审核才能入库——这道人工闸门永远不拆，因为 AI 可能写出「合法但伤害」的措辞。阶段 2 能启动还有个前提：诊断引擎在完全匹配不上时必须发 unknown_error 信号，不能发空字符串或 null——否则聚类分析拿不到任何数据。

不可修改的就是 errorCode 字符串——这是全局外键，改了就等于把门诊系统、药房系统、病历系统的关联全斩断。还有 parentExplanation——写给家长看的解释不能歪曲，不能把学生的错误归因为「态度不认真」「脑子不够用」，必须严格限定在认知层面。

最隐蔽的不可修改项：错因体系不能反向给学生贴标签。犯了 3 次 comprehension_bias 不等于「这个孩子阅读理解有问题」——那是画像系统的事，错因只管「这道题哪里错了」。标签化会把教学变成治疗，把教室变成诊室。

自我进化执行方法就像开药店——芽芽只能调整非处方药的摆放位置（低风险），上架新药必须药剂师（教研员）审核，换配方必须老板（产品经理）签字。信用分就是三本账——低风险、中风险、高风险各自独立，取最低分决定你能开多大权限。A/B 测试就是新药先给 10% 的病人试，90% 继续用老药，效果有统计显著性才能推开。元进化递归上限就是——系统可以建议修路线图（第 1 层），这个建议可以触发修"怎么修路线图"的规则（第 2 层），但不能再往深处嵌套——就像不能让人自己给自己做脑部手术。

安全执行管道：影子模式拿脱敏后的历史数据先跑一遍（学生名字抹掉、时间只留到周）→小范围 A/B 测试→全量上线。每一步都要过北极星安检。

异常处置和熔断：同一个方向连续失败 3 次→冻结 60 天；卡在某个阶段 18 个月→信用分每月扣 5 分。AI 越界碰了不可修改的东西——轻则退回警告，中则冻结 7 天，重则永久封禁并回滚。

内容进化（改错因描述）和代码进化（改算法）走两套流程，混在一起的时候两套都过。

G1-G7 七个卡点在错因体系的每个关键节点预埋——从变更前的边界检查到全量上线前的 PSI 检查，不缺位。
