## 文档定位

这是芽芽所有子系统设计文档的最后一道质量防线。

它不是建议，不是指南，是门禁。任何子系统文档，在提交前必须通过以下全部五级门禁的审查。BLOCK 直接拒收，不允许带风险上线。WARN 记录但允许通过——但连续 3 次 WARN 自动升级为 BLOCK。

本验收标准本身也遵循自我进化原则——它的进化触发条件、进化方式和不可修改边界在本章末尾定义。

## 第一级门禁：格式合规（自动化检查）

执行者：`acceptance_runner.py` 脚本自动执行
阻断规则：任何 BLOCK 直接拒收，不进入下一级审查

| # | 检查项 | 判定标准 | 级别 |
|---|--------|---------|------|
| 1.1 | 必填章节完整 | 文档必须包含：变更记录、北极星对齐、定义与职责、核心设计、它还不会什么、设计演化推理链、自我进化路线图、自我进化执行方法、消费关系清单、不可修改边界。缺任意一章 → BLOCK | BLOCK |
| 1.2 | 变更记录存在且格式正确 | 变更记录至少有一条，且最新版本号与文档标题中的版本号一致。版本号遵循语义化规范（major.minor.patch） | BLOCK |
| 1.3 | 设计理由/历史分离 | 全文搜索，不得出现"因为……之前……所以改成……"的因果混合表述。检测到混合表述 → WARN，由审查员判断是否为误报 | WARN |
| 1.4 | AI-MUTABLE 标注完整 | 所有可进化阈值必须独立成行，且每行附带 `<!-- AI-MUTABLE: xxx -->` 注释。阈值不得嵌入段落文本中。缺任意一处 → BLOCK | BLOCK |
| 1.5 | 消费关系清单表格完整 | 表格必须包含五列：本系统的元素、被消费方、消费方式、故障类型、检测方式。故障类型必须在[静默失败/崩溃/数据错误]枚举内。检测方式必须在[单元测试/集成测试/仪表盘监控/人工发现/无]枚举内。缺列或枚举值非法 → BLOCK | BLOCK |
| 1.6 | 大白话章节存在 | 六类必配大白话的章节类型（系统职责定义/核心设计/自我进化路线图/不可修改边界/接口约定/自我进化执行方法），每类至少有一条独立的大白话段落（以 `### 大白话` 或 `#### 大白话` 子标题开头）。缺 → WARN | WARN |
| 1.7 | 不可修改边界表格完整 | 不可修改边界章节必须包含至少 6 类不可修改项（外部法规定义/全局外键/通信协议/学生原始数据/质量门禁/北极星底线信号覆盖项），每项必须包含：不可修改的原因、AI尝试修改的直接后果、AI的正确行为。缺任一 → BLOCK | BLOCK |
| 1.8 | 间接修改边界表存在 | 不可修改边界章节必须包含间接修改边界表，列出至少 3 种间接修改场景（含间接影响的不可修改项、机制、必须采取的行动）。缺 → BLOCK | BLOCK |
| 1.9 | 越界响应规则存在 | 不可修改边界章节必须包含越界行为的系统响应规则（直接修改拒绝、间接影响评估、人工审核豁免）。缺 → BLOCK | BLOCK |
| 1.10 | 北极星对齐章节存在 | 文档必须包含"北极星对齐"或等效章节，明确本系统如何服务于核心使命、三层检测信号（正/负/底线）有哪些。缺 → BLOCK | BLOCK |
| 1.11 | 进化路线图维度完整 | 进化路线图至少包含 1 个独立维度。每个维度必须包含六要素：维度名称、阶段数、每阶段的触发条件（标注自主/人工依赖类型）、系统能力描述、人工角色描述、终局状态描述。缺任一维度或任一要素 → BLOCK | BLOCK |
| 1.12 | 自我进化执行方法章节存在 | 文档必须包含四类消费者定义、五步执行流程（信号采集→方案生成→审核与自主进化→安全执行与回滚→进化记录与验证）、三级权限分级。缺 → BLOCK | BLOCK |
| 1.13 | 各环节安全机制汇总表存在 | 自我进化执行方法章节必须包含安全机制汇总表，覆盖信号采集/方案生成/审核/安全执行/记录验证五个环节，每环节有关键安全机制和失败时的行为。缺 → BLOCK | BLOCK |
| 1.14 | 验收预览可复现 | 文档中定义的所有数据结构（JSON 示例、表格、字段清单）必须在验收预览 HTML 中能实际渲染出来，且渲染结果与文档中的描述一致（字段名、层级、值域）。无法渲染或渲染结果与文档不一致 → BLOCK。验收预览的生成脚本作为第一级门禁的一部分自动执行——文档提交前跑一遍，预览失败直接拒收 | BLOCK |

> 1.11 与第三级 3.1 的职责分离：一级查「有没有」（格式层面——维度数、阶段数、六要素是否齐全），三级查「配得对不对」（逻辑层面——配套质量）。如果一级因缺要素而 BLOCK，文档不会进入三级——这保证了每级审查只关注自己能判断的东西。

## 第二级门禁：接口对齐（跨文档自动检查）

执行者：`cross_ref_validator.py` 脚本自动执行
阻断规则：任何 BLOCK 直接拒收，不进入下一级审查

| # | 检查项 | 判定标准 | 级别 |
|---|--------|---------|------|
| 2.1 | 字段名一致性 | 本文档引用的其他文档字段名（如 curriculumLevel、parentExplanation），必须与目标文档中的实际字段名完全一致（含大小写、单复数）。不一致 → BLOCK | BLOCK |
| 2.2 | 事件编号一致性 | 引用 09 号事件协议的事件编号（如"事件 5：diagnosis_attempted"），必须与 09 号文档当前版本的实际编号一致。不一致 → BLOCK | BLOCK |
| 2.3 | 版本引用有效性 | 引用的其他文档版本号（如 S07 诊断引擎 v2.0），必须是该文档已发布的版本。引用不存在版本 → BLOCK | BLOCK |
| 2.4 | 枚举值一致性 | 本文档定义的枚举值（如 contentType、diagnosisStatus），必须与消费该枚举的所有下游文档的枚举定义一致。新增枚举值必须在本文档和所有消费方文档中同步声明。不一致或未同步 → BLOCK | BLOCK |
| 2.5 | 知识图谱边一致性 | 引用的 prerequisites/next 关系，必须与 knowledge_graph.json 中的实际边一致。不一致 → BLOCK | BLOCK |
| 2.6 | 策略包编码一致性 | 引用的策略包编码（如 SP_G_C01），必须与 08_STRATEGY_PACKS.md 中的编码体系和 Phase 归属一致。不一致 → BLOCK | BLOCK |
| 2.7 | 错因编码一致性 | 引用的错因编码（如 decimal_point_position_error），必须与 error_taxonomy.json 中的实际 errorCode 一致。不一致 → BLOCK | BLOCK |
| 2.8 | 消费关系清单的反向一致性（三级判定） | 1. 双边都声称消费关系，但字段名/故障类型/检测方式不一致 → BLOCK（数据断裂信号，必须堵死）。2. S01 声称被 S07 消费，但 S07 的消费关系清单中完全找不到 S01 → BLOCK（单边声明，同样危险——S01 改了字段后 AI 不会去检查 S07，导致静默失败）。3. S07 的消费关系清单中包含 S01，但 S01 未在接口约定中列出 S07 → WARN（未声明的依赖，风险稍低但需补齐） | BLOCK / WARN |

#### 大白话

消费关系清单是所有子系统之间的「神经网络」。如果 S01 说被 S07 消费但 S07 不认——这个断裂意味着 S01 改了一个字段后，AI 不会去检查 S07。因为在 S07 的消费关系清单里，它根本没说自己消费了这个字段。这就是静默失败——最危险的那种。

## 第三级门禁：逻辑自洽（AI 初筛 + 人联合审查）

执行者：AI 初筛（通过/疑似/不通过）→ liufeng 抽检或亲自确认
阻断规则：BLOCK 直接拒收。WARN 记录但允许通过——连续 3 次 WARN 自动升级为 BLOCK

| # | 检查项 | 判定标准 | 级别 | 审查员 |
|---|--------|---------|------|--------|
| 3.1 | 触发条件闭环（配套质量——不是查有没有，是查对不对） | 超时天数的设定是否符合教研工作节奏（不是随便填了个 7 天）、反馈格式的字段是否与 review_queue.json 的实际 Schema 一致、二级响应的触发次数（如连续 3 次全部超时）是否在极端情况下会导致审核通道永锁。不合理 → WARN | WARN | AI 初筛 + liufeng 抽检 |
| 3.2 | 消费关系无遗漏 | 所有本系统产出的数据/事件/字段/规则，都在消费关系清单中有对应行。审查员逐项核对——是否有任何本系统定义的东西，没有被任何下游消费，但文档声称它会被消费？有遗漏 → WARN | WARN | AI 初筛 |
| 3.3 | 数据来源明确 | 所有需要外部填充的字段（如 observations、cognitive_primitives），必须标注数据由谁在什么时候写入。未标注 → WARN | WARN | AI 初筛 |
| 3.4 | 冷启动/边界/异常三态覆盖 | 关键规则必须覆盖三种状态：(1) 冷启动时怎么办（如新学生无历史数据时的 fallback 逻辑）；(2) 边界值怎么处理（如最大溯源深度 3 层后仍无结论的出口）；(3) 异常怎么兜底（如事件缺失 chainId 时的降级处理）。缺任一态 → WARN | WARN | AI 初筛 + liufeng 抽检 |
| 3.5 | 漏洞自检真实 | "它还不会什么"章节列举的缺口，必须真实反映当前系统的能力边界——不得把已实现的功能列成缺口，也不得把永远不会实现的能力列成缺口。每个缺口必须对应进化路线图中至少一个维度的至少一个阶段。缺口无对应维度 → BLOCK。缺口描述与路线图阶段能力描述不一致 → WARN | BLOCK | liufeng 亲自确认 |
| 3.6 | 设计理由与设计历史不混淆 | 审查员逐条检查所有标注了"设计理由"和"设计历史"的条目——"设计理由"中不得包含时间信息（如"v2.0 之前"），"设计历史"中不得包含约束性声明（如"因此必须保持"）。混淆 → WARN | WARN | AI 初筛 |
| 3.7 | 不可修改边界无遗漏 | 审查员逐项核对不可修改边界清单——是否有任何本系统定义的东西，属于不可修改边界的六种类型但未被列入清单？有遗漏 → BLOCK | BLOCK | AI 初筛 + liufeng 确认 |
| 3.8 | 间接修改边界的"机制"列明确 | 间接修改边界表中的每一行，"机制"列必须说清楚"修改了什么→通过什么路径→影响了哪个不可修改项"。模糊描述（如"可能影响"）→ WARN | WARN | AI 初筛 |
| 3.9 | 进化路线图的维度之间无循环依赖 | 进化路线图的多个维度之间不得形成循环依赖（如维度 A 触发→维度 B 修改→维度 A 再次触发，形成死循环）。有循环依赖 → BLOCK | BLOCK | AI 独立完成 |
| 3.10 | 自我进化执行方法的三级权限与不可修改边界一致 | 低风险/中风险/高风险示例中，不得出现不可修改边界清单中的项目。出现 → BLOCK | BLOCK | AI 独立完成 |
| 3.11 | 北极星三层信号与进化维度对应 | 北极星对齐章节中定义的正信号/负信号/底线信号，必须在进化路线图的各维度中有对应的检测节点（注明在哪个阶段的哪个环节检查哪个信号）。缺对应 → WARN | WARN | liufeng 亲自确认 |
| 3.12 | 北极星对端指标存在 | 任何进化路线图中标注为 AI-MUTABLE 的阈值修改，必须在北极星对齐章节中对应一个对端指标（被优化的指标升了，对端指标不能降）。缺 → BLOCK | BLOCK | AI 初筛 + liufeng 确认 |
| 3.13 | 元进化被视为高风险操作 | 任何关于"系统自我修改路线图"的描述，必须明确标注为高风险操作，且必须通过北极星检查+人工签批。未标注 → BLOCK | BLOCK | AI 独立完成 |
| 3.14 | 回滚可追溯 | 所有进化操作的数据结构中，modificationHistory 必须包含 startedAt/endedAt/rollbackAt/description/confoundingEvents 五个字段。缺任一 → BLOCK | BLOCK | AI 独立完成 |
| 3.15 | "人工不可绕行"标注 | 北极星检查失败后的处理逻辑中，必须明确标注"人工不可绕行"。未标注 → BLOCK | BLOCK | AI 独立完成 |

#### 大白话

第三级是 AI 先帮你筛一遍，标注好「通过」「疑似」「不通过」，然后你只需要看「疑似」和「不通过」的。15 条里只有 3.5 和 3.11 你必须亲自确认——其余 AI 能独立完成或初筛后由你抽检。

3.5（漏洞自检真实）必须你亲自看，因为只有你知道系统当前的真实能力边界——AI 可能把已实现的功能误判为缺口。3.11（北极星信号与维度对应）必须你亲自看，因为只有你知道哪些信号是北极星的核心、哪些是噪音。

## 第四级门禁：进化安全（最深层的系统性审查）

执行者：liufeng 最终确认
阻断规则：BLOCK 直接拒收，不允许带风险上线

| # | 检查项 | 判定标准 | 级别 |
|---|--------|---------|------|
| 4.1 | 进化路线图的阶段切换不会产生"孤儿数据" | 当系统从阶段 N 切换到阶段 N+1 时，阶段 N 产生的数据（如旧格式的 KP 绑定、旧级别的认知质标记）必须有明确的迁移或归档规则。无规则 → BLOCK | BLOCK |
| 4.2 | 安全执行管道的每一层都有北极星前置检查 | 影子模式→A/B 测试→全量上线的每一层，在上线前必须通过北极星检查（正信号/负信号/底线信号）。缺任一层 → BLOCK | BLOCK |
| 4.3 | 回滚条件覆盖所有关键失败场景 | 回滚条件必须至少覆盖：A/B 测试指标恶化、全量上线超出基线 95% 区间、北极星底线信号触发、不可修改边界告警、下游 BLOCK 错误。缺任一 → WARN | WARN |
| 4.4 | 元进化本身被约束 | 元进化被视为高风险操作，且元进化修改路线图后，新的路线图必须重新通过完整的四级门禁验收（模拟运行）。未标注此约束 → BLOCK | BLOCK |
| 4.5 | 认知质/错因/策略包的 shareable 判定不会导致隐私泄漏 | 跨校复现验证的数据集，不得包含可追溯到单个学生或单个学校的数据。shareable 判定流程中必须明确隐私保护措施。缺 → BLOCK | BLOCK |

#### 大白话

第四级查的不是「文档写对了没有」，而是「进化的时候会不会炸」。大部分验收标准停在第三级——文档本身逻辑自洽就行。但芽芽的子系统要自我进化——进化的时候旧数据怎么办？每条进化管道都过了安检没？万一进化失败有没有退路？进化路线图自己改了自己之后谁来审？这些是真上线才会炸的雷，第四级在你上线前替你排了。

## 第五级门禁：哲学一致性（最终审查）

执行者：liufeng 亲自确认
阻断规则：不通过则退回重写

| # | 检查项 | 判定标准 | 级别 |
|---|--------|---------|------|
| 5.1 | 系统边界清晰 | 文档中明确写明了"这个系统不负责什么"，且"不负责"的边界与芽芽整体架构一致 | 不通过 |
| 5.2 | 系统的"不可为"被尊重 | 事实判断和价值判断的边界清晰——系统能做的（效率优化）和不能做的（价值判断）被明确区分。进化路线图中没有试图让系统跨过价值判断边界 | 不通过 |
| 5.3 | 进化路线图有"终局视野"（两个量化信号） | 信号一：进化路线图的最后阶段，是否描述了该系统在 K12 全学科+大学+终身学习三层扩展中的最终形态？如果最后阶段只写到"小学数学阶段完全自主"，没有提及其他学科或学段的扩展能力→不通过。信号二：文档中是否有至少一处"设计理由"直接引用了终局愿景？比如"此字段设计为不透明标识符——不绑定特定教材版本，为将来跨版本映射预留空间"。没有→不通过 | 不通过 |
| 5.4 | 文档本身是"活的" | 文档中预埋了未来进化的数据结构（如 baselineMetrics、modificationHistory、AI-MUTABLE），让文档本身成为系统自我进化的基础设施 | 不通过 |

#### 大白话

前四级查的是骨头和肌肉——格式对不对、接口对不对、逻辑对不对、进化安不安全。第五级查的是灵魂——这个系统到底有没有想清楚「我最终要成为什么」。

5.3 不再是「你觉得有没有终局视野」，而是两个硬指标：路线图有没有写到三层扩展、设计理由里有没有引用终局愿景。没有就不过——不是感觉问题，是证据问题。

## 验收标准本身的自我进化

本验收标准遵循与 DOCUMENTATION_STANDARDS.md 相同的自我进化原则。

### 进化的触发条件

- **新原则加入**：DOCUMENTATION_STANDARDS 新增一条原则时，本验收标准自动增加对应的门禁检查项
- **某检查项连续 3 次审查未被触发**：说明该检查项太松，实际文档质量已远超标准 → 自动收紧判定标准
- **某类问题在多次审查中重复出现但本验收标准未覆盖**：说明标准有盲区 → 新增检查项
- **DOCUMENTATION_STANDARDS 本身变更**：子原则合并、拆分、重构时，本验收标准对应调整

### 进化方式

- **低风险**（新增检查项、调整 WARN/BLOCK 判定、更新示例）：芽芽自主执行，事后报告
- **高风险**（删除检查项、改变门禁层级、新增门禁层）：推送人工审核

### 本验收标准的不可修改边界

1. 五级门禁的层级结构不可删除或合并
2. BLOCK 的直接拒收规则不可降级为 WARN
3. 第五级"哲学一致性"不可删除——这是系统设计文档的灵魂审查
4. 本验收标准本身的自我进化原则不可删除
5. 1.14 验收预览可复现不可降级为 WARN——这是堵「文档写了但数据文件里没有」的最后防线
