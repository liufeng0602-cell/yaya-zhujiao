## 文档定位

这是芽芽所有子系统设计文档的最后一道质量防线。任何子系统文档提交前必须通过以下全部五级门禁。BLOCK直接拒收，WARN记录但允许通过——连续3次WARN自动升级为BLOCK。

本验收标准本身遵循自我进化原则——触发条件、进化方式和不可修改边界在本章末尾定义。

**本文档的技术描述与大白话描述已通过语义一致性检测（原则六）**。

## 第四级门禁：进化安全（最深层的系统性审查）

执行者：liufeng最终确认
阻断规则：BLOCK直接拒收

| # | 检查项 | 判定标准 | 级别 |
|---|--------|---------|------|
| 4.1 | 阶段切换不产生孤儿数据 | 阶段N切换到N+1时，旧阶段数据有明确迁移或归档规则。无规则→BLOCK | BLOCK |
| 4.2 | 安全执行管道每层有北极星前置检查 | 影子模式→A/B测试→全量上线每层上线前通过北极星检查。缺任一层→BLOCK | BLOCK |
| 4.3 | 回滚条件覆盖所有关键失败场景 | 回滚条件至少覆盖：A/B测试指标恶化（PSI≥0.1）、超基线95%区间、北极星底线信号触发、不可修改边界告警、下游BLOCK错误。缺任一→WARN | WARN |
| 4.4 | 元进化本身被约束 | 元进化被视为高风险操作，新路线图必须重新通过完整四级门禁验收。未标注→BLOCK | BLOCK |
| 4.5 | Shareable判定不导致隐私泄漏 | 跨校复现验证数据集不含可追溯到单个学生/学校的数据。shareable判定流程中明确隐私保护措施。缺→BLOCK | BLOCK |
| 4.6 | 北极星检查模块有自保机制 | 检查模块受四道防线约束（修改者隔离+防旧错+防新错+月度自检），且标注为高风险操作。缺任一→BLOCK | BLOCK |
| 4.7 | 消费关系清单有实时校验机制 | 文档commit时自动扫描全消费链路，字段名变更/字段删除时检测断裂并CI阻断。缺→BLOCK | BLOCK |
| 4.8 | 全局注册表宕机有中断处置 | 注册表宕机流程已定义——30秒检测→中断处置→恢复检查→24小时超时回滚。缺→BLOCK | BLOCK |
| 4.9 | 跨层G1-G7卡点全部预埋 | 七卡点均在该子系统对应的进化流程中有明确触发位置和失败行为。缺任一→WARN | WARN |
| 4.10 | 信用分三子分体系完整 | 聚合规则（展示=min/分级暂停/全系统暂停min<10/月度+20上限）、独立加减分细则、跨子分挪用拦截、破产恢复、重置场景全部定义。缺→BLOCK | BLOCK |

### 大白话

第四级查的不是「文档写对了没有」，而是「进化的时候会不会炸」。大多数验收标准停在第三级——文档本身逻辑自洽就行。但芽芽的子系统要自我进化——旧数据怎么办、每步过安检没、失败有退路没、路线图改了之后谁再审、调度员宕机了怎么办、信用分三本账怎么独立算——这些是真上线才会炸的雷。

前四级查骨头和肌肉——格式、接口、逻辑、安全。第五级查灵魂——这个系统有没有想清楚「我最终要成为什么」。不是你觉得有没有终局视野，是两个硬指标：路线图有没有写到三层扩展、设计理由有没有引用终局愿景。

## 验收标准本身的自我进化

本验收标准遵循与DOCUMENTATION_STANDARDS.md相同的自我进化原则。

### 进化的触发条件

- **新原则加入**：DOCUMENTATION_STANDARDS新增一条原则时，本验收标准自动增加对应的门禁检查项
- **某检查项连续3次审查未被触发**：说明该检查项太松→自动收紧判定标准
- **某类问题在多次审查中重复出现但本验收标准未覆盖**：说明标准有盲区→新增检查项
- **DOCUMENTATION_STANDARDS本身变更**：子原则合并、拆分、重构时，本验收标准对应调整

### 进化方式

- **低风险**（新增检查项、调整WARN/BLOCK判定、更新示例）：芽芽自主执行，事后报告
- **高风险**（删除检查项、改变门禁层级、新增门禁层）：推送人工审核

### 本验收标准的不可修改边界

1. 五级门禁的层级结构不可删除或合并
2. BLOCK的直接拒收规则不可降级为WARN
3. 第五级「哲学一致性」不可删除
4. 本验收标准本身的自我进化原则不可删除
5. 1.14验收预览可复现不可降级为WARN
