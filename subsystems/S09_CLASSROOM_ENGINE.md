## 北极星对齐

课堂生成引擎是芽芽"千人千面"的核心输出层——学生直接看到的东西全部来自这里。北极星校验走"生成内容安全三关"：教材边界关→学生适配关→认知伤害关。

| 优先级 | 信号层 | 判定规则 |
|--------|--------|---------|
| 最高 | 底线信号 | 任一触碰 → 该次生成结果被拦截，回退为通用骨架（不个性化）<!-- 通用骨架——标准定义见 §通用骨架标准定义 — 纯模板输出，不调LLM，不经过质检验-- > |
| 中等 | 负信号 | 任一恶化 → 停止该画像维度的个性化，该维度退回默认值 |
| 基础 | 正信号 | 仅当底线/负信号均无恶化时，正信号至少 2 项改善 → 个性化升级通过 |

**正信号**（个性化有效）：
- 学生在个性化课堂上正确率高于通用课堂（同一 KP 对比）<!-- AI-MUTABLE: 正确率提升阈值 5%, type=float, range=[0.03, 0.10] -->
- 学生求助次数减少（同一 KP 对比通用课堂）<!-- AI-MUTABLE: 求助下降阈值 10%, type=float, range=[0.05, 0.15] -->
- 学生完成课堂后主动回访率上升
- 学生对某个生活场景（游戏/美食/运动）的反应时间明显短于其他场景

**负信号**（个性化失配）：
- 个性化课堂的退出率高于通用课堂 20% 以上<!-- AI-MUTABLE: 退出率差异阈值, type=float, range=[0.10, 0.30] -->
- 低年级学生出现成人化表达（画像系统误判年级）
- 同一学生连续 3 堂课出现"内容跳过"（反馈话术重复或无效→学生对反馈麻木）

**底线信号**（绝对不可触碰）：
- 生成内容含教材外的新概念（超纲）→ 拦截，回退通用骨架<!-- 通用骨架——标准定义见 §通用骨架标准定义 — 纯模板输出，不调LLM，不经过质检验-- ><!-- AI-MUTABLE: 超纲检测严格度 100%, type=float, range=[0.95, 1.0] -->
- 生成内容含"AI""人工智能""机器"等非教育场景用语→ 拦截
- 生成内容包含对学生的负面标签（"你太粗心了""你就是学不会"）→ 拦截
- 低年级（1-3）课堂使用了纯文字无语音引导→ 拦截

## 定义与职责

### 这个系统负责什么

课堂生成引擎是芽芽的"教案老师"——把 S08 策略引擎选好的"怎么教"变成学生真正看到的一堂互动课。

输入：学生画像 + 知识点 + 策略处方 → 输出：一份完整的课堂脚本 JSON，包含开场白、讲解、练习题、反馈话术、收尾语。前端拿到这份 JSON 渲染即可，不需要再调 AI。

六步生成管线：骨架选型 → 知识装载 → 画像注入 → 血肉生成 → 质检 → 输出。

### 这个系统不负责什么

- 不负责选策略（S08 策略引擎负责，S09 只消费 strategy_applied）
- 不负责诊断学生哪里不会（S07 诊断引擎负责）
- 不负责定义策略包内容（S04 策略包体系负责）
- 不负责管理学生画像（S15-S17 画像系统负责）
- 不负责判断分级标准（S06 分级系统负责）
- 不负责定义知识点内容（S01 KP 管理系统负责）
- 不负责存储长期成长记忆（GrowthMemory 由画像系统管理）


### 通用骨架标准定义

通用骨架是课堂生成引擎的最小安全降级方案——当个性化课堂无法生成时(质检拦截/策略超时/信用分过低/画像缺失)，系统回退到此模式。

- 不绑定任何特定骨架类型——独立于七类骨架
- 五环节均使用教材直接定义(非步骤拆解/非概念提炼/非图文建模)
- 环节内容:
  - 开场: 打招呼 + 告知目标
  - 讲解: 教材定义 + 标准正例
  - 练习: 标准练习题×2(难度0.5固定)
  - 反馈: 正确√错误× + 错因提示(来自S03 error_kp_bindings.json中按绑定频率降序排列的第一个常见错因的parentExplanation)
  - 收尾: 学了什么 + 对了多少
- 全程不注入任何画像钩子
- 不调LLM——纯模板输出
- 跳过质检步骤——纯模板输出无AI生成内容，不需检查超纲/AI字眼/学生适配
- 练习题来源: S13教材题库按knowledgeCode随机取2道难度0.5的题。若S13未就绪(⚠️ S13子系统未建)，降级为S01 knowledge.json中keyPoints数组的前2条以纯文本呈现——此降级不保证教学质量，仅防止空白页面

### 大白话

课堂生成引擎就是"私厨"——S08 策略引擎是"营养师"（告诉你要补什么），S09 是"厨师"（把补品做成你喜欢吃的菜）。骨架是菜谱框架——保证这道菜不会做错（不超纲、不教错知识）。血肉是调味——根据你的口味（画像）加不同的料。对喜欢游戏的孩子用"装备分配"讲分数，对喜欢美食的孩子用"分披萨"讲分数——同一个知识点，完全不同的体验。

## 核心设计

### 三阶段进化：C → D → E

课堂生成从手工到自动再到自我进化的完整路线。这不是技术演进，而是"教案从 AI 一次性生成 → 可复用积木 → 积木自己会长大"的认知进化。

| 阶段 | 比喻 | 核心能力 | 适用 Phase |
|------|------|---------|-----------|
| 方案 C | 私厨定菜谱框架，现场调味 | 七类骨架模板 + AI 实时生成血肉（生活例子/话术/场景包装） | Phase 1 MVP |
| 方案 D | 好吃的菜沉淀为预制调料包 | AI 生成过的效果好内容→变成可复用积木块→按画像自动拼装 | Phase 2 |
| 方案 E | 系统自己学会做他最爱吃的 | 积木自动变异（视觉素材/节奏/话术）、拼装规则从人工写变成系统自学 | Phase 3+ |

### 七类骨架模板

骨架按知识点类型分组。骨架定义每个环节"必须包含什么"，保证教学正确性——不教错、不超纲。

| 骨架类型 | 适用场景 | 五环节编排 |
|---------|---------|-----------|
| 计算类 | 小数乘除、分数加减、百分数计算 | 生活场景引入 → 步骤拆解示范 → 跟练（同结构题）→ 变式练习 → 教学纠错（老师纠正变式题的常见错误后再给独立挑战）→ 独立挑战 |
| 概念类 | 因数倍数、分数意义、比例、方程 | 生活场景引入 → 抽象提炼概念 → 互动判断"是不是"→ 例题巩固 → 总结 |
| 应用题类 | 行程问题、工程问题、鸡兔同笼 | 读题（审题侦探）→ 画图建模 → 列式求解 → 变情境练习 → 自编题挑战 |
| 几何类 | 面积、体积、圆、长方体 | 动画演示推导 → 公式记忆锚点 → 基础代公式 → 变图形练习 → 逆向（给面积求边长） |
| 单位换算类 | 长度/面积/体积/时间单位换算 | 生活对比（一厘米多长→一把尺子）→ 桥梁法示范 → 正向换算 → 反向换算 → 混合换算 |
| 统计类 | 折线图、条形图、扇形图 | 读图示范 → 提取信息练习 → 制图练习 → 判断误导图表 → 总结 |
| 复习类 | 混合复习、期末复习 | 快速回顾（记忆唤醒）→ 错题重练 → 新题混入 → 自我评估 | 复习类豁免开场破冰和收尾总结——快速回顾隐含破冰（重温已会内容），自我评估隐含收尾（知不足）。不设独立开场/收尾环节 |

**骨架覆盖规则**：394 个知识点（G1-G4: 205 / G5-G6: 189）全部能匹配到一个骨架类型。匹配逻辑：questionType 优先级 > 知识点名称关键词 > 教学目标的动词。

### 五环节结构

每堂个性化课包含五个环节，按序执行。

**环节一：开场破冰**（30 秒<!-- AI-MUTABLE: 开场时长, type=int, range=[20, 60] -->）
骨架约束：打招呼 + 激活前置知识（前置 KP 掌握度 < 0.6 时先温习<!-- AI-MUTABLE: 温习触发阈值, type=float, range=[0.4, 0.7] -->）+ 告知本堂课目标。
个性化钩子：激励偏好（积分→"今天挑战完成，解锁三颗星"）、状态卡（轻松→"今天慢慢来，不赶时间"）、上节课退出（pendingAction=energy_recovery→不聊分数，轻松开场）。

**环节二：概念讲解**（120 秒<!-- AI-MUTABLE: 讲解时长, type=int, range=[90, 180] -->）
骨架约束：教材定义 + 一个生活化类比或图示 + 至少一个正例+一个反例 + 关键公式/步骤显式展示。
个性化钩子：呈现偏好（动画/图文/语音/直接做题）、错因历史（概念混淆→正反例对比强调；Phase 1 画像系统未上线→此项回退为读取 S03 静态错因库中该 knowledgeCode 在 error_kp_bindings.json 中按绑定频率降序的第一个常见错因的 errorCode（环节二只需 errorCode 判断错因类型做正反例对比——不需要 parentExplanation；环节四反馈话术才需要 parentExplanation 生成纠错文案。两个环节取同一个错因但用不同字段：环节二取 errorCode，环节四取 parentExplanation）。讲解深度仅靠课程级别区分）、课程级别（A 级→1 个例子不讲推导，D 级→2 个场景+推导过程）。

**环节三：互动练习**（180 秒<!-- AI-MUTABLE: 练习时长, type=int, range=[120, 300] -->）
骨架约束：一道模仿题（与示范同结构）+ 一道变式题 + 一道独立挑战题。（注意：计算类骨架编排中的\"纠错\"是环节三内置的教学步骤——老师批改变式题后纠正再挑战——与环节四\"每道题做完后的答题反馈\"是不同层面的东西）题目从教材区域内出，不超纲。
个性化钩子：挑战偏好（easy→题量减半；challenge→加变式题）、行为模式（快但粗心→每题提交前确认步骤；连错退出型→第一题降到"肯定能做对"级别）、策略包（分步引航包→拆成 3 小步做完一步验一步）。

**环节四：反馈话术**（20 秒/题<!-- AI-MUTABLE: 反馈时长, type=int, range=[10, 40] -->）
骨架约束：每道题做完必须反馈。错误时给具体错因指向（不能说"再想想"），正确时给具体肯定（不能说"你真棒"）。
个性化钩子：激励偏好（夸他→"你主动检查了分母，这个习惯特别好"；积分→"+3 分！第三步特别细心"）、行为模式（连错退出型→"这道题的陷阱藏得特别深，芽芽第一次也错过了"）。

**环节五：收尾总结**（15 秒<!-- AI-MUTABLE: 收尾时长, type=int, range=[10, 30] -->）
骨架约束：今天学了什么 + 对了多少 + 下次预告。
个性化钩子：能量状态（低→"今天到这里，你很努力了。保存体力，下次再战"）、家长目标（补基础→"今天补了一个基础，下次继续加固"）。

**总时长约束**：五环节 AI-MUTABLE range 上限合计不超过 480 秒（8 分钟）——任一时长调上限时，其他环节须同步调下限以保证总时长 ≤ 480 秒。无法在约束内分配时放弃优化，使用默认时长 365 秒（30+120+180+20+15）。

```
3. 画像注入
   └─ 输入：studentId → 查询画像系统（呈现偏好/激励偏好/挑战偏好/行为模式/错因历史/家长目标）
   └─ 输出：个性化参数集（场景类型/话术风格/题量/难度调整/提示层级/确认步骤）
   
4. 血肉生成
   └─ 骨架 + 知识上下文 + 个性化参数 → LLM 实时生成具体内容（生活例子/话术/场景包装）
   └─ 满足千人千面：不同画像的学生看到同一个知识点，生活例子不同、话术不同、节奏不同
   
5. 质检
   └─ 教材边界检查（不含超纲概念）+ 禁止词检查（无 AI 字眼。例外规则：教学场景中作为比喻/类比使用的「机器人/计算器/电脑」等词汇不拦截——判断依据为词汇在句中作为比喻对象而非系统自称。如「你算得比计算器还快」「像机器人一样重复」「像电脑一样精准」均属于教学比喻→放行；「我是AI助教」「系统推荐这道题」属于系统自称→拦截）+ 学生适配检查（话术级别匹配年级）
   └─ 未通过→回退第 4 步重新生成（同参数二次生成利用 LLM 随机性，极少连续失败；连续 2 次未通过→回退通用骨架跳过个性化<!-- 通用骨架——标准定义见 §通用骨架标准定义 — 纯模板输出，不调LLM，不经过质检验-- >）
   
6. 输出
   └─ 结构化课堂脚本 JSON（符合 §输出 JSON Schema）+ 发出 lesson_generated 事件（09 号事件协议）
   └─ 环节四的错因指向：学生做错时，S09 从缓存中读取最近 1 次 diagnosis_updated。若缓存中无 diagnosis_updated 且 S03 静态错因库的 parentExplanation 也缺失(数据不一致)→兜底使用通用话术「再检查一下计算步骤，看看哪里出了问题」——保证环节四不会出现空白反馈（按 studentId+knowledgeCode）。S07 完成异步诊断后通过 09 号事件协议主动推送 diagnosis_updated，S09 缓存最近 3 次诊断结果（按 studentId+knowledgeCode 维度聚合——每个学生每个 KP 独立维护一个最多 3 条记录的 LRU 缓存。淘汰策略：第 4 条记录到达时淘汰最早的那条。缓存 3 次而非 1 次的原因: 最近 1 次用于正常反馈，次近 2 次用于以下降级场景: (a)最近 1 次超过 7 天→取次近 1 次；(b)最近 1 次的诊断状态为 pending→优先等待实时诊断完成(500ms超时)，超时后取次近 1 次。缓存自然上限为活跃 student×KP 组合数（每个组合最多3条LRU记录），Phase 1 预计约 1500 个活跃组合（8 种画像×189 个 KP ≈ 1512 组合，实际活跃组合约 1500），对应最多 1500×3=4500 条缓存记录）。课堂反馈时从缓存读取，不实时查询。S07 诊断是异步流程（初筛→匹配→验证→溯源），有延迟——如果最新 diagnosis_updated 的 timestamp 距当前超过 7 天或不存在，回退使用 S03 静态错因库的 parentExplanation（与当前 knowledgeCode 绑定的常见错因模板）。实时诊断到达后（以 diagnosis_updated 事件推送为准），S09 在下一次该学生练该知识点的反馈中使用新诊断
```

## 设计演化推理链

### 为什么不是"一个 LLM 全生成"

如果让一个 LLM 从零生成整个课堂——它会自由发挥，可能教错、可能超纲、可能用不适合低年级的话术。这不只是质量问题，是安全问题。骨架的存在就是为了锁死教育正确性的底线——LLM 只能在骨架划定的框里自由发挥。

### 为什么不是"纯规则引擎"

纯规则引擎能做到"千人一面"——所有学生看到完全一样的课堂。但芽芽的核心是"越用越懂他"——同一个知识点，不同学生看到的应该完全不同。规则引擎做不到这种级别的个性化。

### 为什么是三阶段 C→D→E 而不是一步到位

C→D→E 不是技术妥协，是数据积累的必然路径。方案 C 是种子阶段——先让系统跑起来，积累 AI 生成的血肉和学生的反应信号。方案 D 是沉淀阶段——把 C 阶段积累的好东西提取出来变成可复用的积木。方案 E 是进化阶段——有了足够的数据和积木库，系统才能自学"什么对学生有效"。

核心假设：没有 C 阶段的数据积累，D 和 E 都是空中楼阁。这就是为什么 Phase 1 只做方案 C——不是不做 D/E，是必须先有数据。

### 为什么刚好是七类骨架

394 个知识点按类型聚类后，自然形成了 7 个组。7 不是设计出来的，是数据聚类的结果。组内同质（组内知识点的教学结构高度相似）、组间异质（不同组的教学结构有本质差异）。

### 为什么环节四是"反馈话术"而不是"打分"

教育场景中，反馈比打分重要得多。打分是"对/错"的二元判断——学生看到"错"但不知道为什么错。反馈是"你在这一步做了这个，正确/不正确的点是……"——学生知道下一步怎么改进。骨架约束"错误必须给具体错因指向"就是这条设计原则的工程实现。

### 维度交互矩阵

维度一（生成能力）和维度二（生成效率）高度耦合：

| 交互场景 | 影响 | 仲裁规则 |
|---------|------|---------|
| 维度一阶段 2（积木化）提升维度二（效率） | 积木可用→减少 LLM 调用→维度二自然达标 | 验证步骤：(a)每月检查维度二阶段2的三个触发条件——维度一阶段2是否完成(积木沉淀已启动≥30天)、LLM调用次数是否>2、每类骨架积木数是否≥2。注: 后两个条件已在维度二自身触发条件中定义——此处为月度回顾性核查，非实时判定；(b)若维度一阶段2未完成则维度二不可判定就绪，等待维度一；(c)若维度一已完成但LLM调用次数/积木数不满足，检查是否因积木类型分布不均（某类骨架积木数为0）导致调用量未下降——若是，触发维度一积木分布约束告警；(d)三个条件全部满足后方可判定维度二阶段2就绪 |
| 维度二阶段 2（积木优先）可能降低个性化质量 | 高频组合预计算→个性化维度被锁定→新画像特征无法影响预计算结果 | 判断 rule：预计算结果保留 7 天，7 天后必须重新计算——确保新画像数据被纳入。验证：每月最后一周期对比新计算 vs 预计算结果的个性化质量差异 |
| 维度一自动变异后，积木标签可能过时 | 积木变异后标签（如"适合美食偏好"）可能不再准确 | 每次变异后触发积木标签重新验证——用最近 100 次该积木的使用数据，检查标签准确率 > 80%<!-- AI-MUTABLE: 标签准确率阈值, type=float, range=[0.70, 0.90] -->。不达标→积木标签重置，重新积累效果数据 |
| 维度一退化（回退到 D）同时影响维度二 | 积木变异停止→效率数据停止更新。与维度一路线图§1 退化降级规则联动：触发维度一阶段 3→2 回退（effectScore 连续 100 堂中 ≤0 比例>30%）时，维度二同步从阶段 2 退回阶段 1（关闭积木优先→纯 LLM 生成） | 同时退化：维度一退化→维度二也退化到对应阶段。具体映射：维度一 3→2 → 维度二 2→1；维度一 2→1 → 维度二 2→1。预计算仍可用，但不新增预计算组合 |

### 退化组合场景枚举

当维度一和维度二同时处于退化状态时，积木调用优先级如下：

| 维度一状态 | 维度二状态 | 积木调用行为 | 说明 |
|-----------|-----------|------------|------|
| 阶段2课堂级降级 | 阶段2正常 | 积木继续使用(不新增) + 拼装降级(无画像匹配) | 维度一降级但维度二未被动退化 |
| 阶段2系统级退化(2→1) | 阶段2→1被动退化 | 积木保留但不被调用 | 维度二被动退化优先——关闭积木优先机制 |
| 阶段3→2退化 | 阶段2→1被动退化 | 积木保留但不被调用 | 维度一退到阶段2→维度二同步退到阶段1 |
| 阶段3→2→1连续退化 | 阶段1 | 所有积木冻结 | 全系统回退通用骨架 |
| 阶段1(正常) | 阶段1(正常) | 无积木可用 | Phase 1初始状态 |

仲裁规则: 维度二被动退化优先级始终高于维度一的积木继承规则。即: 无论维度一处于什么状态，只要维度二触发了被动退化(2→1)，积木均不被调用。

### 信用分体系

课堂生成引擎的信用分独立核算，初始值 **50**。

**三子分体系**：

| 子分 | 权重 | 上升条件 | 下降条件 |
|------|------|---------|---------|
| 安全分 | 40% | 连续 100 堂课未触发底线信号<!-- AI-MUTABLE: 安全分上升连续次数, type=int, range=[50, 200] --> | 触发底线信号→即时清零 |
| 效果分 | 35% | 个性化课堂 effectScore 均值 > 通用课堂 effectScore 均值 | 连续 50 堂课个性化效果低于通用效果 |
| 进化分 | 25% | 积木沉淀成功率 / 变异成功率 / 自学规则正向率 | 变异成功率 < 30% 且连续 100 次<!-- AI-MUTABLE: 进化分下降阈值, type=int, range=[50, 150] --> |

### 五步执行流程

1. **提案**：AI 或教研人员提出进化方案（新增骨架/修改钩子表/新积木变异规则）
2. **预检**：跑北极星三轮检查（底线/负/正信号）+ 信用分门槛检查
3. **影子模式**：新方案在后台并行运行（不实际用于课堂），shadowEffectScore 与当前方案对比 30 天。shadowEffectScore 高于当前至少 5%<!-- AI-MUTABLE: 影子模式提升阈值, type=float, range=[0.03, 0.10] --> 且无底线信号→进入 A/B
4. **A/B 测试**：5% 学生进实验组用新方案<!-- AI-MUTABLE: A/B实验组比例, type=float, range=[0.03, 0.10] -->，其余用现有方案。分组随机分层（按年级+骨架类型均匀分布）。p < 0.05，最小样本量 = 30<!-- AI-MUTABLE: A/B最小样本量, type=int, range=[20, 50] -->。实验组 effectScore 必须高于对照组且底线信号无恶化
5. **全量上线**：A/B 通过 + 信用分 ≥ 50<!-- AI-MUTABLE: 全量上线信用分阈值, type=int, range=[40, 60] --> → 全量上线。上线后持续监控 30 天，如 effectScore 下降→自动回滚

### 安全执行管道（三层）

**第一层 — 影子模式**：新生成逻辑在后台并行生成课堂脚本，不实际推送给学生。shadowEffectScore 对比 30 天。数据脱敏：影子模式期间学生的真实答题不流向新逻辑——新逻辑只用历史脱敏数据做模拟评估。

**第二层 — A/B 测试**：5% 学生进实验组。分组随机分层（按年级+骨架类型均匀分布）。底线信号实时监控——实验组触底线→ A/B 立刻终止。

**第三层 — 全量上线**：持续监控 30 天。如 effectScore 下降 > 5%<!-- AI-MUTABLE: 回滚effectScore下降阈值, type=float, range=[0.03, 0.10] --> → 自动回滚到上一个稳定版本。

### A/B 测试分组规则

分组维度：年级（G1-G6）+ 骨架类型（7 类）+ 行为模式（fast/slow/error_prone 等）。保证每组至少 30 个样本。

显著性检验：p < 0.05（双尾 t-test）。最小样本量 = 30（Cohen's d ≥ 0.3）。

### 元进化递归上限

课堂生成引擎的进化操作（修改骨架/修改钩子表/修改生成管线）本身可以被进化——这是"进化规则的进化"。

递归上限：最多 3 层。第一层：课堂内容的进化（C→D→E）。第二层：进化规则的进化（什么时候触发变异、什么条件淘汰积木）。第三层：进化规则的进化规则的进化（什么时候修改触发条件）。

任何试图进入第 4 层递归的操作→暂停该进化维度的所有操作 30 天<!-- AI-MUTABLE: 递归超限暂停天数, type=int, range=[14, 60] --> + 信用分 -5。

### 内容与代码进化分离

课堂生成引擎的进化操作按性质分为三类，走不同的审批和执行管道。

**内容进化**（教研域）：涉及教学内容的增删改——骨架模板新增/修改、个性化钩子表调整、质检词库修改、积木块内容质量审核。由教研人员在编辑器里操作，走人工审批流——不经过 AI 自动执行。变更流程：教研人员提交→教研组长审核→灰度发布（5%学生，1教学周期）→质量监控→全量或回滚。

**代码进化**（工程域）：涉及生成管线逻辑的修改——LLM 提示词优化、缓存策略调整、积木拼装算法改进、质检规则引擎优化、预计算策略变更。由工程师在代码仓库里操作，走影子→A/B→全量管道。变更流程：代码提交→影子模式30天→A/B测试（5%学生）→显著性检验→全量上线→持续监控30天。

**混合进化**（跨域）：同时涉及教学内容和代码逻辑的变更——例如新增骨架类型（内容：骨架定义由教研写；代码：骨架选型逻辑由工程师写）。双通过要求：内容部分走人工审批流 + 代码部分走影子→A/B→全量管道。两者均通过后方可全量上线。若一方未通过，另一方也不得上线。

| 进化类型 | 操作者 | 审批管道 | 示例 |
|---------|--------|---------|------|
| 内容进化 | 教研人员 | 人工审批→灰度→监控→全量/回滚 | 新增骨架模板、修改激励偏好钩子 |
| 代码进化 | 工程师 | 影子→A/B→全量→30天监控 | 优化 LLM 提示词、调整缓存策略 |
| 混合进化 | 教研+工程师 | 双通过（人工审批 + 影子管道） | 新增骨架类型（骨架定义+选型逻辑） |

### 三级权限分级

信用分的三级权限体系决定了课堂生成引擎的自主权范围：

| 权限等级 | 信用分范围 | 允许的自主操作 | 禁止的操作 |
|---------|-----------|-------------|-----------|
| 高级 | ≥ 50 | 全量上线（A/B通过后）、自动沉淀积木、自动变异积木（方案E）、自学拼装规则、跨学科积木迁移 | 无（但修改不可修改边界和 HUMAN-SIGNED 参数仍需人工签批） |
| 中级 | 20-49 | 自动沉淀积木（方案D）、缓存策略调整、预计算组合更新 | 全量上线（需教研审批）、自动变异积木、自学拼装规则 |
| 低级 | < 20 | 关闭积木自动沉淀；< 10 → 关闭所有个性化，仅用通用骨架 | 所有自主进化操作 |

权限回收规则：
- 信用分从 ≥50 降至 20-49 → 自动降级为中级权限，暂停所有方案E操作
- 信用分从 20-49 降至 <20 → 自动降级为低级权限，暂停积木自动沉淀
- 信用分从 10-19 降至 <10 → 全系统缩回最保守模式（通用骨架）
- 安全分（40%权重子分）清零 → 即时降级至中级，无论总分如何
- 恢复规则：信用分回到阈值以上 + 连续 30 天无底线信号 → 恢复对应权限等级

## 大白话

### 系统职责定义

课堂生成引擎就像"私厨"——S08 策略引擎是"营养师"（告诉你今天要补钙），S09 是"厨师"（把钙片做成你最爱吃的菜）。骨架是菜谱框架——保证这道菜不会做错（不超纲、不教错）。血肉是调味——根据你的口味加不同的料。

### 核心设计

七类骨架就像七种菜系——计算题是炒菜，应用题是炖汤，几何题是烘焙。不同菜系做法完全不同，但每种菜都有固定的步骤。千人千面的秘密不在骨架——骨架保证正确。秘密在血肉——同一个知识点，喜欢游戏的孩子看到"装备分配"，喜欢美食的孩子看到"分披萨"。

### 自我进化路线图

三个阶段就像学做饭：方案 C 是用菜谱（骨架）+ 现场调味（AI 血肉）——先保证能做出来。方案 D 是把做过的成功菜沉淀成预制调料包（积木）——下次直接组合，不用每次都从头做。方案 E 是系统自己学会了什么菜对什么人最合适——不再需要人写规则"快且粗心的孩子要提醒"，系统自己从几万次数据里发现规律。

### 不可修改边界

课堂生成有七个绝对不能碰的东西：第一，不能超纲——教的不能超出教材和课标。第二，不能改 knowledgeCode——改了全系统找不到这个知识点。第三，不能改事件类型名——S08/S12 监听的 lesson_started/lesson_completed 字段结构不能变。第四，不能改学生的真实作答数据——画像依赖真实数据，改数据等于对着镜子演戏。第五，不能跳过质检——质检是最后一道锁，跳过可能把超纲内容推给学生。第六，不能触底线信号——超纲/AI 字眼/负面标签/低年级纯文字四条底线任一都不行。第七，不能削减骨架约束——如"开场必须打招呼+激活前置知识+告知目标"少一个教学就不完整。

### 接口约定

课堂生成引擎吃掉多个系统的数据——知识库（S01）+图谱（S02）+错因定义（S03）+策略包话术（S04）+门禁状态（S05）+分级标准（S06）+诊断结果（S07）+策略方子（S08）+教材题库（S13）+画像数据（S15-S17），产出一份课堂脚本 JSON。前端拿到这份 JSON 直接渲染就行，不需要再调 AI。

### 自我进化执行方法

进化的每一步都要先"试跑不看脸"（影子模式跑 30 天），再用 5% 学生做实验，最后才全量上线。上线后还有 30 天监控窗口，效果一掉就自动回滚。

### 异常进化处置

如果生成的内容连续被质检拦截，就该骨架类型暂停个性化——先用最保守的通用模板，等查明原因再恢复。

### 应急熔断

有三档：轻量级=单次生成被质检挡住（只拦这次不拦下次）。子系统=某种骨架类型连续出问题（该骨架暂停个性化 7 天，其他骨架正常）。全系统=多条底线信号同时触发（全停所有生成，已在课堂上的学生收到通用模板，新请求排队等待）。

### 进化信用分

信用分就像"驾驶分"——触发底线信号一次清零，恢复需要连续不出事的记录。分低了就减权限——低于 20 分不能自动沉淀积木，低于 10 分全系统缩回最保守模式。

### 内容与代码进化分离

内容进化（骨架模板修改、个性化钩子表调整）由教研人员在编辑器里操作，走人工审批流——不经过 AI。代码进化（生成管线优化、缓存策略调整、积木拼装算法改进）由工程师在代码仓库里操作，走影子→A/B→全量管道。
