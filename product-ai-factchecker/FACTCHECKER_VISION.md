## 一、为什么需要这个产品

### 1.1 核心矛盾

AI 已经普及到每一个普通人的生活中——查资料、写报告、做攻略、问健康建议。但 AI 会犯错，而绝大多数普通人没有能力识别这些错误。

- 我妈用 AI 查养生知识，AI 编了一个不存在的「哈佛研究」。
- 大学生用 AI 写论文，AI 引用了一篇从未发表过的学术论文。
- 运营用 AI 做竞品分析，AI 把两家完全不相关的公司的数据拼在一起。

**AI 的能力已经超过了大多数用户的辨别能力。** 这是一个信息安全的缺口，也是一个巨大的市场。

### 1.2 为什么现在市场上没有好方案

大模型厂商提供了「自己检查自己」的纠错——这就像一个学生给自己的试卷打分。他能发现一些明显错误，但最致命的思维盲区，他自己看不见。

市面上现有的工具，要么是给开发者用的 API 调试器，要么是给极客用的命令行。**没有一个产品是为「普通人用 AI」场景设计的纠错工具。**

### 1.3 芽芽的独特位置

芽芽做纠错，不是「顺便做一下」，而是它天生就该做这件事。

芽芽诊断的不是「这句话对还是错」，而是「这句话犯的是什么错」——逻辑跳跃、数据编造、概念混淆、引用造假。这需要一套深度错因分类体系，而芽芽已经在 K12 领域打磨了 86 种错因分类、概率假设集、反证优先规则、验证动作库。

换一个诊断对象——从「孩子的数学答案」换成「AI 输出的文本」——诊断内核完全不变。只换输出话术。

## 二、产品定义

### 2.1 产品定位

**芽芽事实之眼是一个独立于所有大模型厂商的第三方 AI 纠错智能体。**

它的唯一使命：确保人类从 AI 那里得到的每一个信息，都是真实、可靠、可追溯的。

它不是「AI 的纠错功能」。它是「站在 AI 和人类之间的独立质检」。

### 2.2 一句话

> 用 AI，也要懂 AI。芽芽帮你看。

### 2.3 产品不是

- 不是浏览器（不替代 Chrome）
- 不是搜索引擎（不替代 Google）
- 不是 AI 聊天工具（不替代 DeepSeek/GPT）
- 不是大模型的附属功能
- 不是给极客用的命令行

### 2.4 产品是

- AI 输出的事实核查官
- 普通人能看懂的「人话版」诊断报告
- 可嵌入任何 AIGC 场景的质检层
- 培养全民 AI 素养的训练工具
- 独立于所有大模型厂商的信任锚点

## 三、核心能力：人话版的诊断报告

### 3.1 诊断维度

芽芽事实之眼诊断 AI 输出时，检查以下维度：

| 维度 | 检查内容 | 示例 |
|------|----------|------|
| 事实可验证性 | AI 说的数据、事实有没有可靠来源支持 | 检查数据是否能在公开可信源中找到 |
| 逻辑完整性 | 推理链条有没有跳步、因果倒置、循环论证 | 检查从 A 到 B 到 C 每一步是否都有 |
| 引用真实性 | AI 引用的论文、研究、文献是否真实存在 | 反向搜索确认引用是否可查到 |
| 概念准确性 | 术语使用是否正确，有没有张冠李戴 | 检查术语定义与标准定义是否一致 |
| 时效性 | 信息是否过时（政策、法律、医学等时效敏感领域） | 检查是否存在更新版本的权威信息 |
| 偏见检测 | 表述是否存在隐藏的立场偏向或选择性呈现 | 检查正反两面信息是否都有提及 |

### 3.2 输出格式

芽芽的输出不是「PASS/FAIL」，而是一份普通人能秒懂的诊断报告：

---

```
🔍 芽芽检查了这段话，发现 2 个需要注意的地方：

⚠️ 数据来源可疑（置信度 85%）
这段话提到「2025 年市场占有率增长 30%」。
芽芽在多个公开可信数据源中没有找到支撑这个数字的证据。
这可能是 AI 编造的数据。建议在使用前自行验证。

⚠️ 逻辑链条不完整（置信度 72%）
这段话的结论是「因为 A，所以 C」，
但它跳过了中间最关键的一步「B → C 的推导」。
这个逻辑是不完整的，基于它的判断可能有偏差。

✅ 其余内容芽芽没有发现问题。

---
📊 整体可信度评分：中等（6.2/10）
建议：数据部分请不要直接使用，需先核实来源。
```

### 3.3 核心差异：诊断 vs 判断

大厂纠错：❌ 这句话是错的。

芽芽纠错：⚠️ 这句话在「数据来源」维度有问题（置信度 85%），错误类型是「编造统计数字」。这个 AI 在涉及数字时经常犯这种错误（历史数据显示该 AI 在统计数字上错误率 23%）。

**芽芽告诉你的不是「对还是错」，而是「哪里可疑、为什么可疑、多可疑」。**

## 四、三种产品形态

### 4.1 浏览器插件（C 端主入口）

**形态**：Chrome / Edge 插件

**使用方式**：
- 在任何网页上选中 AI 生成的文字 → 右键 → 「芽芽查」
- 芽芽弹出一个侧边栏，显示诊断报告
- 3 秒内出结果

**免费层**：每天 20 次检查，基础报告（事实+逻辑+引用三项）

**付费层**：月度/年度会员，不限次数 + 六维全量诊断 + 深度报告（含可替代可信来源推荐）

**目标用户**：普通 AI 用户——我妈查养生知识、大学生写论文、上班族做报告、自媒体做内容核查

### 4.2 API / MCP 服务（B 端）

**形态**：REST API + MCP Server

**使用方式**：
- 其他 AI 应用集成芽芽的质检能力
- 用户在前端用 AI 生成内容后，后端静默调用芽芽 API
- 芽芽返回风险评级 + 诊断报告，前端按需展示

**计费**：按调用量（每千次 / 万次 / 百万次阶梯定价）

**目标客户**：任何想对自己的 AI 输出做质检的 AI 产品——AI 写作工具、AI 客服、AI 教育产品、AI 搜索

### 4.3 终端命令行（开发者 / 极客）

**形态**：CLI 工具 `yaya-check`

```bash
# 检查一段文本
yaya-check --text "AI 生成的内容" 

# 检查文件
yaya-check --file report.md

# 输出 JSON 供其他工具消费
yaya-check --text "..." --format json
```

**目标用户**：开发者、技术写作者、CI/CD 流程中的自动化质检

## 五、商业模式

### 5.1 C 端订阅

| 层级 | 价格 | 内容 |
|------|------|------|
| 免费 | 0 | 每天 20 次，三维诊断（事实+逻辑+引用） |
| 个人版 | 待定 | 不限次数，六维全量诊断，深度报告 |
| 家庭版 | 待定 | 最多 5 个家庭成员共享 |

### 5.2 B 端 API

按调用量阶梯定价。市场参照：新闻/事实核查 API 的市场价一般在 $0.001-0.01/次。

### 5.3 企业私有化部署

面向金融、法律、医疗、新闻等高合规行业。一次性部署费 + 年度维护费。提供行业专属诊断模块。

### 5.4 为什么商业模式成立

- 不是「可能付费」，是「必须付费」——AI 输出不可信是刚需痛点
- 高频使用——用户每次用 AI 都可能需要检查
- 规模化效应——诊断引擎成本随规模摊薄，边际成本趋近于零
- 网络效应——跨模型错题本越大，诊断越准，付费意愿越强

## 六、竞争壁垒：为什么大厂追不上

### 6.1 裁判和运动员不能是同一人

大模型厂商的纠错是「自己检查自己」。真正有效的纠错，必须来自一个独立于被检查模型之外的系统。芽芽不属于任何一家大模型厂商，它的唯一使命就是纠错——这是角色壁垒，不是技术壁垒。

### 6.2 跨模型错题本

芽芽可以同时诊断 DeepSeek、GPT、Claude 等所有主流模型的输出。它记录的不是「一个模型会怎么错」，而是「所有 AI 都在什么类型上犯错」。这是任何单一厂商永远拿不到的上帝视角数据。

### 6.3 深度错因体系

芽芽有 86 种错因分类、概率假设集、反证优先规则、验证动作库——这套东西大厂不会为一个纠错功能从头建。而芽芽已经在 K12 场景里打磨了十几个版本。

### 6.4 信任的数学本质

当一个检查者与被检查者共享任何利益关系时，用户对检查结果的信任度自动打折。这是信息论的基本规律。大厂的纠错永远是打折的。芽芽永远不打折。

### 6.5 时间壁垒

诊断能力只能靠真实数据积累。你越早开始诊断，你越准。后来者可以抄你的功能，抄不了你过去几年积累的所有诊断数据。

## 七、与 K12 教学引擎的关系

### 7.1 共享诊断内核

事实之眼和 K12 教学引擎共享同一套诊断引擎内核：

- **07 号错因分类体系**：86 种错因的诊断逻辑
- **11 号诊断引擎**：初筛→匹配→验证→溯源的四步流水线
- **验证动作库**：每种错因对应的验证题/检查步骤

不同的是：

| 维度 | K12 教学引擎 | 事实之眼 |
|------|-------------|----------|
| 诊断对象 | 孩子的数学答案 | AI 输出的文本 |
| 诊断粒度 | 知识点级别的错因 | 语句/段落级别的错误 |
| 输出话术 | 教育术语（概念混淆、逻辑跳跃） | 人话（数据可疑、逻辑不完整） |
| 验证方式 | 出验证题给学生做 | 搜索可信数据源交叉比对 |
| 用户 | 孩子 + 家长 | 所有 AI 用户 |

### 7.2 独立发展

两条产品线共享内核但独立发展：

- K12 引擎继续深化数学 + 未来扩展语文英语
- 事实之眼扩展通用领域的诊断能力（事实核查、逻辑检查、引用验证）
- 内核升级时双方同步受益（如概率引擎优化、错因体系扩展）

### 7.3 内核共享的具体范围

以下组件在两个产品间完全共享：

- 错因分类体系（`error_taxonomy.json` 的 L1 通用层）
- 概率假设集引擎（乘法概率、min(0.95) 上限、fallback 规则）
- 反证优先原则
- 验证动作生成逻辑

以下组件为事实之眼独有：

- 多源事实核查接口
- 引用真实性验证管道
- 时效性检测规则
- 人话版诊断报告模板

## 八、MVP 范围

### 8.1 Phase 1（可从今天启动）

**诊断维度**：基础事实可验证性 + 基本逻辑完整性

**产品形态**：浏览器插件（Chrome）

**核心体验**：
- 选中文本 → 右键「芽芽查」
- 弹出侧边栏 → 3 秒内显示诊断报告
- 报告含：风险评级（高/中/低）+ 可疑项列表 + 每项置信度

**技术基础**：
- 复用 K12 的 L1 通用错因 10 类，映射为事实核查维度
- 概率引擎直接复用（不用改）
- 需要新开发：联网搜索验证管道、引用真实性检测

### 8.2 不纳入 MVP

- 六维全量诊断（先做三维：事实+逻辑+引用）
- API/MCP 服务（B 端 Phase 2）
- 命令行工具（开发者 Phase 2）
- 领域专业化模块（金融/法律/医疗/新闻 → Phase 3+）
- 跨模型错题本（需要一定用户量积累 → Phase 3）
