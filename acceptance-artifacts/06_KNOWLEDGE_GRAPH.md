# 06 知识图谱创建规则 v2.1.1

## 芽芽助教项目背景

芽芽助教是一个面向 K12 全学段、多学科场景的自我进化 AI 教学适配引擎。它以真实教材知识图谱为骨架、以学习行为为真值、以错因诊断为核心、以策略调度为动作系统、以成长记忆为长期资产。

一句话：孩子用得越久，芽芽越懂他；芽芽教得越久，越知道怎么教他。

**当前建设范围**：MVP 阶段为人教版小学数学 5–6 年级，第二阶段扩展至 1–4 年级。

**数据底座**：
- 394 个知识点（G1-G4: 205 个，G5-G6: 189 个），全量录入
- 86 个错因（L1 通用 10 个 + L2 数学特有 76 个），全量定义
- 1390 条知识点-错因绑定，394 个 KP 全覆盖
- 两层知识图谱（学习路径层 + 诊断依赖层），G1-G6 全部建成
- 题型规格矩阵（260KB）
- 数据质量门禁 data_quality_gate.py（9 铁规全覆盖）

**引擎架构**：诊断引擎（四步：初筛→匹配→验证→溯源）→ 策略引擎（六态调度）→ 家长摘要（中性安全反馈）→ 质量仪表盘（24 指标）。

**文档集**：14 份规划文档（01-14），覆盖产品愿景、开发路线、教材来源、分级规则、覆盖门禁、知识图谱、错因分类、策略包、事件协议、数据模型、诊断引擎、策略引擎、家长摘要、质量仪表盘。

## 本文档背景

这是芽芽助教产品文档集的第 06 号文件，定义两层知识图谱架构：学习路径层（用户可见，单元内按教材顺序串联）和诊断依赖层（引擎专用，跨单元跨年级全量织网）。它回答「学会这个需要先会什么」——是诊断引擎 Step 4 溯源的数据基础。图谱数据存储在 `primary_math_g1_g4_knowledge_graph.json` 和 `primary_math_g5_g6_knowledge_graph.json` 中。

## 1. 知识图谱在芽芽里的角色

知识图谱是芽芽理解"学会什么需要先会什么"的基础设施。它不直接展示给用户，而是被两个子系统消费：

| 消费者 | 用哪个层 | 做什么 |
|--------|----------|--------|
| 前端学习路径展示 | 第一层：学习路径层 | 告诉家长"孩子在学什么，下一个是什么" |
| 诊断引擎 | 第二层：诊断依赖层 | 追查做错的根因，沿依赖链向下溯源 |

---

## 2. 两层架构

### 2.1 第一层：学习路径层（用户可见）

**用途：** 前端展示、学习进度追踪

**规则：**
- 在每个单元内部，按教材教学顺序串成链条
- 不跨单元连线（保持教材的模块边界）
- 年级之间只在"自然衔接"处连一条过渡边（如上一年级最后一单元最后一个节点 → 下一年级第一单元第一个节点）
- 给用户看到的就是"当前年级的这条线上，孩子走到了哪里"

**用户视角示例：**
```
五上·第一单元 小数乘法
  M5S1-0 小数乘整数的计算意义 → M5S1-1 笔算方法 → M5S1-10 简便运算 → ...
五上·第二单元 位置（数对）
  M5S1-15 行列定义 → M5S1-16 数对书写格式 → ...
```

### 2.2 第二层：诊断依赖层（引擎专用）

**用途：** 诊断引擎溯源，追查做错的根因

**规则：**
- 1-6 年级全量织网，所有硬依赖都连上
- 跨单元连线（只要满足硬依赖条件）
- 跨年级连线（从 6 年级一路追到 1 年级）
- 这一层对用户完全不可见，只在引擎内部运转

**诊断依赖示例：**
```
5年级·分数加减法
  ├── 依赖 5年级·通分
  │   ├── 依赖 5年级·最小公倍数
  │   │   └── 依赖 4年级·倍数与因数
  │   │       └── 依赖 3年级·乘法意义
  │   └── 依赖 4年级·分数的基本性质
  │       └── 依赖 3年级·除法意义
  │           └── 依赖 2年级·平均分
  └── 依赖 4年级·同分母分数加减法
      └── 依赖 3年级·分数的初步认识
          └── 依赖 2年级·等分概念
```

> **注**：上例展示的是完整诊断依赖链，仅说明知识点间的硬依赖关系。诊断引擎实际溯源最多 3 层（见 11 号文档 §4.4），不会沿链追到底。

## 3. 硬依赖判定规则

### 3.1 硬依赖（连上）

满足以下任一条件即为硬依赖：

| 条件 | 判定方法 | 示例 |
|------|----------|------|
| 定义包含 | B 的定义或公式里直接使用了 A 的概念 | "通分"的定义用了"最小公倍数"→硬依赖 |
| 教材顺序刚性 | 教 B 之前必须先复习 A，不复习就无法展开 B 的教学 | 教"分数加减法"前必须复习"通分"→硬依赖 |
| 操作前置 | 解 B 类题时，必须先执行 A 操作 | 计算"圆的面积"前必须先知道"半径"→硬依赖 |
| 概念层级 | A 是抽象概念，B 是具体应用，没有 A 就无法理解 B | "乘法意义"（抽象）→"小数乘法"（具体应用）→硬依赖 |
| 已知的硬依赖 | 由教研确认的教学依赖关系，无需通过前 4 条规则验证 | "分数乘除法的前置是分数意义"→硬依赖 |

### 3.2 软依赖（不连）

以下情况不连：

| 情况 | 说明 | 示例 |
|------|------|------|
| 启发关系 | 学 A 对学 B 有帮助但不是必须 | "小数乘法"对"小数除法"有帮助，但除法定义不需要乘法 |
| 教材提一嘴 | 教材在教 B 时提了一句"这和我们之前学的 A 有关"，但不复习 A 也能直接讲 B | - |
| 平行关系 | A 和 B 是同一领域的不同应用，没有直接的概念包含关系 | "圆的周长"和"圆的面积"是平行关系 |

### 3.3 判定流程

```
问：A→B 是否在已知硬依赖清单中？
  是 → 硬依赖，连上
  否 → 问：B 的定义或公式里直接包含了 A 吗？
    是 → 硬依赖，连上
    否 → 问：教材教 B 前必须复习 A 吗？
      是 → 硬依赖，连上
      否 → 问：解 B 类题必须先执行 A 操作吗？
        是 → 硬依赖，连上
        否 → 软依赖，不连
```

## 4. 图谱数据结构

### 4.1 数据文件

两套数据集，分别存 1-4 年级和 5-6 年级：

| 文件 | 内容 | 状态 |
|------|------|------|
| `primary_math_g1_g4_knowledge_graph.json` | 1-4 年级 205 个节点 | 已建成，162 节点有前置边，156 节点有后续边 |
| `primary_math_g5_g6_knowledge_graph.json` | 5-6 年级 189 个节点 | 已建成，174 节点有前置边，162 节点有后续边 |

### 4.2 节点字段

每个节点包含以下字段。注：知识图谱文件使用 `commonErrors`，知识点文件（knowledge.json）使用 `commonMistakes`，两者指向同一概念。引擎代码在读取时需兼容两个字段名，Phase 2 可统一为一个字段名。

```json
{
  "knowledgeCode": "M5S1-0",
  "name": "小数乘整数的计算意义",
  "grade": "5",
  "book": "上册",
  "unit": "第一单元 小数乘法",
  "lesson": "1.小数乘整数",
  "curriculumLevel": "A",
  "prerequisites": ["M4S2-12"],
  "next": ["M5S1-1"],
  "questionTypes": ["decimal_mul_int"],
  "commonErrors": ["decimal_point_position_error"],
  "strategyPacks": ["decimal_point_visual_anchor"],
  "masteryRule": "A_individual_v1"
}
```

### 4.3 边的含义

| 字段 | 含义 | 方向 |
|------|------|------|
| `prerequisites` | 要学这个，先得会什么 | 入边（谁指向我） |
| `next` | 学会这个，可以学什么 | 出边（我指向谁） |

**一致性约束：** 如果 A 的 `next` 包含 B，则 B 的 `prerequisites` 必须包含 A。两边的边必须双向一致。

### 4.4 学习路径层 vs 诊断依赖层的实现方式

两份图谱 JSON 共用同一套数据结构。区分两层的方式：

| 层 | 存储位置 | 包含范围 |
|----|----------|----------|
| 学习路径层 | `prerequisites` + `next` 字段 | 单元内教材顺序边 + 年级衔接过渡边 |
| 诊断依赖层 | `prerequisites` + `next` 字段 | 上述全部 + 所有跨单元硬依赖边 + 所有跨年级硬依赖边 |

**也就是说：** `prerequisites` 和 `next` 字段同时承载两层信息。引擎使用全量边做诊断溯源；前端展示时过滤掉跨单元的边，仅展示学习路径。

如果后续需要更干净的分离，可以增加独立的 `diagnosticPrerequisites` 字段。当前先用同一字段实现，保持数据结构简洁。

**已知限制（Phase 2 技术债务）**：共用字段在前端展示时需加载全量边再过滤跨单元边，边数增长后存在查询效率隐患。诊断引擎消费全量边时，单元内教材顺序边（对诊断溯源无意义）会参与溯源步进，可能产生噪音。Phase 2 建议引入 `diagnosticPrerequisites` 字段独立存储诊断依赖层边，`prerequisites`/`next` 仅保留学习路径层。

## 6. 与诊断引擎的关系

- 诊断引擎 Step 4（溯源）依赖第二层诊断依赖层的边
- 溯源从当前出错节点出发，沿 `prerequisites` 边逐层向上追溯
- 最大溯源深度 3 层
- 冷启动期（有数据的节点 < 30）最多溯源 1 层

详见 `11_DIAGNOSIS_ENGINE.md`。

---

## 7. 后续维护规则

- 新增知识点时，必须同时补上所有硬依赖边（单元内 + 跨单元 + 跨年级）
- 修改教材顺序时，同步更新图谱边，并通过双向一致性校验
- 每次修改后运行 `data_quality_gate.py` 确保图谱完整性（R8）
- 图谱边的错误会导致诊断引擎溯源方向错误——这是最严重级别的数据问题，必须零容忍
- 删除知识点时，必须同步清理该节点所有相关的 `prerequisites` 和 `next` 引用，并通过 R8 双向一致性校验。未被清理的悬空引用视为 blocker
