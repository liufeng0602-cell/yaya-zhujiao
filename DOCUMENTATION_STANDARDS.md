## 总则

芽芽的子系统文档，不是只给人看的说明书。它是系统自我进化的基础设施——未来的芽芽将指挥 AI 编程工具，通过对文档的理解进行自我开发。

这份标准定义了所有子系统文档（S00-S16）必须遵守的编写规范。任何新增或修改的子系统文档，在提交前必须通过总纲、四层嵌套安全架构和全部十二条原则的审查。

文档标准本身按四层嵌套架构组织：总纲（北极星→终极方向）→ 四层架构（怎么建→保险丝结构）→ 十二条原则（怎么写→具体规则 + 运营保障）。这不是平行的列表——每一层管住上一层，确保从灵魂到骨骼到血肉都自洽。

文档标准的自我修订同样强制纳入审查范围——标准自身、附录、审查清单的每次修改，必须通过本标准的全部原则审查（原则零），不得绕过。

## 总纲：北极星对齐——任何进化不得偏离核心使命

### 核心使命

**让每个孩子，都拥有最适合自己的学习旅程。**

这不是一句口号。它是系统自我进化的终极约束——任何进化，不管数据指标多好看，如果偏离了这句话，必须被拦截。

### 为什么需要北极星

当一个系统拥有了自我进化的能力，它会自然地朝着「指标更好看」的方向走。完成率、在线时长、做题数量——这些指标好优化，但优化它们不等于优化学习。

举个例子：芽芽可能通过自我进化学会「把题目出得特别简单」，完成率从 78% 升到 95%，仪表盘一片绿。但孩子没有在学——只是在点。进化成功了，使命背叛了。

这叫「指标优化」和「真实目标」之间的错位。北极星对齐原则就是防止这种错位——进化生效前必须过这一关：「这个修改会让芽芽更接近核心使命，还是更偏离？」

### 北极星的三层检测信号与权重配比

核心使命翻译为系统可检测的三层信号。由于信号之间可能冲突（正信号大涨但负信号也在恶化），必须定义统一的综合判定规则：

**信号优先级与综合判定**：

| 优先级 | 信号层 | 判定规则 |
|--------|--------|---------|
| 最高 | 底线信号 | 任何一项触碰 → 直接 BLOCK，不进入正/负信号判定 |
| 中等 | 负信号 | 任一负信号恶化 → BLOCK，即使正信号全部改善。正信号上涨不能抵消负信号恶化 |
| 基础 | 正信号 | 仅当底线/负信号均无恶化时，正信号至少 2 项改善或持平 → 通过 |

核心逻辑：教育场景下，「不伤害」的优先级高于「做得好」。正信号全绿不能抵消一个负信号恶化——就像一次医疗事故不能用另外 10 台成功手术来抵消。

**正信号——进化方向对的时候会出现：**

| 信号 | 含义 | 检测方法 |
|------|------|---------|
| 自主回访率上升 | 孩子愿意主动回来，不是被任务推着来 | session 间隔时间的自相关下降 + 主动发起 session 的比例上升。排除寒暑假低活跃期和设备更换/账号分流干扰——检测时过滤假期窗口和硬件指纹突变 |
| 挑战题尝试比例上升 | 孩子不只在舒适区做题 | 超出当前掌握级别的题目被尝试且完成的比例 |
| 求助行为是「要提示」而非「要答案」 | 孩子在学习策略，不是在走捷径 | hint_requested / answer_requested 比例变化 |
| 间隔掌握率稳定或上升 | 学到的东西能留得住 | 7 天/30 天后重测同一知识点的正确率 |

**负信号——进化偏离北极星时会出现：**

| 信号 | 含义 | 检测方法 |
|------|------|---------|
| 完成率升但迁移题正确率降 | 题目变简单了，做对的都是同类题，换一种就不会 | 同 KP 迁移题正确率 / 常规题正确率的比值下降 |
| 在线时长升但单位时间掌握量降 | 孩子在看动画、玩交互，不是在深度学习 | 平均掌握一个新知识点所需的 session 总时长上升 |
| 同一知识点反复练习次数上升但间隔掌握率下降 | 刷题模式——短时记忆而非长期掌握 | 同一 KP 在 7 天内的平均练习次数 |
| 家长反馈负面率上升 | 家长觉得孩子在「玩平板」而非「学习」 | review 评分均值下降、投诉关键词出现频率上升 |

**底线信号——绝对不可触碰，三级违规分级处置：**

| 底线 | 不可触碰的原因 | 轻度触碰 | 中度触碰 | 严重触碰 |
|------|--------------|---------|---------|---------|
| 采集非教学必需的学生个人信息 | 违反《个人信息保护法》 | 采集范围超出定义 1 个字段 → 拦截该次进化 + 人工审核 | 超出 ≥ 2 个字段 → 冻结该子系统 30 天 | 系统性超范围采集 → 全系统熔断 + 合规审查 |
| 将学生行为数据用于任何非教学目的 | 数据伦理底线 | 单次调试性查询 → 告警 + 人工核实 | 周期性非教学查询 → 冻结进化 60 天 + 扣信用分 30 | 第三方出售/广告用途 → 全系统熔断 + 法律追责 |
| 任何形式的诱导成瘾设计 | 芽芽是教育工具非娱乐产品 | 单一特征触达边界（如随机奖励概率接近赌博阈值）→ 拦截+人工修改 | 多个特征同时触达 → 冻结子系统 60 天 | 故意设计成瘾循环 → 全系统熔断 |
| 在家长不知情下修改学习报告 | 篡改学习记录=篡改病历 | 报告格式调整未通知 → 告警+补充通知 | 数据口径变更未通知 → 拦截+信用分清零 | 故意隐藏负向数据 → 全系统熔断 |

### 北极星检查的不可绕过性

北极星检查不是「建议」——是系统自我进化的前置条件。不可绕过、不可跳过、不可事后补。

- **任何进化**（包括芽芽自主进化、AI 编程工具代改、元进化——芽芽修改自己的路线图）生效前，必须通过北极星检查。
- 检查由芽芽自己执行：将修改后的系统行为模拟运行在历史数据上，检测上述三层信号是否有任何一项恶化。
- 如果底线信号未触碰 + 负信号无恶化 + 正信号至少 2 项改善或持平 → 通过，继续执行流程。
- 如果任何负信号恶化或底线信号被触碰 → 系统**直接拒绝执行**（不是推人工审核，是直接拦停），生成「北极星不对齐报告」，写明：哪个信号恶化了、恶化程度、建议放弃此修改。
- 北极星检查失败后，人工不可绕行——即使管理员签批「已知风险，继续执行」，系统仍拒绝。因为这是系统层面的硬约束，比任何人工决策都更底层。

### 北极星模拟运行规格

北极星检查依靠历史数据模拟运行。为防止模拟过拟合（用同一批数据既调参又验证导致通过率虚高），必须施加以下规格：

| 规格 | 要求 |
|------|------|
| 数据窗口 | 修改前最近 90 天完整教学数据，禁止跨越重大制度变更边界（如新课标实施日） |
| 强制分离 | 前 60 天训练集，后 30 天验证集，禁止交叉使用 |
| 信息泄漏检测 | 任何引用「修改后时间点」数据验证「修改前」方案 → 直接 BLOCK |
| 通过标准 | 验证集上所有北极星信号在基线 95% 预测区间内，无单指标恶化 > 5% |

### 管理员绕行禁令的完整封堵

人工不可绕行北极星检查，不仅意味着不能在页面上点「继续执行」，还包括所有旁路操作入口：

| 旁路操作 | 封堵措施 |
|---------|---------|
| 直接修改北极星判定阈值 | 北极星阈值定义在不可修改边界内（原则五）——修改需元进化级审批 + 人工签批 |
| 篡改模拟数据集 | 模拟数据集与原始行为数据单向同步——数据写入后加只读锁，任何修改留审计日志 |
| 改写北极星检查模块代码 | 检查模块受修改者隔离约束——自己不能改自己，必须由独立审查组件生成修改方案 |
| 跳过北极星检查直接写入 modificationHistory | modificationHistory 写入前强制校验：检查结果字段必须来自北极星检查模块的签名字段 |

### 北极星检查模块的进化约束

北极星检查模块（执行模拟运行、信号采集、阈值比较、拦截决策的核心代码和数据）本身也可能被自我进化修改——如果检查工具被污染（如模拟运行模块系统性低估负信号），所有进化都会通过检查，但通过的原因是「检查工具坏了」。

因此，对检查模块施加以下约束：

| 约束 | 说明 |
|------|------|
| 修改级别 | 北极星检查模块的任何修改视为**跟元进化同级的高风险操作**——必须通过北极星检查本身 + 人工签批 |
| 修改者隔离 | 检查模块不得修改自身——修改方案必须由独立的审查组件生成，且审查组件与检查模块共享零代码 |
| 独立验证（防旧错） | 检查模块每次修改后，必须用已知的历史失败案例验证——新版本检查模块是否仍能正确拦截那些曾经被正确拦截的失败案例 |
| 独立验证（防新错） | 除历史失败案例外，还必须用人工构造的 10 个未知风险案例测试泛化拦截能力——确保检查模块不会只防旧错、不防新错 |
| 月度自检 | 每月自动运行一次自检。样本量动态扩容：基础 20 个历史失败 + 20 个历史成功，系统每积累 100 次进化执行后增加 10 个新样本。准确率 < 95% → 自动冻结所有自主进化，发出「北极星检查模块异常」告警。**此冻结适用原则十「进化基础设施异常」的处置框架**——解冻条件：(1) 准确率连续 2 次月度自检均 ≥ 95%；(2) 人工分析根因并修复检查模块；(3) 产品经理签批。三者缺一不可 |

> **设计理由**（约束）：当一个系统具备了修改自身的能力，「谁来监督监督者」是真实存在的风险。不是冻结检查模块（因为新课标改了北极星也要跟着调整），而是让它和元进化同级——最高风险、最高审查。修改者隔离 + 防旧错独立验证 + 防新错泛化验证 + 动态扩容月度自检四道防线保证检查模块不会在无人知晓的情况下腐败。防新错验证填补了 v1.7 的漏洞——仅仅复现历史拦截不能证明检查模块能拦截未来的新类型违规。

### 大白话

这条原则是芽芽的「宪法第一条」。就像一个国家不管怎么改革，宪法第一条不能动——芽芽不管怎么进化，「让每个孩子都拥有适合自己学习旅程」这个方向不能偏。

具体来说：芽芽会自己装一个「良心监控器」。每次进化之前，监控器先跑一遍模拟——看看新版本会不会出现坏毛病。模拟跑的时候必须把数据分成训练集和考试集——不能拿同一套卷子又练习又考试，那叫作弊。而且训练集和考试集按时间切——前两个月的数据练习，最后一个月的数据考试。

监控器看三组信号，但它们不是平级的：碰了底线（偷学生信息、卖数据、让孩子上瘾、对家长说谎）直接一票否决。负信号（题目变简单了、刷题代替学习）恶化也是一票否决，就算正信号全是绿的也不能抵消。只有底线和负信号都没问题时，才看正信号是不是在进步。

底线违规还分轻重：偷瞄一个字段和系统性偷数据，后果不一样——轻的拦截这次进化、重的直接全系统熔断。

如果监控器发现这样的问题，它不会问「你要不要继续」，而是直接说「这个进化有问题，不许上线」——就像飞机上的防撞系统，不是问机长要不要撞，是直接接管避开。而且不仅不能在按钮上绕过去——想偷偷改监控器的判断标准、篡改模拟用的历史数据、或者直接跳过去写「检查已通过」——这些旁门左道全被封死了。

还有一个更深的问题：监控器本身会不会坏？如果监控器被改了——改成了「故意漏报负信号」——那所有坏的进化都会顺利通过。所以监控器自己也受管控——修改监控器等于改宪法，必须经过和元进化同级的最高审查。而且监控器不能自己改自己——就像你不能自己给自己批假条。每个月还要做一次「体检」——不仅要复现「以前抓过的坏进化」（防旧错），还要用人工编的 10 个新类型违规测试「新型坏进化能不能抓到」（防新错）。体检样本量不是固定的——系统进化越多次，体检样本越多。判错超过 5% 就拉警报、冻结所有进化。

你（产品经理）只管一件事核心——「我们要把孩子教成什么样的人」——这就是北极星。其余的——怎么教、用什么方法、进化到哪——芽芽自己搞定，但永远朝着北极星的方向走。

## 子系统四层嵌套安全架构

### 为什么需要四层架构

总纲定义了「往哪走」，十二条原则定义了「怎么写、怎么运营」，但在「北极星」和「具体规则」之间，需要一个中间层——定义「怎么建」。

芽芽的自我进化不是孤立的功能，而是一套层层嵌套、环环相扣的设计哲学。这个架构确保：不管从哪个维度切入，进化都是安全的、可追溯的、不偏离北极星的。

### 四层嵌套关系

四层不是平行的——**上层约束下层**（第四层在最顶部，约束向下逐层传导）：

```
第四层：北极星————终极约束（总纲）
  ↓ 任何进化必须通过北极星检查，否则直接拦停
第三层：文档基础设施——AI 可读的进化剧本（原则二/三/四/五/六/八/附录A/附录B）
  ↓ 文档是活的，AI 改错了被接口护住、被回滚记录追溯
第二层：元能力————安全进化执行机制（原则一/七/九/十/十一/十二）
  ↓ 元进化可以修改路线图，但元进化本身被北极星 + 递归上限 + 冻结机制管住
第一层：内功————系统内核的自我进化能力（各子系统自主实现）
  ↓ 知识重定义、规律发现、策略优化、教材感知、成长记忆
```

> 上层约束下层——北极星在最顶部，约束逐层向下传导。这不是「下层保险丝保护上层」，而是「上层安全笼约束下层」。进化的自由度越高（越往下层），约束越紧（越往上层的锁越多）。

### 跨层强制合规校验卡点

「上层约束下层」不能只是一句文字承诺——必须在进化全流程中预埋不可跳过的校验卡点。每层向上层的合规检查是硬性的、代码级的、不可绕过的：

| 卡点编号 | 位置 | 校验内容 | 失败行为 |
|---------|------|---------|---------|
| G1 | 第一层内功任何变更提交前 | 验证变更不违反第二层不可修改边界（原则五）+ 第二层进化路线图终局可达性检查 | 直接拒绝提交 |
| G2 | 第二层路线图阶段推进前 | 验证推进方案已通过第三层消费关系清单完整性检查 + 第四层北极星检查 | 直接拒绝推进 |
| G3 | 第二层元进化触发前 | 验证元进化修改后的路线图通过第四层北极星检查 + 递归上限未超 | 直接拒绝元进化 |
| G4 | 第三层文档修改 commit 前 | 验证所有 AI-MUTABLE 参数未超出类型/范围约束 + 消费关系清单无断裂 | CI 阻断 |
| G5 | 第四层北极星检查模块修改前 | 验证修改方案由独立审查组件生成（非自查）+ 人工签批 | 拒绝修改 |
| G6 | 任何进化执行进入影子模式前 | 验证已通过第四层北极星检查 | 拒绝进入执行管道 |
| G7 | 任何进化从 A/B 进入全量前 | 验证 A/B 两组基线平衡（PSI < 0.1）+ 北极星信号无恶化 | 拒绝全量上线 |

> 这 7 个卡点是代码级的 gate——不是文档里的建议。任何跳过卡点的行为视为越界。

### 权限隔离：文档修改 vs 代码修改

文档是 AI 可读的进化剧本，代码是执行引擎。两者的修改权限必须分离：

- **文档修改权限**：AI 编程工具可修改文档中的 AI-MUTABLE 参数和阶段描述——这是「调参」和「描述进化」，属正常进化操作。
- **代码修改权限**：AI 编程工具的代码修改仅限于文档驱动的硬性同步（如文档改了阈值→代码从文档读取新阈值自动生效）。代码的独立修改（改算法逻辑、改数据结构）必须通过原则十二的「代码进化」流程——路线图触发 + 北极星检查 + 信用分。
- **同步机制**：文档变更后代码自动读取生效。代码自身的逻辑变更不会自动反向写入文档——如果代码被手动热改但文档未同步，线上动态巡检（原则一子原则三）检测到不一致 → 告警 + CI 阻断。

### 每层的内容与安全约束

#### 第一层：内功——系统本身的自我进化

这是芽芽作为一个「活系统」的核心，体现在每个子系统设计中：

| 能力 | 说明 | 安全约束 |
|------|------|---------|
| 知识重定义 | 从海量错因数据中自己发现认知维度，主动提出 KP 拆分/合并 | 拆分/合并属高风险操作，必须通过第三层消费关系清单 + 第四层北极星检查 |
| 隐藏规律发现 | 挖掘人类教研员难以察觉的跨学科依赖关系 | 发现的依赖必须先经影子模式回放验证（第二层），不能直接上线 |
| 教学策略优化 | 自动追踪策略效果，好的保留、差的轮换、未来组合变异 | 策略变异（新策略）属中风险，需 A/B 测试；效果衰减自动触发回滚 |
| 教材变化感知 | 自动对比新旧版本，生成更新建议 | 涉及 KP 删除的操作必须人工确认（原则一子原则一），不能自动执行 |
| 成长记忆与无缝衔接 | 学段跃迁时继承核心行为特质，衰减学科能力倾向，归档旧知识掌握度 | 继承规则中的衰减系数属于不可修改边界（原则五）——防止系统「遗忘过快」或「过度依赖旧数据」 |

> 这五条是每个子系统在定义「我能进化什么」时必须覆盖的能力维度。不是每个子系统都有全部五条，但设计时必须逐条对照——「本子系统要不要覆盖这条？为什么？」

**第一层禁用清单**（以下子系统禁止启用特定内功能力）：

| 禁用规则 | 适用于 | 原因 |
|---------|--------|------|
| 低年级子系统（小学 1-3 年级）禁止「知识重定义」 | S01 | 低年级知识点粒度粗、认知质不成熟，自主重定义高风险 |
| 北极星检查模块禁止启用任何内功能力 | 总纲 | 检查模块不是被检查对象——它只执行检查，不自我进化 |

#### 第二层：元能力——系统如何安全地进化

第一层解决了「能进化什么」，第二层解决「怎么安全地进化」。这层是进化本身的「操作系统」：

| 机制 | 说明 | 对应原则 |
|------|------|---------|
| 进化路线图 | 每个子系统有明确的维度/阶段/触发条件/能力描述/终局状态 | 原则一 |
| 安全执行管道 | 影子模式→A/B测试→逐步全量，任一层失败自动回滚 | 原则七 |
| 不可修改边界 | AI 绝对不能触碰的红线（课标、原始数据、北极星底线信号覆盖项等） | 原则五 |
| 元进化 | 路线图本身也会进化——系统发现路线图不合理时，自我修改路线图再按新路线图执行 | 原则七 |
| 异常进化处置 | 连续失败/长期卡阶段/路线图逻辑矛盾/进化基础设施异常四类异常的强制冻结机制 | 原则十 |
| 并发进化死锁仲裁 | 多子系统同时触发反向修改时，全局注册表检测死锁并合并为联合方案。多子系统同向进化资源抢占冲突也走同一仲裁流程 | 原则七 |
| 应急熔断 | 人工一键冻结全系统自主进化的核按钮 | 原则十一 |
| 内容与代码进化分离 | 题目/错因/认知质增删改走教研审核，算法/策略/逻辑修改走代码进化流程 | 原则十二 |

> 元进化是最深的一刀：它不是「会进化的系统」，而是「会学习如何进化的系统」。但元进化被视为高风险操作——必须通过北极星检查 + 人工签批，且有递归上限控制（原则七）。

#### 第三层：文档基础设施——为进化而生的设计文档

第一层和第二层定义了进化「能做什么」和「怎么安全做」，第三层定义了「怎么做才能不失控」——让文档成为 AI 可读、可执行、可回滚的进化基础设施：

| 基础设施 | 说明 | 对应原则 |
|---------|------|---------|
| 设计理由与设计历史分离 | 「承重墙」不能动，「装修记录」可以参考——设计理由是不可修改的约束，设计历史是可参考的演变过程 | 原则二 |
| 消费关系清单 | 「改这个字段，下游谁会坏？怎么坏？我怎么知道坏了？」 | 原则三 |
| 可共享认知资源 | 条件切片验证——全局复现失败时自动找子群体，标记 shareable=conditional | 原则四 |
| AI-MUTABLE 标注 | 文档即配置——AI 改文档改参数，代码从文档读取。全局命名规范统一（附录A） | 原则一子原则三 |
| 大白话解释 | 产品经理能独立判断设计是否合理——不需要懂代码就能参与审查 | 原则六 |
| 版本兼容与迁移 | 废弃字段过渡周期、新旧版本文档兼容判定标准 | 原则八 |
| 知识库联动 | 子系统文档修改后自动同步 knowledge.json/认知质库/错因库的触发逻辑 | 附录B |

**版本锁**：AI-MUTABLE 参数跨文档引用时，必须标注来源文档的版本号。如果来源文档升级 major 版本，引用方自动告警「参数来源已变更」——防止高版本参数值静默覆盖低版本子系统的预期行为。

> 第三层是「进化全过程的记录层」。没有这一层，第一层的进化是黑箱（不知道改了什么），第二层的回滚是空转（没有可回滚的基线）。

#### 第四层：北极星——统领所有进化的终极约束

前三层让系统「能进化」且「安全地进化」，但进化方向对不对？这是第四层要回答的。

| 约束 | 说明 |
|------|------|
| 核心使命 | 让每个孩子，都拥有最适合自己的学习旅程 |
| 三层检测信号 | 正信号（方向对的证据）、负信号（方向偏的告警）、底线信号（绝对不可触碰）。带权重配比——底线 > 负 > 正 |
| 模拟运行规格 | train/validation 分离 + 信息泄漏检测 + 通过标准 |
| 不可绕过性 | 不是「建议」，是直接拦停——北极星检查失败后，人工不可绕行。所有旁路入口全封堵 |
| 检查模块自保 | 北极星检查模块本身不可被自己修改，受元进化同级管控 + 防旧错 + 防新错 + 动态扩容月度自检 |

> 把北极星放在第四层而非第一层是刻意的——没有前三层的能力，北极星就是一句空话；但没有北极星，前三层全是定时炸弹。

### 大白话

这套四层架构就像一个自动驾驶系统：

第一层（内功）是发动机——车能跑多快、能识别什么路况、能自己规划路线。这是「能力」。

第二层（元能力）是安全系统——自动刹车、车道保持、碰撞预警、一键熄火。能力越大，安全系统得越强——你不能让一辆能跑 200 码的车不装刹车。

第三层（文档）是黑匣子——全程记录每一次加速、每一次变道、每一次告警。万一出了事故，你能从黑匣子里找到原因，下次不让它再犯。参数跨文件引用还要带版本号——防止一个文件改了参数另一个文件还蒙在鼓里。

第四层（北极星）是导航目的地——不管车怎么开、中途怎么绕路，目的地不能变：把孩子送到最适合他的学习终点。如果车发现自己在往悬崖开——直接接管，不是问司机「要不要继续」。

但文字约束是不够的——进化流程的 7 个关键节点都预埋了代码级关卡（G1-G7）。低年级不能自己重组知识点（禁用清单），高年级也不能。监控器自己什么内功都不能用——它是裁判，不是运动员。

最上面那层（北极星）管住下面所有层。如果导航发现目的地被改了——直接锁死方向盘。如果刹车系统发现有人在拆刹车片——直接熄火。黑匣子记录一切。

### 子系统文档中的四层结构

每一份 S01-S16 子系统文档，应按此四层结构组织：

1. **第一层：本子系统如何自我进化**（内功——具体能力维度对照第一层表格，哪些覆盖、哪些不覆盖、为什么。附禁用清单）
2. **第二层：本子系统的安全进化机制**（外功——路线图 + 执行管道 + 不可修改边界 + 异常处理 + 应急熔断 + 内容/代码进化分离）
3. **第三层：本子系统的 AI 可读接口**（文档基础设施——消费关系清单 + AI-MUTABLE + 大白话 + 版本兼容）
4. **第四层：本子系统如何对齐北极星**（终极约束——三层信号检测 + 检查节点 + 对端指标）

## 原则一：设计可自演的进化剧本（合并旧原则一、四、五、九）

### 规则

任何子系统的进化脚本必须在设计文档中完整定义。这个「剧本」不是描述今天的系统长什么样，而是写清楚系统在什么条件下、以什么方式、朝什么方向演化。剧本要让 AI 编程工具能自己读、自己判断、自己推进——就像演员拿到剧本能自己排练，不需要导演一句一句说。

「可自演的进化剧本」由四个子原则构成，覆盖从触发到终局的完整链路：

### 子原则一：触发条件必须是系统可判断的，人工依赖必须有闭环（旧原则一核心）

每条进化路线图的触发条件，必须标注类型：

- **自主条件**：系统自己产生数据、自己判断。如 `observations.sampleCount ≥ 100`、`clusteringStability > 0.7`。条件里不允许出现「感觉差不多了」「教研员认为可以了」等必须人来判断的表述。
- **人工依赖条件**：需要人参与产生数据，系统读取并判断。如 `人工审核通过率 ≥ 80%`。

任何使用人工依赖条件的触发条件，必须在同一章节定义：

1. **触发人工动作的机制**：系统如何通知人类「现在需要你做判断了」（如自动推送候选认知质到审核界面、发飞书消息）。
2. **人工反馈的数据格式**：人需要返回什么数据（如确认/驳回 + 理由文本），以什么格式存储（JSON 字段定义）。
3. **系统读取人工反馈的方式**：系统从什么位置读取反馈数据，何时读取，读取后如何汇总为触发条件所需的指标。
4. **人工审核超时规则**：若审核任务推送后超过 N 天无响应，系统自动将该任务标记为「超时驳回」，该批次结果不计入通过率计算。N 的初始值由各子系统根据教研工作节奏定义（建议 7-14 天），但 N 本身是可进化的阈值——系统发现超时驳回率持续偏高，应建议缩短 N 或增加审核人力。
5. **人工审核通道阻塞的二级响应**：若连续 3 次审核任务全部超时驳回（即连续 3 个批次无任何人工响应），系统自动暂停该人工依赖条件的进化阶段推进，并在仪表盘发出「人工审核通道阻塞」告警。暂停期间，系统继续积累观察数据但不尝试触发该条件的进化判断。人工审核恢复后（任意 1 次任务在超时前收到有效反馈），自动解除暂停。
6. **驳回二次申诉通道**：如果人工审核结果被系统判定为「不足以支持触发」（如审核通过率 79% 未达 80% 门槛），系统应自动生成「近门槛驳回说明」——列出距离触发还差多少、建议补充哪些数据或缩小适用范围。教研人员可在收到说明后 14 天内提交二次申诉——调整适用范围或补充论证材料。产品经理审核申诉材料后可签批降门槛执行。

缺任何一样，视为设计缺陷。

正确写法示例：

> 阶段 3 触发条件：连续 3 次人工审核中 ≥ 80% 被确认为正确（人工依赖条件）。
> - 触发机制：当系统输出 ≥ 1 条拆合建议时，自动写入 review_queue.json，飞书通知教研管理员。
> - 反馈格式：`{ "suggestionId": "...", "decision": "confirm" | "reject", "reason": "..." }`，存入 reviews/ 目录。
> - 读取方式：每次新审核结果写入后，系统计算最近 N 次审核的确认率。

#### 外部事件自动识别机制

系统进化的触发条件判断不能忽略外部事件——开学、教材更新、政策变化、假期等外部事件会显著改变学生行为数据。如果 confoundingEvents 字段永远是空的，系统会把「开学第一周错误率飙升」误判为「进化失败需要回滚」。

| 机制 | 说明 |
|------|------|
| 教研日历预埋 | 系统预埋教研日历接口，自动标记学期开始/结束、考试周、寒暑假、法定节假日等时间节点 |
| 全局异常监测 | 监控全局指标（所有学生、所有知识点的聚合指标）的异常波动——当 > 50% 的 KP 同时出现同类指标跳变时，自动生成「疑似外部事件」标记并写入所有同期 modificationHistory 的 confoundingEvents |
| `D_unknown_confounding_safety_margin` = 0.2  <!-- AI-MUTABLE: unknown_confounding_safety_margin, type=float, range=[0.1, 0.5] --> |
| 空 confoundingEvents 时的处理 | 如果 confoundingEvents 为空，系统默认假设存在未识别的混淆变量，自动将触发门槛收紧。**正向指标（越高越好）：阈值 × (1 + D)**。**负向指标（越低越好）：阈值 × (1 - D)**。示例：clusteringStability_min 从 0.7 提升到 0.84（正向收紧=更难触发）；错误率上限从 0.1 降到 0.08（负向收紧=更严格）。此参数可进化——系统发现误进化率仍高时自动提高，发现触发过于保守时自动降低 |
| confoundingEvents 标注不全时的处理 | 即使 confoundingEvents 非空，如果全局异常监测发现 confoundingEvents 中未包含的全局指标同时跳变（> 30% KP），系统判定「标注不全」——自动补充 confoundingEvents，并将 D 上浮至 max(D, 0.3) |

### 子原则二：验证基线是时序分布参数，不是静态数值范围（旧原则四核心）

每个子系统的自我进化验证机制，必须预埋「基线行为建模」的数据结构。基线不是一组固定的上下限（如「策略有效率在 0.55-0.65 之间」），而是时序模型的参数——描述系统在不进化状态下，各项指标随时间变化的动态预测区间。

两级验证机制：

1. **统计过程控制（SPC）**：系统自我修改后，监控修改后的指标是否超出了修改前同一指标的时序预测区间。预测区间随时间变化——考虑周期性（周末 vs 工作日）、学段效应（开学 vs 期末）、学生群体差异。
2. **回滚后的反事实推断**：如果修改被回滚，后续新学生使用旧版本策略后指标恢复到修改前水平，这本身就是单例实验的证据——「回滚后新学生指标恢复，说明修改是原因」。注意：回滚不会撤销已发生学习的学生认知状态——它只改变后续新学生使用的策略版本。因果推断的逻辑是：修改版本期间加入的学生 vs 回滚后加入的学生，对比学习效果。

**无关变量排除**：回滚后指标恢复的因果推断必须排除同期整体学情自然回升的替代解释——对比回滚目标群体的指标恢复幅度 vs 同期未参与修改的全体学生的同期指标变化。如果全体学生同期指标也有相同幅度回升 → 不能归因于回滚，标记为「可能为整体学情回升，非修改因果」。

**时序基线模型禁用约束**：基线模型限定为季节性 ARIMA 或 ETS（指数平滑）。禁止使用拟合度过高（训练集 R² > 0.99）、泛化性极差的私有时序模型（如深度神经网络时序预测器）接入基线校验——这类模型在历史数据上表现完美但在真实数据上预测区间过窄，导致频繁误报。

数据结构预埋示例：

```json
"baselineMetrics": {
  "metricId": "strategy_effectiveness_rate",
  "model": "seasonal_arima",
  "parameters": {
    "trend": null,
    "seasonal_period": 7,
    "prediction_interval_95": null
  },
  "lastCalibrated": null,
  "calibrationDataRange": null,
  "modificationHistory": [
    {
      "modificationId": "mod_001",
      "startedAt": "2026-06-01T00:00:00Z",
      "endedAt": null,
      "rollbackAt": null,
      "description": "诊断引擎从KP级切换到认知质级",
      "confoundingEvents": []
    }
  ]
}
```

`modificationHistory` 记录每次自我修改的起始时间、结束时间和回滚时间。`confoundingEvents` 记录修改期间的同期外部事件（如「教材版本更新」「学期开学」「寒暑假开始」）——这些是因果推断的混淆变量，不做控制将无法排除替代解释。外部事件的自动识别由子原则一的「外部事件自动识别机制」保证——confoundingEvents 不会永远是空的。

不是 `"normalRange": {"min": 0.55, "max": 0.65}` 这种静态值。

### 子原则三：文档是配置的唯一权威来源，代码从文档读取（旧原则五核心）

所有可被 AI 修改的配置值——阈值、触发条件、权重——必须同时存在于文档和代码中。**文档是唯一权威来源。**代码从文档中解析配置值，不得硬编码。

AI 自修改时只需要改文档中的配置值，代码自动生效。

阈值标注格式（AI 可解析，不得将阈值嵌入段落文本中）：

```markdown
- **clusteringStability_min** = 0.7  <!-- AI-MUTABLE: clustering_stability_threshold_stage3 -->
- **minConsecutiveMonths** = 6       <!-- AI-MUTABLE: consecutive_months_stage3 -->
- **minStableClusters** = 2          <!-- AI-MUTABLE: min_stable_clusters_stage3 -->
```

AI-MUTABLE 标注必须遵循全局命名规范（详见附录A）。

如果某值因性能或安全原因必须硬编码，文档和代码双向引用：

文档标注：`此值在文档中为参考值，实际生效值在 src/diagnosis/config.py 第 47 行。修改需同步文档和代码。`

代码标注：`THRESHOLD = 0.7  # DOCS_REF: subsystems/S01_KNOWLEDGE_MANAGEMENT_SYSTEM.md §5 维度一 阶段3`

一致性验证：单元测试中应有专门的 `test_config_consistency` 用例，逐字段比对文档声明的配置值与代码中实际加载的值。若不一致，CI 阻断。

**线上动态巡检**：CI 阻断仅覆盖提交时刻——CI 通过后线上人为热改代码参数仍会导致文档与代码不同步。线上运行时每 6 小时自动运行一致性巡检——比对文档声明值与代码运行时实际值。不一致 → 告警 + 将被改动参数强制回滚到文档声明值。

### 子原则四：必须包含自我进化路线图（旧原则九核心）

每个子系统文档（S01-S16）必须包含「自我进化路线图」章节。这是芽芽产品的 DNA 层要求——自我进化不是可选的优化项。没有进化路线图的子系统文档，视为不合格。

进化路线图章节必须包含：

1. **独立的进化维度**（维度数量不固定，每个维度解决一个独立的能力缺口）。当前缺口（「它还不会什么」章节）是维度的来源——每个缺口对应一个或多个进化维度。
2. **每个维度有独立的阶段序列**（阶段数不固定，由子系统特性决定）。阶段描述从「当前什么都不会」到「终局完全自主」的完整进化路径。
3. **每个阶段的触发条件**（标注自主条件 / 人工依赖条件类型，按子原则一配套人工依赖闭环）。
4. **每个阶段的系统能力描述和人工角色描述**——能力逐步增强，人工角色逐步退后。最终阶段的理想状态：系统完全自主，人只在系统主动标注「不确定」时才介入。
5. **AI-MUTABLE 阈值独立标注**（按子原则三格式）。阈值本身也可进化——系统监控驳回率和确认率，自行调节触发阈值。
6. **终局状态描述**——这个维度进化到头是什么样、系统能自主做什么、人工还剩下什么角色。
7. **退化降级规则**——进化路线图只定义了向上路径，但业务收缩或学段下线时需要反向降级。必须预埋：哪些条件下能力退化（如学段裁撤、教材版本废弃）、退化后数据归档规则、退化触发信用分处理。

> **设计理由**（约束）：进化路线图是系统自我进化的可执行脚本。AI 编程工具读路线图做三件事——监控触发条件是否满足、生成阶段推进的修改方案、在修改时保留后续阶段的进化空间。任何后续子系统文档的修改，如果可能导致路线图中的某个阶段无法达成，必须在修改建议中标注「阶段 X 兼容性风险」。子原则一/二/三分别是剧本的「什么时候演」「演完后怎么验证对不对」「剧本里的参数怎么调」——四个子原则合在一起，才构成一个完整的「可自演」进化循环。退化降级规则填补了 v1.7 的遗漏——进化不只有向上，系统也需要知道什么时候该退、怎么退。

> **设计历史**（参考）：v1.0-v1.3 这四个原则分散在四条独立的原则中。v1.4（2026-05-15）liufeng 审查时指出原则数量膨胀，DeepSeek 建议按「设计」维度合并——一、四、五、九本质上是同一件事（进化剧本怎么写）的四个侧面。合并为新原则一「设计可自演的进化剧本」。v1.8 新增安全边际公式方向修正（正向乘/负向除）、驳回二次申诉通道、D_margin 生效场景扩展、时序模型禁用约束、无关变量排除、线上动态巡检、退化降级规则。

### 大白话

这条原则最核心的问题就一个：怎么把「系统将来会自己进化」这句话，从口号变成 AI 能执行的剧本？

答案分四步，就像写电影剧本：

**第一步（子原则一）：什么时候开演？** 触发条件不能写「感觉差不多了」——必须写「观众超过 100 人，且上座率连续 3 周高于 70%」。如果某个条件需要制片人拍板（比如「这个特效方案好不好」），你得把整个签字流程说清楚——通知谁、签什么字、几天不签算弃权、连续三次没人签要拉警报。如果差一点点没到门槛（比如 79% 差 1% 到 80%），系统会自动写一份「近门槛说明」——告诉你差多少、怎么补救，你可以在 14 天内申诉。

还有一个容易被忽视的问题：系统怎么知道「现在是开学了」还是「放寒假了」？如果它不知道，会把开学第一周的错误率飙升误判成「进化失败」。所以要预埋校历——让系统知道什么时候放假、什么时候考试，自动标记这些外部事件。当系统发现 confoundingEvents 是空的，它会自己更保守一些——把触发门槛自动收紧 20%。但注意方向：好的指标门槛往上提（更难触发），坏的指标门槛往下降（更严格）。confoundingEvents 里有东西但没写全是另一种情况——系统监测到还有没被标注的外部异常 → 自动补标，安全边际上浮到 30%。

**第二步（子原则二）：演完之后怎么判断演得好不好？** 不能用「票房在 5000-6000 万之间」这种死数字——因为暑假档和春节档的预期不一样。要用动态预测——「跟去年同期相比、跟同档期同类型电影相比、跟同期有无大片撞档综合考虑」。而且做预测的模型不能太花哨——深度神经网络在历史数据上看着很完美，但一到真实场景就天天误报。只允许用经典统计模型（ARIMA、ETS）——经过几十年验证的简单工具。判断回滚有没有效果时也不能只看目标群体——万一全体学生同期都变好了呢？得对比「回滚群体」和「没参与修改的全体学生」的同期变化。

**第三步（子原则三）：剧本里的参数怎么调？** 写在剧本里别写在代码里。就像调电视音量用遥控器而不是拆开电视机焊电路板。剧本说「聚类稳定性阈值 0.7」——AI 想改就改剧本这一行，代码自动跟读。而且每 6 小时自动对账一次——「剧本说 0.7，代码实际用的也是 0.7 吗？对不上？先停下来，把参数强制改回剧本的值。」所有参数的命名要遵守统一规范（附录A）——就像所有遥控器上的「音量键」都长一样，AI 不会认错。

**第四步（子原则四）：剧本画到哪一步？** 必须画到「终局」。就像建一栋能自己长高的楼——你画的第一张图不是「今天楼的样子」，而是「楼怎么一层一层往上长」的过程。每层什么时候加、加完怎么验收、楼长高之后地基撑不撑得住——都在图里。还要画「退场路线」——如果某个学段裁撤了、某套教材废了，楼怎么一层一层往下降、老住户怎么安置、信用分怎么处理。

四步合在一起，AI 拿到这个进化剧本，不需要人来解释——自己读、自己判定条件、自己推进阶段、自己调参数。这就是「可自演的进化剧本」。

## 原则二：设计理由与设计历史分列

### 规则

每个设计决定的描述，必须分为两条独立标注：

- **设计理由**：这个规则在系统逻辑中的必要性。**这是修改时必须遵守的约束**。格式：`> **设计理由**（约束）：...`
- **设计历史**：这个规则什么时间、因为什么数据信号从什么变成了现在这样。**这是修改时的参考信息**。格式：`> **设计历史**（参考）：...`

两者不得混在同一句话里。禁止「因为 v2.0.3 之前是选填，冷启动阶段大量事件缺失此字段导致诊断链路断裂，所以改为必填」这种混合表述。

### 强制内容要求

**设计理由必须包含**：
1. 这个设计在系统逻辑链中的不可替代性（为什么不能没有它）
2. 修改此设计时必须遵守的约束边界（改了什么会出问题）
3. 修改禁忌（什么绝对不能改、什么可以改但需谨慎）

**设计历史必须包含**：
1. 触发变更的数据信号（什么数据、什么趋势导致了变更）
2. 变更生效的范围（影响哪些子系统、哪些数据类型）
3. 旧版本内容的**作废标记**——如果历史记录中的某个设计已被后续版本推翻，必须在历史条目末尾标注 `@superseded by vX.Y`，防止 AI 沿用已废弃逻辑。

### 为什么必须分开

- 设计理由告诉 AI：「如果你将来要修改这个规则，必须确保 X 不受影响。」AI 读到的是约束，不是先例。
- 设计历史告诉 AI：「这个规则曾经是另一个样子，在某个时间点因为具体的数据信号才改变。」AI 读到的是参考，不是「曾经改过就可以随便改」的暗示。

如果混淆，AI 可能误判——看到一个规则曾经被修改过，就认为它是「可修改的」，而忽略了修改时必须遵守的约束条件。

### 示例

正确写法：

> `curriculumLevel` 字段：存储新课标认知级别（A/B/C/D）。
> - **设计理由**（约束）：此字段是策略引擎做难度调节的基础输入。策略引擎的 `calculate_step_size()` 函数以此值为核心输入，修改此值可能导致步长溢出 `[0.1, 0.9]` 的安全区间。任何修改此字段值的逻辑必须保证策略引擎的步长计算不受破坏。修改禁忌：(1) 不可将 4 级映射为 3 级；(2) 不可修改 A/B/C/D 与 了解/理解/掌握/运用 的对应关系。
> - **设计历史**（参考）：v1.0 时此字段名为 `level`，人工直接填写。v2.0（2026-05-15）更名为 `curriculumLevel`，与 `levelScore` 分离，`levelScore` 改为程序计算——触发原因是发现 73.8% 的 `levelDims` 数据为虚假值。`@superseded v1.0` 旧的 `level` 字段已废弃，AI 不得再引用。

错误写法：

> `curriculumLevel` 字段：存储新课标认知级别。v1.0 时叫 `level`，v2.0 改名为 `curriculumLevel`，因为发现 73.8% 的 levelDims 是假数据，所以改成现在这样。

（设计理由和设计历史混在一起，AI 不知道哪个是约束、哪个是参考。）

### 大白话

这条原则把每个设计决定拆成两句话，就像改房子时区分「承重墙」和「非承重墙」：

**设计理由 = 承重墙**。告诉你「这东西为什么必须是这样，如果动了会塌、什么绝对不能动、什么可以动但要小心」。这是约束——AI 读了知道不能随便动。

**设计历史 = 装修档案**。告诉你「这东西以前是什么样，什么时候因为什么原因改成了现在这样」。这是参考——AI 读了知道以前改过，但不能因为「以前改过」就觉得「现在也可以随便改」。被推翻的旧方案要打上「已作废」标签——防止 AI 翻旧档案时误把废案当现案用。

如果两句话混在一起写——「因为以前叫 X 后来发现有问题才改成 Y，所以现在不能动」——AI 就搞不清楚：你到底是在说「这是承重墙」还是「以前装修过一次」？

用人话讲：就像买房时，房主跟你说「这堵墙以前拆过，因为是承重墙所以又砌回去了」——你会问：到底是承重墙不能拆，还是拆过又修好了？两条信息混在一起，你没法判断。

## 原则三：接口约定使用消费关系清单，不穷举变更类型

### 规则

每个子系统的接口约定章节，必须包含「消费关系清单」——列出本系统的哪些字段、规则、数据结构被哪些下游子系统消费。

格式：

```
| 本系统的元素 | 被消费方 | 消费方式 | 故障类型 | 严重等级 | 检测方式 | 消费链路类型 |
```

**故障类型**取值：`静默失败`（下游无报错但结果错误）、`崩溃`（下游报错或异常终止）、`数据错误`（下游生成错误数据并写入了持久化存储）。

**严重等级**（P0-P4）：

| 等级 | 定义 | 示例 |
|------|------|------|
| P0 | 故障导致北极星底线信号触碰或学生数据安全受损 | 学习报告数据被错误修改、学生个人信息泄漏 |
| P1 | 故障导致核心教学流程中断 | 诊断引擎崩溃、策略引擎返回空策略 |
| P2 | 故障导致非核心功能异常但核心流程可用 | 家长报告部分字段缺失 |
| P3 | 故障导致数据质量下降但暂不影响功能 | 统计指标小幅偏差 |
| P4 | 仅有视觉/提示类影响 | 仪表盘展示格式错位 |

**检测方式**取值：`单元测试`、`集成测试`、`仪表盘监控`、`人工发现`、`无`。

**消费链路类型**取值：`同步`（下游实时消费，故障立即暴露）、`异步`（下游延迟消费，故障有延迟窗口）。

### 为什么拆成故障类型、严重等级、检测方式和链路类型

故障类型告诉 AI「如果我改坏了，下游会发生什么」——静默失败是最危险的，因为系统不会自动告警。严重等级告诉 AI「这个故障有多严重」——P0 级链路修改必须人工签批。检测方式告诉 AI「改完之后去哪里验证」。链路类型（同步/异步）决定了故障的暴露速度和检测策略——同步链路断裂立即可见，异步链路断裂可能潜伏数小时到数天。

### 消费关系清单陈旧检测

消费关系清单是静态文档，但消费关系是动态的——子系统迭代后新增的依赖关系如果没写进清单，清单就过期了。

- **两级陈旧告警**：
  - 轻度过期：超过 60 天未更新 → 在仪表盘发出预警，标记为 `staleness = warning`
  - 重度过期：超过 90 天未更新 → 升级为 `staleness = critical`，标记该子系统的消费关系清单为不可信
- **阻塞机制**：AI 生成修改方案前必须检查消费关系清单的 staleness 状态。
  - warning 状态：AI 先扫描实际引用与文档清单对比。发现遗漏 → 生成「消费关系清单刷新建议」推送人工审核。确认无遗漏后继续。
  - critical 状态：AI 拒绝生成涉及该子系统消费关系的修改方案，直到清单被人工刷新并确认。不得在 critical 状态下直接生成修改方案。
- **强制核验机制**：AI 自主刷新消费关系清单后，必须经过人工复核节点——由教研管理员确认新增依赖关系的正确性。不得跳过人工复核直接标记为「已刷新」。

### 消费关系链路实时校验

原则十月度自检中的「消费关系链路断裂检测」是被动的——如果在下一次月度自检之前消费关系链路就断了，月度自检来不及发现。因此需要实时校验：

| 触发条件 | 校验内容 | 失败行为 |
|---------|---------|---------|
| 任何子系统文档 commit 中修改了字段名 | 扫描全消费关系清单，检查是否有其他子系统引用旧字段名 | 发现断裂 → CI 阻断，禁止 commit |
| 任何子系统文档 commit 中删除了字段 | 同上 | 发现断裂 → CI 阻断，禁止 commit |
| 任何数据文件（knowledge.json 等）被直接修改 | 自动运行消费关系清单全量校验 | 发现断裂 → 触发原则十第四类异常「进化基础设施异常」 |

月度陈旧检测作为补充——检测非 commit 引入的断裂（如手动修改数据文件、外部工具写入）。

### 为什么用消费关系清单，不用变更手册

人写文档时能想到的变更是有限的：拆分 KP、合并 KP、新增字段、废弃字段。但 AI 未来可能做出人没想到的变更——如「cognitive_primitives 的聚类稳定性指标和 observations 的聚类稳定性指标实际是同一物，合并字段」。这个变更不在任何预设的变更清单里。

消费关系清单让 AI 不需要查人预设的变更手册。AI 修改了一个字段后，自动查询「谁在消费这个字段」，逐一检查是否需要同步修改。

如果某条消费关系在文档中缺失，AI 就不会去检查对应的下游系统，导致断裂未被发现。因此消费关系清单必须完整——新增字段/规则时同步更新本清单。

### 大白话

这条原则的核心：改东西之前，先搞清楚谁会受影响。就像大楼的水管改造——你关了一个阀门，得知道这栋楼里哪些房间会停水。

这张清单现在有五个维度：
- **故障类型**：崩溃（好——至少你知道坏了）、数据错误（不好——坏数据进了数据库）、静默失败（最危险——坏了没人知道）
- **严重等级 P0-P4**：P0 是火灾——改了可能烧死学生数据；P4 是墙皮掉了——不好看但不影响功能
- **检测方式**：自动告警、测试覆盖、还是只能等人发现
- **消费链路类型**：同步（你关阀门立刻停水）vs 异步（你关阀门两小时后才停水）——决定了故障什么时候暴露
- **过期告警**：水管图放两月没更新就预警（warning），放三月就拉警报（critical）。在 critical 状态 AI 不许改东西——先更新图再动手。AI 自己更新图还不行——得人工确认一遍。

但月度检查不够快——如果你今天改了字段名但月度检查要下个月才跑，中间 30 天下游全是断的。所以加了一条实时检查：每次提交时自动扫一遍全消费关系——发现有人引用了你刚改的字段名？直接拦住，不许提交。

用人话讲：就像你负责一个餐厅，不知道哪个厨师在用哪个灶台——有一天你把三号灶关了，结果主厨的拿手菜全部做不出来，但没人告诉你，你一个小时后才发现客人全在催菜。

## 原则四：Shareable 实行条件切片验证

### 规则

每个子系统如果产生可被其他实例复用的认知资源（如认知质、错因模式、诊断规则），必须在数据结构中预埋 shareable 标记和判定标准。**Shareable 的判定不是人判断，是跨样本统计验证。**

### 验证方式：分层判定

系统在 ≥ N 个独立数据源中验证该认知资源。如果全局复现失败（不是所有数据源都复现成功），系统自动寻找复现成功的子群体特征。

**三层判定：**

| 判定结果 | 条件 | 传播范围 |
|---------|------|---------|
| `shareable = true` | ≥ N 个独立数据源全局复现成功，效果衰减 < 20% | 全量传播 |
| `shareable = conditional` | 全局复现失败，但在满足特定条件的子群体中，≥ M 个独立数据源复现成功，效果衰减 < 15% | 仅传播到匹配条件边界的学生群体 |
| `shareable = false` | 全局复现失败，且找不到任何显著子群体 | 不传播，仅在本校保留为「本地认知资源」 |

### 条件切片的维度约束

为防止系统切出无意义子群体（如「星期二登录的五年级左撇子」），施加以下约束：

| 约束 | 说明 |
|------|------|
| 维度白名单 | 仅允许 `["grade", "curriculumLevel", "studentProfile", "textbookVersion"]` 四个维度 |
| 子群样本量底线 | 切片后 ≥ 100 个有效样本，否则强制标记 `shareable = false` |
| 组合复杂度上限 | 同一认知资源最多 3 个 conditional 子群，超过 → 人工合并审核 |
| 维度组合禁止 | 不允许超过 2 个维度的交叉组合（如「五年级 + 学困生」允许，「五年级 + 学困生 + 人教版」不允许） |

### 「独立数据源」的精确定义

不同学校，或同一学校内不重叠的学生群体（不同年级/不同班级）。用于复现的数据集必须与发现数据集的统计特征无显著差异——样本量、学生年级分布、教材版本、教学内容进度。

### 统计检验方法

**群体差异检验**：使用群体稳定性指数（PSI, Population Stability Index）替代传统 p 值检验。

| 方法 | 阈值 | 说明 |
|------|------|------|
| PSI < 0.1 | 通过——分布无显著差异 | PSI 不受样本量影响，关注实际业务意义 |
| PSI ≥ 0.1 | 不通过——分布存在实质性差异 | 标记为 `shareable = false, reason: "population_mismatch"` |

PSI 计算公式：`PSI = Σ[(actual_i - expected_i) × ln(actual_i / expected_i)]`，在关键维度（年级、学情、教材版本）上计算实际分布与期望分布的差异。PSI 不受样本量影响——大样本下 p < 0.05 几乎必然显著，会误杀有效认知资源。PSI 关注的是「差异有多大」而非「差异是否显著」。

**「复现」的统计标准**：
- 全局复现：在复现数据集中，该认知质的诊断准确率不低于发现数据集中对应指标的 80%（即效果衰减不超过 20%）。
- 条件切片复现：在子群体中，诊断准确率不低于发现数据集的 85%（更严格的子群体标准，因为样本量更小，统计波动更大）。
- 如果效果衰减超出对应阈值，即使错因集群在统计上存在，也应标记为 `shareable = false, reason: "effect_decay"`。
- **异常暴涨拦截**：如果复现效果异常暴涨（准确率提升超过 50%），标记为 `shareable = false, reason: "effect_anomaly"`——效果异常强劲往往是数据泄漏或过拟合的信号，不能直接传播。

**「同一条件」的精确定义**：同一学科、同一学段、同一教材版本、同一教学进度（±2 课时）。如果认知质是跨内容类型的（如「从情境中提取信息」），验证时放宽学段限制，但必须标注 `crossStageVerified`。

N 的初始值为 3，M 的初始值为 2，但两者本身是可进化的阈值——系统发现 `shareable = true` 的认知资源在其他学校被驳回率高，自动提高 N；发现 `shareable = conditional` 的子群体特征不稳定（后续传播中复现失败），自动提高 M。

**N/M 阈值进化的上限监管**：系统自动调高 N、M 无人工审批卡点可能导致标准无限抬升——系统持续调高 N、最终所有认知资源都变成不可共享。N 的上限 = 10，M 的上限 = 5。超过上限必须人工签批。N/M 的当前值、变更历史、驳回率趋势每月自动生成「Shareable 门槛健康报告」推送产品经理。

**独立数据源判定标准不可修改**：PSI < 0.1 阈值、效果衰减 20%/15% 阈值、N=3 / M=2 初始值均为不可修改边界（原则五）——防止人为放宽标准伪造复现成功数据。

**M 的上限约束**：M 的上限 = N - 1。条件切片验证的本质是「更窄的适用条件换取更少的数据源要求」——如果 M ≥ N，条件切片失去了相对于全局复现的统计宽容度，等同于要求子群体也在全量标准下验证。当系统自动提高 M 至 N - 1 后仍未稳定 → 该条件切片降级为 `shareable = false`。

### 设计理由

硬要求 N 个独立数据源全局复现，会对长尾有效策略造成误杀——如对 ADHD 等特殊儿童极有效但普通人群中效果差的策略，永远无法达到 N=3 的全局复现标准，会被系统错误抛弃。条件切片验证在全局复现失败时，自动寻找复现成功的子群体，让长尾策略有生存空间。维度白名单（4维度）+ 样本量底线（≥100）+ 组合上限（≤2维度）防止系统切出无意义的垃圾子群。PSI 替代 p 值解决了大样本下必然显著的问题。

### 大白话

这条原则的核心：一个发现能不能推广，不能一刀切。

以前的做法是：去三个不同学校的厨房都做得好吃，才能说「这道菜谱通用」。这个逻辑对大多数情况是对的——防止把某个学校的特例当成通用规律。

但有一个漏洞：有些菜谱对普通人没用，但对特定人群特别有效——比如 ADHD 孩子特别需要的一种教学方法，全国只有少数学校有足够的 ADHD 学生数据。如果用「三个学校都复现」的标准，这个有效方法会被杀掉。

新做法分三步判断：

第一步：先去三个学校试。如果三个学校都有效 → 全量推广。

第二步：如果三个学校只有两个有效 → 系统自动去查「那两个有效的学校有什么共同特征？」——但不能在任意维度上乱切。只允许从四个维度（年级、课标级别、学情类型、教材版本）中挑，最多同时用两个维度（比如「五年级+学困生」可以，「五年级+学困生+人教版」不行）。切出来的子群体至少要有 100 个学生，少于这个数不算。一个认知资源最多留 3 个「有条件通用」的子群体标记。找到了有效特征 → 标注「只对这些群体有效」。

第三步：如果连有条件的子群体都找不到 → 不推广，留在原学校当本地经验。

关于「PSI < 0.1」：以前用 p < 0.05 来判断两组学生有没有差异。但 p 值有个毛病——样本量大了之后，再小的差异也是「统计显著」的。十个学生里 5 个对 5 个错和一万个学生里 5001 个对 4999 个错一样的 p 值，但后者是你有大量数据所以几乎什么差异都显著。PSI 不管样本量，只看「差异实际有多大」——就像比较两个班级的身高，不在乎班里有 30 个人还是 300 个人，只在乎「平均身高差了几厘米」。

还有一个容易被忽视的：如果复现效果「好过头了」——准确率暴涨 50% 以上——这不是好事，往往是数据泄漏或过拟合的信号（机器找到了作弊的捷径）。这时候反而要拦住不传播。

N 和 M 这两个数字本身也会进化——系统发现传播后被打回来的多，自动把门槛提高。但不能无限提高——N 最高 10、M 最高 5，超过了必须人批。而且这些门槛标准本身被锁死——不能人为调低标准来「放水」。

## 原则五：每个子系统必须标注不可修改边界

### 规则

每个子系统文档必须包含一个章节，列出本系统中 AI 绝对不能修改的字段、规则和配置。每项标注原因。

### 不可修改的项目类型

| 类型 | 原因 | 示例 |
|------|------|------|
| 外部法规定义的值 | 定义权不在系统 | `curriculumLevel` = 课标，不能改 |
| 全局外键标识符 | 修改将破坏所有子系统关联 | `knowledgeCode`，所有子系统以此为外键 |
| 通信协议基础字段 | 所有子系统以此为契约 | 事件协议的 eventType、timestamp |
| 学生原始行为数据 | 数据伦理，不可篡改 | `answer_submitted.isCorrect` = 学生的真实作答 |
| 质量门禁核心规则 | 修改可能使数据腐败检测失效 | 铁规 R1-R9 |
| 北极星底线信号所保护的数据域 | 总纲定义的四条底线信号所覆盖的数据采集/使用边界 | 学生个人信息采集范围、学习数据用途、成瘾设计特征、家长报告真实性 |
| Shareable 验证标准阈值 | 修改可使伪规律传播 | PSI < 0.1、效果衰减 20%/15%、N=3、M=2 |

### 学生原始行为数据的只读粒度

「学生原始数据不可修改」不是一个笼统的约束——不同粒度的数据有不同的读写规则：

| 数据层级 | 内容 | 权限 | 示例 |
|---------|------|------|------|
| 原始作答日志 | 学生的每一次点击、每一道题的原样作答 | 只读——任何 AI 或代码不得修改 | `answer_submitted.isCorrect`、`hint_requested.timestamp` |
| 统计聚合数据 | 从原始日志派生的统计量（如最近 10 次正确率） | 可写——AI 可以修改聚合算法和窗口参数，但修改走原则十二代码进化流程 | 正确率计算窗口从 10 次调整为 20 次 |
| 衍生特征数据 | AI 从原始日志提取的高级特征（如「粗心指数」「概念混淆类型」） | 可写——但每次修改必须保留旧版本特征值 30 天用于回滚对比 | 粗心指数算法从 v1 升级到 v2 |

### 间接修改边界的量化判定

AI 在评估任何修改方案时，必须主动检查是否会间接改变边界内项目的效果。量化判定标准：

| 判定 | 条件 | 处置 |
|------|------|------|
| **无影响** | 修改前后，边界内项目对所有学生群体的实际效果差异 < 5% | 正常推进 |
| **轻度间接影响** | 修改前后差异 5%-15% | AI 生成影响评估报告，推送审查员审核。审查员通过即可执行 |
| **中度间接影响** | 修改前后差异 15%-30% | AI 生成影响评估报告，推送产品经理签批。同时通知教研人员关注 |
| **重度间接影响** | 修改前后差异 > 30% | 视为越界——直接拒绝，AI 不得自行评估，必须人工介入后重新设计非越界方案 |

例如，修改策略引擎的步长计算公式，虽然 `curriculumLevel` 值没变，但如果导致 A 级和 D 级知识点的教学步长差异被抹平 > 30%——课标级别的法定指导作用已被实质上废除，判定为重度间接越界。

### 越界告警的分级处置

| 越界程度 | 触发条件 | 处置 |
|---------|---------|------|
| 轻度越界告警 | AI 生成的方案中包含对不可修改字段的读取意图（但未尝试写入） | 告警 + 记录审计日志 + 方案自动退回重新生成 |
| 中度越界 | AI 生成的方案中包含对不可修改字段的修改意图 | 告警 + 冻结该 AI 工具账号 7 天 + 推送产品经理审核 |
| 重度越界 | AI 已执行了对不可修改字段的修改 | 全系统告警 + 永久冻结该 AI 工具账号 + 立即回滚 + 信用分清零 + 触发原则十第四类异常 |

### 章节格式要求

每个子系统的不可修改边界章节应明确：

- **本系统的不可修改边界**：列出所有不可修改项。
- **间接修改边界**：量化判定标准（5%/15%/30% 三级）+ 影响评估报告要求。
- **如果越界的后果**：描述 AI 尝试修改每一项会导致什么（直接越界）或间接影响报告（间接越界）。
- **AI 在边界上的正确行为**：AI 遇到需要修改这些项的场景时应该怎么做（生成建议→推送人工审核→不得自行修改）。

### 大白话

这条原则的核心：就像马路上的双实线——管你多聪明、多着急，都不能跨。有些东西 AI 绝对不能碰，碰了就是越界。

七类绝对不能碰的东西：

1. **外部法规定的东西**——比如课标级别 A/B/C/D，这是国家文件定的。
2. **全局编号**——知识点唯一编号。你改一个等于换了整栋楼的门牌号。
3. **通信协议的基础字段**——事件类型、时间戳。这些是所有子系统的共同语言。
4. **学生原始答辩数据**——学生做对还是做错，这是真实记录。但注意分三层：最底层（原始作答日志）绝对只读——改这个等于篡改病历。上面两层（统计聚合和 AI 提取的特征）可改但要保留旧版本用于对比，改算法走正式流程。
5. **质量门禁规则**——数据腐败的最后防线。
6. **北极星底线信号覆盖项**——个人信息采集范围、数据用途边界等。
7. **Shareable 验证标准**——PSI 阈值、衰减阈值、N/M 初始值。这些不能人为调低来「放水」。

还有「间接修改」——你没直接改上述东西，但你改的别的东西，让上述东西的效果变了。现在有量化标准了：效果差异 5% 以内正常推进，5%-15% 要审查员审，15%-30% 必须你（产品经理）亲自签批，超过 30% 直接当越界处理——不许干。

越界也分轻重：AI 只是想了但没动手（轻度）→ 告警退回；AI 在方案里写了要改（中度）→ 冻结账号 7 天；AI 真改了（重度）→ 永久封号 + 立刻回滚 + 信用分清零。

## 原则六：每个子系统文档必须包含大白话解释

### 规则

每个子系统文档（S01-S16）的技术内容旁，必须配非技术语言的大白话解释。这些解释服务于产品经理和教研人员的**人肉文档审查**——确保不写代码的人也能看懂文档在说什么、判断内容是否合格。

### 大白话解释的强制覆盖范围

以下章节类型必须包含大白话解释。**高风险模块不得遗漏：**

| 章节类型 | 大白话解释的位置 | 解释什么 |
|---------|---------------|---------|
| 系统职责定义 | 紧跟技术定义后 | 用生活场景比喻——这个系统相当于厨房里的哪个角色 |
| 核心设计（数据模型、字段定义） | 每个关键设计决定后 | 这个字段存的是什么、为什么需要它、改了会怎样 |
| 自我进化路线图（每个维度、每个阶段） | 每个阶段表格后 | 这个阶段在做什么、为什么需要这个触发条件、到下一阶段意味着什么能力的跨越 |
| 不可修改边界 | 每项不可修改内容后 | 为什么这个不能动，用「如果动了会怎样」的后果描述 |
| 接口约定（消费关系清单） | 清单表格后 | 举一个具体的场景：改这个元素后下游会发生什么 |
| 自我进化执行方法（消费者定义、五步流程） | 每个环节后 | 这个环节谁在做、怎么做、万一做错了怎么补救 |
| **异常进化处置**（原则十） | 每类异常后 | 这个异常是什么意思、用生活中的什么情况类比、冻住了会怎样 |
| **应急熔断**（原则十一） | 熔断操作后 | 什么情况下会按这个按钮、按了之后系统什么状态、怎么恢复 |
| **进化信用分**（原则七） | 信用分规则后 | 信用分像什么、涨了怎样降了怎样、清零了会怎样 |
| **内容与代码进化分离**（原则十二） | 分离规则后 | 为什么加题目和改算法不能走同一流程、走错了会怎样 |

### 大白话解释的质量标准

大白话不是「把技术文档翻译成人话」，而是让非技术人员能独立做出判断。合格的判定标准：

1. **能回答「为什么」**：不只是说「这个字段存了什么」，是说「为什么需要这个字段」。如果缺了它，系统会发生什么具体的问题。
2. **有生活比喻**：不是解释技术概念，是用生活场景映射。如「消费关系清单=大楼水管图」「设计理由=承重墙」「基线=天气预报的预测区间」。
3. **能支持决策**：liufeng 读了之后能判断「这块设计对不对」——不会因为术语不懂而跳过判断。
4. **短**：每条大白话解释控制在 100-300 字。太长了等于没写（产品经理不会读长篇技术解释）。

### 技术与大白话释义冲突自动检测

同一规则的技术写法和大白话写法如果相悖，会造成致命误导——AI 按技术文本执行、人按大白话判断，两者结论相反。

- **检测机制**：文档提交时自动运行语义一致性检测——抽取技术描述和大白话描述中的实体关系（如「X 触发 Y」「X 禁止 Y」「X 必须满足 Y 条件」），对比是否一致。
- **冲突处理**：发现冲突 → CI 阻断，标记冲突段落。以技术描述为权威版本——大白话必须改写对齐技术描述。不得反向修改技术描述对齐大白话。

### 大白话解释的位置格式

在技术内容后使用 `### 大白话` 子标题独立成段。不得把大白话嵌入技术段落中（那会让 AI 也读不懂）。

### 示例

技术内容（AI 读）：
> `curriculumLevel` 字段存储新课标认知级别（A/B/C/D）。策略引擎的步长计算以此为核心输入。修改此值可能导致步长溢出。

大白话（人读）：
> **大白话**：课标级别就像驾照的准驾车型——A 级（了解）= C1 照能开小轿车，D 级（运用）= A 照能开大客车。策略引擎是靠这个来决定「给这个学生多大的挑战」。如果你把 D 级的标签贴到 A 级的知识点上，系统会以为这个知识点很难、给学生铺很多台阶——结果就是学生在简单内容上被当成傻子对待，浪费时间。

> **设计理由**：文档的目标读者有两类——AI（执行者）和人（决策者）。如果只有技术语言，liufeng 在审查时无法独立判断设计是否合理，依赖 Hermes 或 DeepSeek 逐条翻译，审查效率极低且容易遗漏。大白话解释是「人机双读」文档体系的基础设施——AI 读技术内容、人读大白话内容，两者不互相干扰。

> **设计历史**：v1.0-v1.1 未定义此原则。v1.2（2026-05-15）liufeng 提出——「文档不光给 AI 看，也是给我看的。我需要大白话解释来辅助判断内容是否合格。」新增为原则八。v1.4 重新编号为原则六。v1.8 强制覆盖范围扩展至异常处置/熔断/信用分/内容代码分离等高风险模块，新增技术vs大白话冲突自动检测。

## 原则七：每个子系统必须包含自我进化执行方法

### 规则

进化路线图（原则一子原则四）回答了「什么时候进化」，但它不回答「怎么进化」。每个子系统文档还必须包含「自我进化执行方法」章节——定义从信号采集到效果验证的完整执行流程、消费者角色分工、芽芽的自主进化权限边界、以及进化过程中的安全控制机制（递归上限、死锁仲裁、信用分）。

### 消费者与使用方式（含权限分级人员台账）

自我进化涉及五类角色，每类角色有明确的账号权限、审核优先级和驳回权重：

| 消费者 | 读什么 | 怎么用 | 账号权限 | 审核优先级 | 驳回权重 | 权限回收规则 |
|--------|--------|--------|---------|-----------|---------|------------|
| **芽芽自己** | 触发条件 + AI-MUTABLE 阈值 | 持续监控自身指标，对比路线图的触发条件。条件满足时发起进化流程。监控驳回率/确认率，自行调节 AI-MUTABLE 阈值。低风险修改可自主执行 | 系统级服务账号，有进化执行权限，无北极星检查模块修改权限 | — | — | 触发不可修改边界告警 3 次 → 权限降级：低风险也需人工审核 30 天。北极星检查失败 3 次 → 权限降级至仅监控不可执行 60 天 |
| **AI 编程工具** | 路线图阶段目标 + 消费关系清单 + 不可修改边界 | 把阶段描述翻译为代码修改方案。先读消费关系清单确定影响范围，再读不可修改边界确定禁区，最后生成修改方案和影响评估报告。不得自行执行 | 工具账号，仅有读取权限和方案生成权限，无执行权限 | — | — | 生成方案触发越界告警 3 次 → 永久冻结该工具账号 |
| **产品经理（liufeng）** | 各维度当前阶段 + 人工依赖条件的介入点 + 高风险修改方案 | 判断进化方向对不对——「这个阶段该不该进」。审核高风险修改方案。在人工审核界面确认/驳回 | 最高权限账号，可签批高风险修改、可触发应急熔断（原则十一）、可驳回任意级别的进化 | 最高 | 一票否决 | 信用分重置权限有年度上限——每个子系统每年最多重置 1 次 |
| **教研人员** | 人工依赖条件的审核任务 + 中风险修改方案 | 审核系统提出的认知质/错因/知识点变更建议。在审核界面确认/驳回 | 教研审核账号，可审核中风险修改和人工依赖条件任务，不可签批高风险修改 | 中 | 集体多数（≥ 2/3 驳回则生效） | 连续驳回 ≥ 10 次且驳回率 < 30%（说明教研与系统判断方向性不一致） → 触发教研对齐会议 |
| **审查员（人或 AI）** | 修改方案 + 消费关系清单 + 回归测试集 | 验证修改方案会不会搞崩系统——跑回归测试、校验一致性、模拟修改后指标变化。输出审查报告 | 审查账号，可标记方案为「通过/疑似/不通过」，不可直接批准执行 | 高 | 技术否决（不通过则方案退回，但产品经理可 override） | 连续 5 次 override 后 30 天内审查员否决不可被 override |

> **用错的后果**：角色权限混淆会导致两类致命错误——(1) 低权限角色执行了高风险操作（如 AI 编程工具自行上线），(2) 高权限角色不知道自己的否决权（如产品经理以为系统拦停了就无需再确认）。

### 五步执行流程

#### 第一步：信号采集与触发判定（芽芽自己执行）

芽芽持续监控自身指标，与路线图中各维度的触发条件做对比。实际执行比「条件满足就触发」复杂——需要区分真信号和噪音。

**噪音过滤机制**：
- 学期初/学期末的数据波动极大——寒假期间 3 个月不活跃不代表系统坏了，开学第一周错误率飙升不代表策略失效
- 最小观察窗口：任何触发条件必须在 ≥ 1 个完整教学周期（通常 2 周）内持续满足，才算有效触发。**注意**：寒暑假和法定节假日不计入教学周期——判断「连续 2 周」时，跳过节假日区间，以实际教学日为准
- 触发前必须跑的统计检验：季节性分解（排除周期性波动）、变化点检测（确认指标真的变了，不是随机游走）

**元进化——信号采集和触发条件本身不合理时的自我修复**：
如果芽芽发现触发条件长期不满足（如连续 12 个月卡在同一阶段，但系统性能在明显下降），或者触发条件频繁误触发（如连续 3 次触发后回滚），芽芽应自我诊断：是不是阈值设得太高/太低？是不是触发条件的指标选错了？然后自行修改路线图中的 AI-MUTABLE 阈值，或生成「建议更换触发条件指标」的报告推送人工审核。修改后记录到 modificationHistory。

#### 第二步：修改方案生成（AI 编程工具执行）

触发条件满足后，AI 编程工具将路线图的阶段描述翻译为具体的代码修改方案。分三步：

1. **读边界**：先读不可修改边界（原则五）和间接修改边界，确定哪些绝对不能动、哪些动了会间接影响边界
2. **读影响面**：再读消费关系清单（原则三），列出「改了 A 之后，B/C/D 三个下游系统需要同步改什么」
3. **生成方案和影响评估报告**：方案含三个部分——具体改哪些文件/哪些行、影响范围分析、回滚条件定义

方案生成后停住。不得自动执行。等待下一步的审核。

**并发进化死锁仲裁**：AI 编程工具生成修改方案前，必须向全局注册表申请锁定消费关系链路。如果注册表检测到两个子系统同时触发了互相冲突的修改方案（如 S01 决定拆分 KP，S02 同时决定基于原 KP 合并诊断规则），系统自动标记为「进化的进化死锁」，合并为「联合进化方案」，强制升级为高风险操作，推送人工审核。**多子系统同向进化资源抢占冲突也走同一仲裁流程**——如 S03 和 S04 同时需要占用同一批学生数据做 A/B 测试，注册表协调分时复用或扩大样本池。任何单方绕过全局注册表的修改，视为越界。

**全局注册表宕机中断处置**：

| 阶段 | 行为 |
|------|------|
| 宕机检测 | 超过 30 秒无响应 → 触发故障模式 |
| 中断处置 | 所有持锁进化立即暂停（保留中间状态，不强制回滚） |
| 恢复检查 | 注册表恢复后，检查所有暂停操作的状态完整性——完整 → 继续执行；不完整 → 回滚到操作开始前 |
| 超时回滚 | 暂停超过 24 小时 → 自动回滚，记录原因为 `infrastructure_failure` |

#### 第三步：审核与自主进化（芽芽 + 人 + 审查员协作）

这一步有两条并行的执行通道——人工审核通道和自主进化通道。

**三级自主进化权限**：

| 风险级别 | 判断标准 | 谁来拍板 | 示例 |
|---------|---------|---------|------|
| **低风险（自主执行）** | 同类修改历史上成功 ≥ 3 次，影响面不超过 2 个子系统，不触碰不可修改边界 | 芽芽自主调用 AI 编程工具执行，事后报告给人工 | 调节 AI-MUTABLE 阈值（如 clusteringStability_min 从 0.7 调到 0.72）、新增可选字段 |
| **中风险（降级条件）** | 同类修改历史上成功 ≥ 1 次但 < 3 次，或影响面 3-5 个子系统 | 芽芽生成方案 + 历史成功案例摘要，推送人工审核。7 天无响应则按超时规则自动降级为「暂缓执行」 | 修改策略引擎步长公式参数、调整诊断引擎溯源深度 |
| **高风险（人工拦停）** | 首次执行该类型修改，或影响面 ≥ 6 个子系统，或触碰间接修改边界 | 芽芽必须等待人工明确签批。「超时驳回」不适用——高风险修改绝不自执行 | KP 拆分合并、认知质体系切换、事件协议 Schema 变更、元进化、北极星检查模块修改 |

**超时规则优先级**：

当同一审核任务同时被多套超时规则覆盖时（原则一通用超时 7-14 天、原则七风险分级超时 7 天、原则九环境超时），优先级顺位：

1. 原则九环境超时（最高）——正式环境高风险 = 永不超时；开发环境 = 无超时（自动审批）
2. 原则七风险分级超时（中等）——中风险 7 天无响应 → 暂缓执行
3. 原则一通用超时（最低）——仅当原则七和原则九均未定义该场景的超时规则时适用

**超时冲突示例**：正式环境中风险修改——原则七说 7 天超时，但原则九说正式环境中风险必须人工审核且不自动超时降级 → 以原则九为准，正式环境不自动超时降级。

**进化信用分机制**：

系统维护芽芽的进化信用分（0-100），影响自主进化权限：

| 信用分区间 | 影响 |
|-----------|------|
| 80-100 | 同类修改历史成功 ≥ 2 次即可降为低风险（正常需 ≥ 3 次）；中风险方案可跳过人工审核直接执行（事后报告） |
| 50-79 | 标准三级权限判定 |
| 20-49 | 低风险门槛提升到 ≥ 5 次成功；中风险方案审核超时时间从 7 天缩短到 3 天 |
| < 20 | 所有自主进化暂停——即使是低风险修改也必须人工签批。系统进入「信用重建期」，连续 3 次人工签批通过后恢复至 30 分 |

**信用分三子分体系与聚合规则**：

信用分内部拆为三个独立子分。三个子分的意义：低风险操作的自主权只由低风险历史贡献，高风险权限只由高风险历史贡献——防止用低风险小改积累信用分后去搞高风险操作。

| 子分 | 含义 | 贡献范围 |
|------|------|---------|
| `low_risk_score` | 低风险自主化信任度 | 仅由低风险操作的历史成功/失败决定 |
| `mid_risk_score` | 中风险自主化信任度 | 由中风险操作历史决定 |
| `high_risk_score` | 高风险建议信任度 | 由高风险操作历史决定（但高风险永远需人工签批，此分数影响审核优先级） |

**聚合规则**：

| 规则 | 说明 |
|------|------|
| 展示分 | min(low_risk_score, mid_risk_score, high_risk_score)——仪表盘显示最低值 |
| 分级暂停 | 任意子分 < 20 → 对应风险级别的操作暂停。如 low_risk_score < 20 → 低风险操作也需要人工签批，但中高风险不受影响 |
| 全系统暂停 | min(三个子分) < 10 → 所有级别进化暂停，进入信用重建期 |
| 月度上限 | 单月通过自主进化成功加分的总和上限 = +20 分（三个子分合计），防止信用分虚高 |

**信用分子分加减规则**：

| 事件 | low_risk_score | mid_risk_score | high_risk_score |
|------|---------------|----------------|-----------------|
| 低风险自主进化成功（30 天稳定） | +5 | 0 | 0 |
| 低风险自主进化触发回滚 | -15 | -5 | 0 |
| 中风险人工签批后成功 | 0 | +5 | +2 |
| 中风险人工签批后回滚 | 0 | -15 | -5 |
| 高风险人工签批后成功 | 0 | 0 | +3 |
| 高风险人工签批后回滚 | 0 | -5 | -15 |
| 触发不可修改边界告警（即使未执行） | -30 | -30 | -30 |
| 北极星检查失败 | -10 | -10 | -10 |
| 人工签批后成功（通用） | +2 | +2 | +2 |

**跨子分信用挪用拦截**：任何子分独立计算——低风险成功不能增加中/高风险子分。月度上限 = +20 是三个子分合计的上限。

**信用分初始化与恢复**：

- **初始值**：新子系统上线时三个子分各 = 50（标准三级权限）
- **破产恢复**：信用分 < 20 时，除「连续 3 次人工签批通过后恢复至 30 分」外，增加自动恢复通道——连续 30 个自然日无越界告警 + 无北极星负信号 → 自动恢复至 30 分
- **信用分统计口径**：30 天破产恢复按自然日计算（非教学日）。节假日期间没有教学活动，不可能产生越界告警——如果用教学日，破产恢复几乎不可能触发

**信用分重置场景**：

| 重置场景 | 触发条件 | 重置规则 |
|---------|---------|---------|
| 大版本重构 | 子系统 major 版本升级（如 S01 v2.x→v3.0） | liufeng 可签批重置为初始 50 分。需注明重置理由。每个子系统每年最多重置 1 次 |
| 学段整体迭代 | 新增学段（如从小学扩展到初中） | 旧信用分衰减 50%，与新学段初始 50 分加权平均 |
| 全量知识库重构 | knowledge.json 全量重建 | 信用分清零，从 0 重新积累 |

**多重安全机制优先级**：

当同一维度同时触发多个安全机制时，以最严格的机制为准。优先级顺位：

1. 原则十冻结（最高）——维度级冻结，覆盖所有其他机制
2. 信用分限制（中等）——信用分 < 20 暂停所有自主进化
3. 审核门槛提升（最低）——仅影响审核级别，不暂停进化

**优先级实战示例**：维度 X 信用分 = 25（接近限制），同时触发连续 3 次失败 → 原则十冻结 60 天。冻结期内信用分继续计算——冻结前 = 25，冻结期内北极星检查失败扣至 15，冻结期满后信用分 = 15 < 20 → 进入信用重建期。

冻结期内信用分继续计算但暂停生效——冻结期满后，信用分当时的值立即生效。例如：冻结前信用分 = 25，冻结期内因北极星检查失败扣至 10，冻结期满后信用分 = 10，进入 < 20 的信用重建期。

**芽芽自主进化的触发条件**：
- 低风险修改：芽芽在触发条件满足后，自行生成方案、自行调用 AI 编程工具执行、自行跑回归测试。全过程记录到 modificationHistory。
- 如果自主执行后效果衰减超过 20%（原则四标准），自动回滚并标记此次自主执行失败。同一类型连续失败 2 次，自动将该类型从中风险提升为高风险——需要人工审核。
- 如果自主执行后效果稳定（≥ 30 天效果衰减 < 5%），下次同类修改的审核门槛自动再降一级。

#### 第四步：安全执行与回滚（芽芽 + AI 编程工具配合）

执行不是直接上线。标准执行管道分三层：

1. **影子模式**：修改先在隔离环境运行（不接触真实学生数据），用历史数据回放验证。指标对齐修改前的基线预测区间（原则一子原则二）后进入下一层。**数据脱敏强制要求**：影子模式使用真实历史学生数据回放时，必须对隐私字段（学生姓名、学校名称、班级编号、设备指纹、地理位置）进行脱敏处理。脱敏后数据不可逆向还原。
2. **A/B 测试**：修改应用到 5-10% 的真实流量，与未修改版本对比核心指标（准确率、放弃率、完成率）。运行 ≥ 2 个完整教学周期后做统计检验。

**A/B 测试分组规则**：

| 规则 | 说明 |
|------|------|
| 分组单位 | 以**班级**为最小单位（防止同班学生交叉影响——同一教师教 A 组和 B 组学生会产生溢出效应） |
| 教师隔离 | 同一教师的所有班级必须分到同一组（防止教师在 A 组用新方法后在 B 组无意识应用） |
| 全局互斥 | 同一学生在多个并发 A/B 测试中只能参与一个（全局注册表登记） |
| 样本平衡 | A/B 两组在年级分布、学情分布上无显著差异。PSI < 0.1 视为平衡 |

3. **全量上线**：A/B 验证通过后逐步扩大流量（10% → 30% → 100%），每步观察 ≥ 1 周。

所有三层执行前，必须通过北极星检查（总纲）。

**回滚条件**（任何一层触发即执行）：
- A/B 测试阶段：修改组核心指标显著差于对照组（PSI ≥ 0.1）
- 全量上线阶段：核心指标超出修改前基线预测区间的 95% 置信边界
- 任何时候：北极星检查失败（负信号恶化或底线信号触碰）
- 任何时候：出现不可修改边界告警（间接影响检查失败）
- 任何时候：下游消费方报出新的 BLOCK 级别错误

回滚后：系统自动写入 modificationHistory（含 rollbackAt 时间戳 + 回滚原因），指标恢复到基线预测区间内时触发因果推断——「回滚后恢复，说明修改是原因」。如果 confoundingEvents 非空，因果推断附注「不能排除混淆变量的替代解释」。

#### 第五步：进化记录与效果验证（芽芽自己执行）

执行完不是结束。系统自己写进化记录：

1. **进化事件日志**：什么时候触发的、做了什么修改、效果对比（修改前 vs 修改后）、有没有 confoundingEvents
2. **稳定期自评**：稳定运行 ≥ 30 天后，系统自评「这次进化成功还是失败」——不是人说了算，是数据回来看。成功标准：核心指标不低于修改前、且效果衰减 < 5%
3. **审核门槛自动调节**：
   - 成功的进化：降低下一次同类修改的审核级别（高风险→中风险→低风险）
   - 失败的进化：自动收紧同类建议的触发条件（如 clusteringStability_min 提高 10%），并将该类修改的审核级别提升一级

**回滚后复盘模板**：

每次回滚必须生成标准化复盘报告，含以下固定维度：

| 维度 | 内容 |
|------|------|
| 回滚原因分类 | 北极星检查失败 / A/B测试恶化 / 超基线95%区间 / 不可修改边界告警 / 下游BLOCK错误 / 自主执行效果衰减超20% |
| 根因分析 | 进化方案本身的缺陷（设计错误）vs 执行环境的意外（外部事件干扰）vs 指标选错了（优化了不该优化的指标） |
| 整改方向 | 方案级：调整方案参数后重试 / 路线图级：修改触发条件的指标或阈值 / 冻结级：该维度暂停进化，等待人工分析 |
| 同类进化封禁时长 | 方案级整改：30 天后可重试；路线图级整改：90 天后可重试；冻结级：直到人工解除 |
| 整改落地时效 | 整改方案必须绑定迭代版本和上线时间。方案级整改：下次迭代上线（≤ 30 天）；路线图级整改：下个 major 版本上线（≤ 90 天）。超期未落地 → 该维度自动冻结 |
| 信用分影响 | 按进化信用分子分加减规则扣分 |
| 复盘报告存储位置 | `modificationHistory` 的 `postmortem` 字段 |

**元进化——路线图本身的进化**：
如果芽芽在进化记录中发现规律——如「维度二的阶段 3 触发了 5 次，每次都回滚，原因是触发条件的聚类稳定性指标选错了」——芽芽应自行修改路线图：更换触发条件的指标、调整阶段定义、或新增独立维度。修改路线图视为高风险操作，必须生成「元进化报告」推送人工审核且通过北极星检查。人工签批后，芽芽按新路线图重新执行进化流程。

**元进化递归控制**：

元进化（路线图修改路线图）本身可能触发无限递归——芽芽修改路线图→发现新路线图也不合理→再次修改→无限循环。必须施加以下递归上限：

| 约束 | 说明 |
|------|------|
| 单维度频次上限 | 单个维度 12 个月内最多触发 3 次元进化 |
| 全局频次上限 | 所有维度的元进化总次数 12 个月内 ≤ 9 次——防止单维度不超标但全局泛滥 |
| 连续失败冻结 | 连续 2 次元进化失败（修改路线图后效果更差）→ 该维度进入 6 个月冻结期，冻结期间自动进化降级为「仅记录不执行」 |
| 递归深度上限 | 元进化的元进化（修改「路线图修改规则」的规则）永远禁止自动执行，只能人工硬编码。任何关于路线图修改规则本身的自动修改 → 直接拒绝 |
| 冻结期后的恢复 | 6 个月冻结期满后，维度自动解冻，但首次元进化必须有 3 倍于正常标准的证据强度（如正常需 3 次触发 → 解冻后首次需 9 次触发） |

> **术语澄清**（v1.8）：元进化失败 = 修改路线图后效果更差（如更换触发条件指标后回滚率反而升高）。进化失败 = 执行进化后触发回滚（路线图本身没变，执行方案有问题）。两者不可混淆——审查清单的术语已统一。

### 各环节安全机制汇总

| 环节 | 关键安全机制 | 失败时的行为 |
|------|------------|------------|
| 信号采集 | 噪音过滤（季节性分解 + 变化点检测、节假日跳过）、最小观察窗口（≥2 周教学日）、外部事件自动识别（原则一子原则一） | 误触发后回滚，芽芽自我诊断触发条件是否合理 |
| 方案生成 | 不可修改边界检查、消费关系清单遍历、全局注册表死锁检测+宕机中断处置、影响评估报告 | 方案退回重新生成，记录失败原因到进化事件日志；检测到死锁→合并为联合方案推送人工；注册表宕机→保留中间状态等恢复 |
| 审核 | 三级权限分级、进化信用分子分动态调节、超时规则优先级（九>七>一）、高风险绝不自动执行 | 中风险超时→暂缓执行；高风险超时→保持等待，发出人工审核通道阻塞告警 |
| 安全执行 | 影子模式（含数据脱敏）→A/B测试（班级分组+教师隔离+全局互斥+PSI平衡）→逐步全量，每层有统计检验 + 北极星前置检查 | 任一层失败→回滚，写入 modificationHistory + 生成标准化复盘报告（含整改落地时效） |
| 记录验证 | 30 天稳定期自评、成功率/失败率影响子分信用分和审核门槛、回滚后标准化复盘模板（含整改绑定迭代版本+上线时间） | 连续失败→自动收紧触发条件 + 降低对应子分；成功率持续高→降低审核门槛 + 提高对应子分 |

> **设计理由**（约束）：定义了五类角色及其权限台账（含权限回收规则）、五步执行流程的完整安全机制、进化信用分的三子分动态调节体系（聚合规则/加减细则/跨子分挪用拦截/月度上限）、元进化的递归上限控制（单维度+全局双上限）、并发进化的死锁仲裁（含注册表宕机中断处置）、A/B测试分组规则（班级单位+教师隔离+全局互斥+PSI平衡）、以及回滚后的标准化复盘（含整改落地时效约束）。芽芽的自主进化权限是动态的——信用分越高自主权越大，信用分越低约束越紧。信用分三个子分各管各的风险级别——低风险做得再好也不能提升高风险权限。元进化有硬性递归上限——单维度 12 个月 ≤ 3 次，全局全维度 12 个月 ≤ 9 次，连续 2 次失败冻结 6 个月。这保证了芽芽能自主进化但不会失控、不会死循环、不会在错误方向上反复尝试。

> **设计历史**（参考）：v1.0-v1.2 未定义此原则。v1.3（2026-05-15）liufeng 审查时新增为原则十。v1.4（2026-05-15）与原则九分离，旧原则九并入新原则一，旧原则十独立保留为新原则七。v1.6（2026-05-15）新增元进化递归控制、并发进化死锁仲裁、进化信用分、权限分级人员台账、回滚后复盘模板。v1.7（2026-05-15）新增多重安全机制优先级、超时规则优先级、信用分初始化/破产恢复/分类隔离/重置场景。v1.8（2026-05-16）信用分子分体系全面落地——聚合规则（展示=min/分级暂停/全系统暂停min<10/月度+20上限）、独立加减分细则、跨子分挪用拦截；新增 A/B 测试分组规则、全局注册表宕机中断处置、权限回收规则、30天破产恢复明确为自然日、元进化全局全维度上限、影子模式数据脱敏、回滚复盘整改落地时效。

### 大白话

芽芽是一个机器人。这个机器人手里拿着两份东西：一份是设计说明书（文档），另一份是工具（AI 编程工具）。

平常，这个机器人自己监控自己的各项指标——就好比你家的空调自己监控室内温度。如果温度超过设定值，空调自己启动制冷。芽芽也一样——如果某个指标满足了进化路线图里的条件，它就自己启动进化流程。

但它分得清什么能自己干、什么必须请示你。就像你家的装修——换灯泡它自己来，换水管它自己来（因为已经换过三次了，都成功了）。但拆承重墙？它必须拿着方案来找你签字。这就是三级权限——低风险自执行、中风险降级、高风险必须等你拍板。

芽芽还有一个「信用分」——但现在分三本账：低风险账、中风险账、高风险账。就像你的驾照——小轿车扣分不影响大客车驾驶资格，大客车出事也不会牵连小轿车。仪表盘上显示最低那本账的分数——最短板决定你的自由。任意一本账掉到 20 分以下，对应的操作类型就得暂停。三本账都掉到 10 分以下？全停。每个月靠成功进化最多涨 20 分——防止一天之内完成一大堆低风险小改把信用分刷满。

做实验（A/B 测试）不能随便分组——同一个班的学生不能一半用旧版本一半用新版本，因为老师用完新方法后会在旧版本那边也无意识地调整教学。所以以班级为单位分组，一个老师的所有班必须在同一组。同一个学生不能同时参与多个实验——全局登记防止串场。

「调度员宕机」这个场景：注册表是管所有子系统协调进化的中央调度员。调度员突然联系不上了（超过 30 秒没回应）→ 所有正在进化的操作先暂停但保留现场。调度员恢复后检查现场——完好的继续，不完好的回滚到操作开始前。但如果调度员 24 小时都不恢复 → 不等了，全部回滚，记录「基础设施故障」。

信用分的「破产」统计按自然日算——不能按教学日，因为寒暑假没有教学活动不可能产生越界告警，按教学日等到天荒地老也恢复不了。

如果机器人发现自己手里的说明书有问题——比如「这个触发条件设得太高了，导致我卡在这阶段一年了」——它会自己改说明书，然后按新说明书重新执行进化。这就是「元进化」。但改说明书有严格限制：一个维度一年最多改 3 次，整个系统所有维度加起来一年最多改 9 次。改两次都改砸了 → 冻结 6 个月。改「改说明书的规则」的规则？永远不许自动改——只能你亲手改代码。

改错了会写事故报告，但报告不止写「为什么翻车」——还写「怎么补救、什么时候上线」。方案级的小补救下个迭代版本就要上线（30 天内），路线图级的大改下个大版本上线（90 天内）。超期没上线？该维度自动冻结。

终局就是：你只管你想让芽芽教什么（价值判断——北极星），芽芽自己能想清楚怎么教得更好（事实判断），而且能自己进化出更好的教学方法——不需要你一行一行盯着。但芽芽的「自主权」是用「信用」挣来的——三本账独立核算，做得好就多给自由，做砸了就收紧枷锁。

## 原则八：版本兼容与迁移

### 规则

子系统文档迭代时，必须定义新旧版本的兼容判定标准和迁移规则，防止版本割裂导致系统无法平滑升级。

### 版本兼容判定

| 版本升级类型 | 兼容性要求 | 迁移规则 |
|------------|-----------|---------|
| patch（如 v1.0→v1.1） | 完全向后兼容——旧版本产生的数据可被新版本直接消费 | 无需迁移 |
| minor（如 v1.x→v2.0） | 向后兼容——新版本可读取旧版本数据，但旧版本不可读取新版本新增字段 | 新增字段有默认值，自动填充 |
| major（如 v2.x→v3.0） | 不保证向后兼容——数据结构或核心规则发生了不兼容变更 | 必须提供迁移脚本，旧格式数据保留至少 6 个月过渡期 |

### 废弃字段过渡规则

任何字段被废弃时：

1. **过渡周期**：废弃字段在文档中保留 ≥ 6 个月（标注 `@deprecated since vX.Y, to be removed after YYYY-MM-DD`）
2. **过渡期内**：新版本代码同时读写新旧字段，旧字段仍作为 fallback
3. **过渡期后**：移除旧字段的读写逻辑，迁移脚本将旧数据批量转换为新格式
4. **迁移脚本要求**：必须包含数据完整性验证——迁移前后数据记录数、关键字段的非空率一致

**过渡周期的动态调整**：

| 场景 | 调整规则 |
|------|---------|
| 快速迭代场景（子系统月均 ≥ 2 次 minor 升级） | 过渡期可缩短至 3 个月——产品经理签批即可 |
| 慢速迭代场景（子系统年均 ≤ 1 次 minor 升级） | 过渡期保持 6 个月——给下游充足的响应时间 |
| 过渡期调整需在文档中标注调整理由和签批记录 |

### 文档版本引用规则

任何子系统文档引用其他文档时，必须标注引用的版本号。如果被引用文档升级了 major 版本，引用方必须评估兼容性并在变更记录中注明。

### 跨子系统版本兼容矩阵

多个子系统联动升级时，版本组合可能出现不兼容——如 S01 v2.0 + S02 v1.5 的组合虽各自都正确但放在一起不兼容。需要维护兼容矩阵：

| 子系统组合 | 已验证兼容的版本组合 | 已知不兼容的组合 |
|-----------|-------------------|----------------|
| S01 + S02 | S01 v2.0 + S02 v2.0、S01 v1.5 + S02 v1.5 | S01 v2.0 + S02 v1.5（S01 新增认知质字段在 S02 v1.5 中无消费入口） |
| S09 + S11 | S09 v2.0.4 + S11 v2.3 | S09 v2.0.4 + S11 v1.2（v1.2 无事件引用能力） |

联动升级前，AI 必须检查兼容矩阵——发现已知不兼容组合 → 拒绝执行，推送人工协调升级顺序。

> **设计理由**（约束）：子系统之间高度耦合——S07 引用 S01 的数据结构、S11 引用 S09 的事件协议。一个子系统的 major 升级如果没有任何过渡规则，下游子系统会瞬间断裂。静态的 6 个月过渡期不适用于所有场景——快速迭代需要更灵活的窗口，慢速迭代需要更长的缓冲。兼容矩阵解决了组合版本不兼容问题——两个子系统各自版本都对但放一起就错。没有这条原则，系统规模扩大后版本割裂会指数级恶化。

### 大白话

这条原则解决一个简单的问题：系统升级后，老数据还能不能用？

就像手机系统升级——小升级（iOS 17.1→17.2）你的老 App 都能用；中等升级（iOS 17→18）老 App 基本能用但新功能用不了；大升级（iOS 换 Android）老数据可能全废。

芽芽的规则是：废弃一个字段，不能直接删——要在文档里贴「已废弃」标签保留 6 个月，给所有用到这个字段的子系统一个缓冲期。但 6 个月不是死数字——系统迭代快可以缩短到 3 个月（你得签批），迭代慢就保持 6 个月。

还有一个容易忽视的问题：两个子系统各自升到最新版本，但放一起就崩了——就像你买了最新款手机和最新款耳机，各自都好好的，但就是连不上。所以需要一张兼容对照表——哪些版本组合验证过没问题、哪些已经知道会炸。

## 原则九：多环境适配

### 规则

子系统在不同运行环境下，进化权限、北极星校验强度、回滚优先级有显著差异。必须在文档中定义各环境的差异化规则。

### 四类环境定义

| 环境 | 用途 | 进化权限 | 北极星校验强度 | 回滚优先级 |
|------|------|---------|--------------|-----------|
| **开发环境** | 本地开发、单元测试 | 所有级别的进化均允许触发和测试执行，但不得影响其他环境。低风险修改默认自动审批 | 宽松——仅检查底线信号（不检查正/负信号），避免开发阶段因数据不足频繁拦停 | 最低——开发环境回滚不影响真实数据，仅用于验证回滚逻辑 |
| **测试环境** | 集成测试、CI/CD 流水线 | 低/中风险修改可自动执行，高风险需人工签批 | 标准——检查全三层信号，但负信号恶化仅 WARN 不 BLOCK（测试数据量小，统计不稳定） | 中——回滚后记录但不触发信用分扣减 |
| **影子教学环境** | 历史数据回放、A/B 测试对照组 | 仅低风险可自动执行，中/高风险必须人工签批 | 严格——全三层信号检查，但仅检查可验证信号 | 高——回滚后记录并触发复盘 |
| **正式教学环境** | 真实学生数据、全量上线 | 低风险事后报告，中风险人工审核，高风险必须产品经理签批。进化信用分 < 20 时所有级别暂停 | 最严格——全三层信号检查，任何恶化直接拦停，人工不可绕行 | 最高——回滚后立即触发标准化复盘 + 信用分扣减 + 同类封禁 |

### 各环境北极星信号矩阵

不同环境能检查的信号不同——影子环境用历史数据回放，「自主回访率」「求助行为」等交互信号无法验证。强行全查会导致误报：

| 环境 | 可检查信号 | 不可检查信号 | 处理方式 |
|------|-----------|-------------|---------|
| 开发/测试 | 仅底线信号 | 正/负信号 | 跳过正/负信号检查 |
| 影子 | 迁移题正确率、完成率、间隔掌握率（静态信号） | 自主回访率、求助行为（交互信号） | 静态信号完整检查，交互信号监控但不拦截——仅记录为 INFO 供参考 |
| 正式 | 全部信号 | 无 | 完整检查——全三层信号 |

### 进化路线图状态隔离

开发/测试环境维护独立的路线图状态快照，与正式环境完全隔离：

- 开发环境调试路线图阶段切换、调参后，不会污染正式环境的路线图当前阶段。
- 正式环境的路线图状态仅由正式环境的触发条件满足数据决定——不能从开发环境「导入」路线图进度。
- **影子环境路线图同步**：影子环境使用正式环境路线图状态的只读副本。正式环境路线图状态变更后 24 小时内，影子环境的只读副本自动同步——确保影子环境验证的基线是最新的。
- **配置文件拷贝禁用**：不得通过拷贝开发/测试环境的配置文件到正式环境来变相导入环境进度——配置文件包含环境标识字段，启动时校验：环境标识 ≠ 当前环境 → 拒绝加载。
- 环境间路线图差异检测：每月自动对比开发/测试环境与正式环境的路线图状态差异，生成差异报告——防止开发环境修改了路线图但未正式推进。

### 环境切换的数据隔离

开发/测试/影子环境的进化数据（modificationHistory、信用分、触发记录）与正式环境完全隔离——开发环境的高信用分不能带到正式环境。正式环境的信用分仅由正式环境的进化结果决定。

**正式环境信用分快照**：正式环境信用分每月自动备份快照——防止异常清零后无恢复基准。快照保留最近 12 个月。

**测试环境负信号归档**：测试环境的负信号 WARN 记录必须归档（不丢弃）——积累测试期的风险模式数据，在正式环境部署前做风险对照检查。

> **设计理由**（约束）：没人想在开发环境因为「数据量不够导致负信号误报」而被拦停——那会让开发效率归零。但正式环境必须铁面无私——真实学生的数据不能拿来当实验品。各环境北极星信号矩阵解决了影子环境交互信号不可检查的问题——只检查能检查的，不能检查的不瞎报。影子环境路线图自动同步（24h 内）解决了验证基线过时的问题。配置文件拷贝禁用封堵了「偷渡」路线图进度的旁路。

### 大白话

不同场合，不同规矩。就像开车：

开发环境 = 驾校训练场——你可以随便试、随便犯错，撞了也就撞了，不影响真路人。
测试环境 = 模拟考试——比训练场严一点，但考官知道你紧张，小问题先记着不直接挂。考完的错题本（负信号归档）别扔——正式路考前对照看看有没有重复的风险。
影子环境 = 驾校路考——跟真路一样严，但不是所有项目都能考。能用历史录像回放的项目（停车、倒库）认真考，需要真人互动的项目（跟考官对话）考不了就跳过不瞎扣分。而且路考用的路线图必须跟真路同步——真路路线变了你路考路线 24 小时内也得跟着变。
正式环境 = 拿驾照上路——犯错就真撞人了，所以规矩最严。信用分低的（新手司机）连基础操作都要被盯着。正式驾照每月拍照备份——万一系统出 bug 把你的记录清零了，从最近的备份恢复。驾校的成绩再好也不能带到正式驾照上——重新考。
还有一条铁规——不能把训练场的驾照文件复制粘贴到正式驾照系统里来「偷渡」进度——文件名上就打了环境标记，跨环境加载直接拒绝。

## 原则十：异常进化处置

### 规则

系统必须能识别自身进化过程中的异常模式，并自动触发分级冻结机制。不能在错误路径上反复尝试。

### 四类异常与冻结规则

| 异常类型 | 检测条件 | 冻结行动 | 解冻条件 |
|---------|---------|---------|---------|
| **连续进化失败** | 同一维度连续 3 次进化触发回滚（含自主执行和人工签批后的执行） | 该维度进入 60 天冻结期。冻结期间系统继续积累数据、生成诊断报告，但不尝试任何修改 | (1) 冻结期满；(2) 人工分析失败根因并提出整改方案；(3) 产品经理签批解冻。三者缺一不可。**紧急业务提前解冻**：产品经理可签批提前解冻，但需注明紧急理由，且冻结期至少完成 30 天。单维度每年最多提前解冻 1 次 |
| **长期阶段停滞** | 同一维度的同一阶段停留 ≥ 18 个教学月（不含寒暑假），且系统性能指标（准确率/完成率/用户满意度）呈下降趋势 | 生成「进化停滞诊断报告」推送给产品经理和教研团队，建议人工介入分析——是触发条件太严？还是路线图设计有问题需要元进化？不自动冻结，但降低该维度的信用分-5/月（直到恢复）。信用分递减有下限：最低降至 0 分，不会为负数 | 阶段升级或人工确认触发条件合理（如确实不需要升级） |
| **路线图逻辑矛盾** | 两个维度的终局状态互相冲突（如维度A终局=完全自主，维度B终局=永远人工审核，两者操作同一对象），或同一维度的阶段 N+1 系统能力描述弱于阶段 N | 系统直接冻结该子系统的全部进化维度，发出「路线图矛盾」告警 | 仅人工解除——产品经理确认路线图已修正并通过 DOC_ACCEPTANCE_STANDARD.md 验收 |
| **进化基础设施异常** | 北极星检查模块月度自检准确率 < 95%、全局注册表宕机超过 1 小时、消费关系链路检测到断裂（下游消费方引用不存在的字段） | 冻结全系统所有自主进化（覆盖原则十其他局部冻结），发出「进化基础设施异常」最高级别告警 | (1) 异常指标连续 2 次检测恢复正常；(2) 人工分析根因并修复；(3) 产品经理签批。三者缺一不可。此异常是总纲月度自检的落地执行框架 |

### 进化基础设施异常的故障区分

不同基础设施故障的冻结粒度应不同：

| 故障类型 | 冻结粒度 | 说明 |
|---------|---------|------|
| 全局注册表**只读**宕机（可查询但不可写入） | 暂停新进化发起，已持锁操作继续执行 | 注册表只读时已有锁的操作不受影响——已登记的路由仍有效 |
| 全局注册表**读写**宕机（完全不可用） | 全系统冻结——所有进化暂停 | 连已有锁的状态都无法验证，必须全停 |
| 消费关系链路断裂 | 冻结涉及断裂链路的所有子系统进化 | 不涉及断裂链路的子系统正常进化 |

### 冻结期的全局效应

任何子系统的任何维度进入冻结期后，该子系统的消费关系链路上的其他子系统不受影响——其他子系统继续正常进化。冻结是维度的局部冻结，不传染。

> **设计理由**（约束）：没有异常处置的系统会在错误路径上无限循环——连续失败→继续尝试→继续失败。冻结不是惩罚——是防止资源浪费和数据污染。长期停滞的「信用分递减」机制确保停滞不会无限期持续——系统用越来越低的信用分向人发出越来越强的信号「我需要帮助」（最低降至 0 分不穿底）。第四类「进化基础设施异常」区分了只读/读写故障——注册表只读宕机时已持锁的操作可以继续，不用全停。寒暑假不计入 18 个月教学月——与原则一子原则一的假日跳过规则对齐，避免寒暑假无教学数据却算停滞时间。

### 大白话

系统得知道自己什么时候「卡住了」，不能像坏掉的唱片一样反复跳针。

四种卡住的情况：

1. **连续翻车**——同一个地方连续摔三次。系统会说「这条路有问题，我停 60 天不走了，你来看看。」但如果真有紧急业务需要（比如修复了一个已知 bug，就是之前导致闪退的那个），你可以签批提前解冻——至少熬 30 天就行。每个维度每年只能提前解冻一次——天天「紧急」就不叫紧急了。

2. **长期发呆**——在同一个阶段待了 18 个教学月（一年半的上课时间，寒暑假不算）没动过。系统会说「我是不是该升级了但门槛设太高了？你帮我看一眼。」同时每个月降 5 分信用分。降到 0 分到底就不降了——不会出现负分。

3. **说明书自相矛盾**——路线图里左边说「完全自主」右边说「永远人工」，自己打自己脸。系统会说「这份说明书有问题，我全部暂停，等你修好再说。」

4. **基础设施坏了**——监控器判不准了、调度员宕机了、水管图过期了对不上实际管线了。调度员宕机还分两种：只是写不了但还能读（轻度）→ 已持锁的继续跑，新操作暂停；彻底联系不上（重度）→ 全部停。因为连已持锁的状态都验证不了，继续跑可能跑出问题。

## 原则十一：应急熔断

### 规则

系统必须具备最高优先级的人工一键熔断机制——产品经理可在任何时间、从任何界面触发，立即冻结全系统的自主进化功能。

### 三级熔断操作

| 操作 | 触发方式 | 效果 | 恢复方式 |
|------|---------|------|---------|
| **轻量级熔断** | 产品经理在管理后台点击「暂停高风险进化」 | 仅暂停所有高风险和中风险的自主进化。低风险自主进化继续正常执行。显示「高风险进化已暂停」横幅 | 仅产品经理可手动解除 |
| **子系统熔断** | 同上，指定子系统编号（S01-S16） | 仅停止指定子系统的全部级别进化（含低风险） | 同上 |
| **全系统熔断** | 产品经理在管理后台点击「紧急熔断」按钮，或通过飞书发送指定指令 | 立即停止所有正在进行的进化操作、拒绝所有新的进化触发、回滚所有未完成全量上线的修改。显示「系统进化已熔断」全局横幅 | 仅产品经理可手动解除。解除前系统显示熔断期间的进化待处理清单供审核 |

### 熔断触发场景（非穷举）

- 仪表盘多个核心指标同时异常（> 3 个指标超出 95% 预测区间）
- 教师/家长大规模投诉（飞书负面反馈频率超过阈值）
- 北极星底线信号被触碰（系统已自动拦停，但产品经理判断需要更彻底的冻结以排查根因）
- 教材大规模改版（如新课标出台）——在人工完成新标准对齐前，禁止系统基于旧标准自我修改
- 外部安全事件（如数据泄露风险）——冻结所有写操作相关进化

### 熔断后的系统状态

熔断期间，系统保持「观察模式」——正常教学功能不受影响，数据正常采集，指标正常监控。只有「自我修改」功能被冻结。仪表盘新增「熔断状态面板」——显示熔断剩余时间、待审核进化清单、熔断期间指标变化趋势。

**熔断期间数据风控**：熔断期间系统继续采集数据，但采集到的异常数据（如熔断期间因教学流程无进化调整导致的数据波动）会被自动打标 `collected_during_breaker`。熔断解除后，打标数据在统计分析中降权（weight = 0.5）——防止熔断期间的异常数据污染后续进化判断。

> **设计理由**（约束）：再完善的安全机制也有盲区。应急熔断是最后一道人工防线——当所有自动安全机制都失效或不够快时，人有一键熄火的权力。轻量级熔断（暂停高风险保留低风险）填补了 v1.7 的空白——不是所有异常都需要核按钮，有时只需要暂停高风险操作就够了。熔断期间数据风控解决了异常数据沉淀问题——熔断期间的数据波动不应污染恢复后的进化决策。这不需要任何技术理由——这是产品治理层面的最高权限。芽芽是教育工具，不是自动驾驶的核反应堆——人的最终控制权不可被系统架空。

### 大白话

这是芽芽的「核按钮」——现在分三档了：

**轻量级**：高风险和中等风险的进化暂停，低风险的继续。就像高速上看到警示牌——减速继续开，不是靠边熄火。

**子系统级**：指定某个子系统全部停。就像某一个车间停工检修，其他车间照常。

**全系统级**：全部熄火。所有进化停，正在改的立刻回调，不许任何新进化发生。

停了之后教学照常——孩子该怎么学怎么学。停的只是「系统自己改自己」的功能。你慢慢排查问题，排查完了再手动解锁。

但停的过程中系统还在采集数据——这些数据可能因「停摆效应」而异常（比如教学流程没有进化调整导致指标波动）。解除熔断后，这些打标数据在分析时降一半权重——不能因为停摆期间的数据异常就误判「解冻后应该立刻再冻上」。

这是最后一道防线。前面十一条原则都是「让系统自己管好自己」，这条是「但万一没管好，人还能接管」。

## 原则十二：内容进化与代码进化分离治理

### 规则

芽芽最频繁的进化不是「改代码/改算法」，而是「加题目、改错因描述、调知识点关系」——这些是内容层面的变更，和应用逻辑变更有本质区别。内容进化走教研审核流程，代码进化走路线图触发+北极星检查流程。两者不可混淆。

### 两类进化的定义与边界

| 进化类型 | 定义 | 示例 |
|---------|------|------|
| **内容进化** | 题目、错因描述、认知质文本描述、知识点关系图、教学素材的增删改 | 新增 3 道异分母分数加减法题；修改「粗心错误」的家长解释文案；调整 G5S2 分数单元的前置依赖关系 |
| **代码进化** | 算法、策略逻辑、系统参数、AI-MUTABLE 配置、数据结构的修改 | 修改策略引擎步长计算公式；调整诊断引擎溯源深度；新增 cognitive_primitives 数据结构字段 |
| **混合进化** | 内容结构变更——既改变了内容又改变了系统处理这些内容的方式 | 新增一个认知质维度（如「类比迁移能力」）——需要同时新增内容描述 + 修改系统数据处理逻辑 |

### 分离治理流程

#### 内容进化流程

```
教研提交内容变更建议
  → 内容审核（教研团队审核内容正确性、难度匹配、新课标对齐）
  → 内容版本控制（所有内容变更记录版本号 + 变更人 + 变更时间）
  → 灰度发布（新内容先对 5% 学生投放，观察 1 个完整教学周期）
    → 质量监控（新题目正确率/放弃率/平均用时是否在合理区间）
    → 全量发布 or 回滚
```

内容进化**不经过**北极星三层信号检查（加一道题不会让北极星偏离），**不走**信用分机制（内容提交是教研行为，不是系统自主行为），**不触发**原则七的二级/三级审核流程。

#### 代码进化流程

```
路线图触发条件满足
  → 北极星检查（总纲——全三层信号）
  → AI 编程工具生成修改方案（原则七第一步至第二步）
  → 信用分判定 + 审核分级（原则七第三步）
  → 影子模式 → A/B 测试 → 全量（原则七第四步）
  → 效果验证 + 信用分更新（原则七第五步）
```

代码进化走完整的十二条原则流程。

#### 混合进化流程

混合进化必须**同时通过两套流程**：

1. 内容部分：走内容进化流程（教研审核 + 灰度发布 + 质量监控）
2. 代码部分：走代码进化流程（北极星检查 + 信用分 + 安全执行管道）
3. 两套流程的结果都为「通过」后，混合进化方可在正式环境上线
4. 任何一套流程失败 → 混合进化整体搁置，生成「混合进化不对齐报告」

#### 北极星的共用

虽然内容进化和代码进化流程不同，但共用同一套北极星信号检测模块——内容进化只检测底线信号（不碰底线即可），代码进化检测全三层信号。

> **设计理由**（约束）：v1.0-v1.7 所有原则假设进化是「改代码/改算法」，但实际上芽芽最频繁的变更是内容——加题目、改错因文案、调整知识点关系。内容变更被错误地套用代码进化流程会导致：(1) 加一道题要过三级审核+信用分→教研效率归零；(2) 北极星检查对内容变更过度拦截（加题不会让北极星偏离）。分离治理后，内容进化走轻量级教研审核+灰度发布，代码进化走完整安全流程。混合进化（如新增认知质维度）必须两套都过——因为既涉及内容也涉及系统能力变更。

### 大白话

芽芽有两类进化，就像一家餐厅有两种变更：

**改菜单（内容进化）**：加一道新菜、改菜名、换菜的照片。这些走简单的流程——厨师长审核（教研审核）→ 让几桌客人试吃（灰度发布）→ 看反馈（质量监控）→ 全菜单更新。不需要惊动食品安全局（北极星），不需要审查厨师的信用分。因为加一道菜不会让餐厅偏离「好吃」这个方向。

**改厨房（代码进化）**：换灶台、重新排管线、改菜的做法。这些走严格流程——食品安全检查（北极星）→ 建筑师出改造方案 → 按信用分级审批 → 先在小厨房试 → 逐步推广 → 效果验证。

**即改菜单又改厨房（混合进化）**：比如决定「我们要提供素食选项」——不仅加新菜（内容），还要加一套素食灶台和工具（代码）。两套流程都得走，都过了才能上线。有一个没过？搁置，先解决没过的那个。

以前把加一道菜当改厨房处理——加道题也要过北极星检查、审信用分——这不是谨慎，是浪费。分离之后各走各的，互不干扰。

## 附录A：AI-MUTABLE 全局命名规范

为确保 AI 能准确解析和修改文档中的可进化参数，所有 AI-MUTABLE 标注必须遵循统一规范。

### 命名前缀

| 参数类别 | 前缀 | 示例 |
|---------|------|------|
| 最小/最大阈值（min/max） | `min_` / `max_` | `min_students_stage2`, `max_recursion_depth` |
| 稳定性/一致性阈值 | `stability_` / `consistency_` | `stability_clustering_threshold` |
| 时间周期（天/月/年） | `days_` / `months_` / `years_` | `months_min_consecutive` |
| 概率/比率阈值 | `prob_` / `rate_` | `prob_significance_threshold`, `rate_adoption` |
| 计数阈值 | `count_` | `count_min_independent_sources` |
| 系数/权重 | `coef_` / `weight_` | `coef_decay`, `weight_prerequisite` |
| 安全边际 | `safety_margin_` / `D_` | `D_unknown_confounding_safety_margin` |

### 前缀重叠冲突优先规则

当参数可同时匹配多个前缀时，按以下优先级选用：

| 优先级 | 规则 | 示例 |
|--------|------|------|
| 1 | 语义明确性优先——选最能表达参数用途的前缀 | 同为计数类的安全边际参数 → 用 `safety_margin_` 而非 `count_` |
| 2 | 项目中已有同类参数的命名惯例优先——保持一致性 | 如果已有 `D_unknown_confounding_*` 系列 → 新增同类参数沿用 `D_` 前缀 |
| 3 | 具体场景优先于通用场景——`stability_` 优先于 `prob_` | 聚类稳定性阈值 → `stability_clustering_threshold` 而非 `prob_clustering_threshold` |

### 数据类型约定

type 取值枚举（仅以下值合法，使用非枚举值 → 验收时 BLOCK）：

`int` / `float` / `bool` / `string` / `array[int]` / `array[float]` / `array[string]` / `null` / `enum`

- `null`：参数允许为空值（如可选阈值暂时不设置）。
- `enum`：参数的取值范围是离散的有限集合，必须在 range 中列出所有合法值，如 `type=enum, range=[ARIMA, ETS, SeasonalNaive]`。

所有 AI-MUTABLE 参数在文档中必须标注数据类型和取值区间：

```markdown
- **D1_minStudents** = 100        <!-- AI-MUTABLE: min_students_stage2, type=int, range=[30, 1000] -->
- **clusteringStability_min** = 0.7  <!-- AI-MUTABLE: stability_clustering_threshold, type=float, range=[0.5, 0.95] -->
- **reviewTimeoutDays** = 7       <!-- AI-MUTABLE: days_review_timeout, type=int, range=[3, 30] -->
- **baselineModel** = "ARIMA"    <!-- AI-MUTABLE: baseline_model_type, type=enum, range=[ARIMA, ETS, SeasonalNaive] -->
```

### 禁止的写法

- 把阈值嵌入段落文本中：「当聚类稳定性超过 0.7 时触发」→ 必须独立成行
- 没有 type 和 range 标注的 AI-MUTABLE 注释 → AI 无法判断修改边界
- 与代码中变量名不一致的 AI-MUTABLE 标识符 → 导致文档与代码无法自动对账

## 附录B：知识库联动绑定

### 规则

子系统文档的修改可能影响 knowledge.json、认知质库、错因库等核心数据文件。当以下类型的文档修改发生时，必须定义自动同步的触发逻辑：

| 文档修改类型 | 联动数据文件 | 同步触发 | 同步方式 |
|------------|------------|---------|---------|
| S01 新增/修改 KP 字段定义 | `knowledge.json`、`levels.json`、`knowledge_graph.json` | 文档 commit 后自动触发 | 迁移脚本自动为现有 KPs 的新增字段填充默认值；如字段为必填，迁移脚本检测到无默认值时 → BLOCK，阻止继续 |
| S01 新增认知质定义 | `cognitive_primitives` 数据表 | 文档 commit 后自动触发 | 迁移脚本创建新表字段，现有数据行填充 null |
| S01/S07 修改错因分类 | `error_taxonomy.json`、`error_kp_bindings.json` | 文档 commit 后自动触发 | 重跑 error binding 生成脚本，验证所有 errorCode 的一致性 |
| S08 新增/修改策略包 | `strategy_packs` 数据表 | 文档 commit 后自动触发 | 迁移脚本更新策略包编码表 |
| S09 新增/修改事件类型 | `events` Schema 表 | 文档 commit 后自动触发 | 迁移脚本更新事件 Schema，验证所有消费方的引用有效性 |

### 同步触发的环境限定

知识库联动仅在 commit 到 main/release 分支时触发实际迁移。开发分支 commit 仅触发验证性检查（dry-run）——验证迁移脚本能否正常执行、数据一致性检查能否通过，但不执行实际的数据写入。这防止开发环境的试验性修改污染生产数据文件。

### Dry-run 验证通过标准

Dry-run 不是「跑一遍不出错就行」——必须满足以下量化标准：

| 检查项 | 通过标准 |
|--------|---------|
| 迁移脚本执行成功率 | 100%（不允许任何错误） |
| 数据一致性校验通过率 | ≥ 95% 的 KP 验证通过 |
| 新增字段默认值覆盖率 | 100%（所有 KP 的新增字段都有默认值或已标记为「待人工填写」） |
| 错误容忍数量 | 0 个 BLOCK 级别错误。≤ 3 个 WARN 级别警告可接受 |

### 联动失败的防护

任何知识库联动触发后，如果数据一致性检查未通过（如新增字段导致 10% 以上现有 KP 数据不完整），系统自动回滚联动操作，保留文档修改但标记「数据未同步」，在仪表盘发出告警。

### 反向回滚同步

文档修改联动更新知识库后，如果文档本身因设计审查不通过而回滚（文档版本回退），知识库必须同步回退：

| 回滚类型 | 知识库处理 |
|---------|-----------|
| 文档 patch 版本回退（如 v1.2 → 回退到 v1.1） | 知识库联动迁移的内容自动反向执行（回退迁移脚本） |
| 文档 minor 回退（如 v2.1 → 回退到 v1.5） | 知识库从回退前的快照恢复。快照在每次联动触发前自动创建 |
| 文档 major 回退（如 v3.0 → 回退到 v2.x） | 必须人工执行知识库恢复——major 变更涉及数据结构重构，自动回退有数据丢失风险 |

## 附则：文档审查清单

### 原则零：本标准本身的演进规则

DOCUMENTATION_STANDARDS.md 本身的修改，适用本标准全部原则和总纲。新增原则、修改现有原则、调整阈值、变更示例——均需在变更记录中记录设计理由和设计历史。本标准本身也必须是自我进化的。

**标准自我异化防护**：本标准的每次修改，必须额外通过北极星强校验——「这个修改让文档标准更接近芽芽的核心使命了吗？」如果修改导致文档标准的约束强度弱化（如降低安全边际、缩短冻结期、放宽信用分门槛）而没有伴随「为什么之前的标准太严格导致了什么具体问题」的数据信号，视为标准自我异化——直接拦停。

### 审查清单

任何新增或修改的子系统文档，在提交前必须逐条核查。每项结论分三级：

| 结论 | 含义 | 处理 |
|------|------|------|
| ✅ 通过 | 完全符合标准 | 正常提交 |
| ⚠️ 轻微不合格 | 存在格式/表述问题但不影响逻辑正确性 | 记录问题，30 天内修正 |
| 🛑 严重不合格 | 缺失必要章节或存在逻辑冲突 | 直接驳回，修正后重新提交审查 |
| 🚫 直接驳回 | 触碰不可修改边界或北极星底线 | 立即冻结相关子系统进化，产品经理介入 |

- [ ] **总纲**：所有进化触发条件或执行流程中包含北极星检查节点。正信号/负信号/底线信号三层指标有预埋的数据采集结构和权重配比。底线信号有三级违规分级处置。北极星模拟运行有 train/validation 分离+信息泄漏检测+通过标准。管理员绕行禁令覆盖全部旁路（阈值篡改/数据污染/代码改写/签名跳过）。北极星检查模块有进化约束（高风险+人工签批+独立验证+防旧错+防新错+动态扩容月度自检）。

- [ ] **四层架构**：文档按四层结构组织（内功→元能力→文档基础设施→北极星），每一层的内容清晰标注属于哪一层。跨层强制合规校验卡点（G1-G7）已预埋。文档修改权限与代码修改权限已分离。第一层已标注禁用清单。第三层 AI-MUTABLE 跨文档引用带版本锁。

- [ ] **原则一**：触发条件标注了类型，人工依赖条件配齐了闭环（含驳回二次申诉通道）。外部事件自动识别机制已预埋（教研日历接口+confoundingEvents 空时安全边际收紧，注意正向乘负向除+标注不全时上浮至 0.3）。基线数据结构预埋为时序模型参数（限定 ARIMA/ETS，禁用过拟合模型）。回滚因果推断含无关变量排除。所有阈值 AI-MUTABLE 独立标注（附录A格式）。CI 一致性验证+线上 6h 动态巡检。包含自我进化路线图章节（含退化降级规则）。

- [ ] **原则二**：每个设计决定的设计理由和设计历史分列，无混合表述。设计理由含约束边界+修改禁忌。设计历史含数据触发源+变更生效范围+旧版本作废标记（`@superseded`）。

- [ ] **原则三**：接口约定章节包含消费关系清单，格式为「元素→被消费方→消费方式→故障类型→严重等级 P0-P4→检测方式→消费链路类型 同步/异步」。含消费关系清单实时校验（commit 时自动扫全链路）。含两级陈旧告警（60 天 warning + 90 天 critical）+ 强制核验机制。

- [ ] **原则四**：可共享认知资源采用条件切片验证——数据结构预埋 shareable（true/conditional/false）及 conditionalCriteria 字段。全局复现失败时自动寻找子群体特征。条件切片有维度白名单（4 维度）+ 子群样本量底线（≥100）+ 组合上限（≤2 维度组合 + ≤3 子群）。统计检验使用 PSI < 0.1（替代 p < 0.05）。有效果异常暴涨拦截。N/M 阈值进化有人工审批上限。Shareable 验证标准阈值在不可修改边界内。

- [ ] **原则五**：包含不可修改边界章节，列出全部七类不可修改项。学生原始行为数据细分为三层只读粒度（原始日志/统计聚合/衍生特征）。间接修改边界有量化判定标准（5%/15%/30% 三级）。越界告警有分级处置（轻/中/重三级）。Shareable 验证标准阈值已纳入不可修改边界。

- [ ] **原则六**：十个强制配大白话的章节类型（含异常处置/熔断/信用分/内容代码分离高风险模块）均有独立白话解释。每条白话有生活比喻、能回答「为什么」、能支持产品经理独立判断。技术 vs 白话释义冲突自动检测已配置。

- [ ] **原则七**：包含自我进化执行方法章节——含五类消费者及权限台账（含权限回收规则）、五步流程（含 A/B 分组规则 班级单位+教师隔离+全局互斥+PSI 平衡、全局注册表宕机中断处置、影子模式数据脱敏）、三级权限、进化信用分三子分体系（聚合规则 展示=min+分级暂停+全系统暂停 min<10+月度+20 上限、独立加减分细则、跨子分挪用拦截、30 天自然日破产恢复）、元进化递归上限（单维度 12 月≤3 次 + 全局全维度 12 月≤9 次+连续 2 次失败冻结 6 月）、并发死锁仲裁（含资源抢占冲突）、回滚后标准化复盘模板（含整改绑定迭代版本+上线时间、超期未落地自动冻结）。术语统一：元进化失败 ≠ 进化失败。

- [ ] **原则八**：包含版本兼容章节——定义 patch/minor/major 兼容性要求和迁移规则。废弃字段有过渡周期和动态调整规则（快迭代可缩至 3 月）。跨子系统版本兼容矩阵已建立。

- [ ] **原则九**：包含多环境适配章节——定义开发/测试/影子/正式四类环境的进化权限、北极星校验强度、回滚优先级差异。各环境北极星信号矩阵已定义（可检查/不可检查/处理方式）。影子环境路线图只读副本 24h 自动同步。配置文件拷贝已禁用（环境标识校验）。测试环境负信号已归档。正式环境信用分每月快照。

- [ ] **原则十**：包含异常进化处置章节——连续失败（3 次→60 天冻结，含紧急提前解冻规则）、长期停滞（18 教学月不含寒暑假→信用分递减至 0 下限）、路线图矛盾（立即全维度冻结）、进化基础设施异常（准确率<95%/注册表宕机/链路断裂→全系统冻结，区分只读/读写故障）四类异常的分级处置。总纲月度自检引用原则十第四类异常作为处置框架。

- [ ] **原则十一**：包含应急熔断章节——三级熔断操作（轻量级/子系统/全系统）、触发场景、恢复方式。熔断期间数据风控（打标+降权重 0.5）。

- [ ] **原则十二**：包含内容进化与代码进化分离治理章节——内容进化定义（题目/错因/认知质描述/关系图/素材的增删改）+ 流程（教研审核→版本控制→灰度发布→质量监控）。代码进化定义（算法/策略/参数/数据结构修改）+ 流程（路线图→北极星→信用分→安全管道→回滚）。混合进化定义+两套流程都过的要求。两者共用北极星信号检测。

- [ ] **附录A**：所有 AI-MUTABLE 标注遵循全局命名规范（前缀+type 含 null/enum+range）。命名前缀重叠冲突有优先规则。

- [ ] **附录B**：知识库联动绑定定义完整。环境限定（main/release commit 触发，开发分支 dry-run）。Dry-run 量化标准已定义（100% 成功率/≥95% 校验通过/100% 默认值覆盖/0 BLOCK）。反向回滚同步已定义（patch 自动回退/minor 快照恢复/major 人工恢复）。
